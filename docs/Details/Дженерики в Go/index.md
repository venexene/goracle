# Дженерики в Go

## 1. Введение: зачем Go понадобились дженерики

До версии 1.18 в Go не было языковых средств для написания обобщённого
кода. Разработчикам приходилось либо дублировать функции для каждого
типа, либо прибегать к `interface{}` с ручным приведением типов, теряя
статическую типобезопасность. Рассмотрим типичный пример — функция
минимума для двух чисел. Чтобы поддержать `int`, `float64`, `int32`,
требовалось написать три почти идентичные функции, отличающиеся только
сигнатурой. Любое изменение логики нужно было тиражировать по всем
копиям.

**Дженерики** (generics, обобщённое программирование) решают эту
проблему: один раз описываем алгоритм или структуру данных,
параметризованную типом, и компилятор автоматически генерирует
типобезопасный код для каждого конкретного типа, с которым эта
конструкция используется. В отличие от `interface{}`, дженерики дают
полную проверку типов на этапе компиляции — отладка ошибок
несоответствия типов в runtime исключена.

Дженерики появились в Go 1.18 (март 2022) после многолетнего
проектирования. Команда Go намеренно отложила их введение, стремясь
найти решение, органичное для философии языка: простое, без
метапрограммирования и без специализации шаблонов, как в C++.

> **Зачем это Go-разработчику.** Дженерики сокращают объём типового
> кода на порядок для структур данных (связные списки, кеши, деревья) и
> алгоритмов над срезами и map. При этом код остаётся идиоматичным:
> явным, читаемым и типобезопасным.

## 2. Параметры типа (Type Parameters)

Функциям и типам разрешено иметь **параметры типа** (type parameters).
Список параметров типа выглядит как обычный список параметров, за
исключением того, что вместо круглых скобок используются квадратные.

Чтобы показать, как это работает, начнём с базовой функции `Min` для
значений с плавающей запятой:

```go
func Min(x, y float64) float64 {
    if x < y {
        return x
    }
    return y
}
```)

Мы можем сделать эту функцию универсальной — заставить её работать с
разными типами — добавив список параметров типа. В этом примере мы
добавляем список параметров типа с одним параметром типа `T` и заменяем
использование `float64` на `T`.

```go
func GMin[T constraints.Ordered](x, y T) T {
    if x < y {
        return x
    }
    return y
}
```)

Теперь эту функцию можно вызвать с аргументом типа, написав вызов
следующим образом:

```go
GMin[int](2, 3)
```)



### Инстанцирование (instantiation)

Передача аргумента типа функции `GMin`, в данном случае `int`, называется
**созданием экземпляра (instantiation)**. Создание экземпляра происходит
в два этапа:

1. Компилятор заменяет все аргументы типа их соответствующими
   параметрами типа во всей обобщённой функции или типе.
2. Компилятор проверяет, удовлетворяет ли каждый аргумент типа
   соответствующему **ограничению** (рассмотрено в разделе 3). Если
   второй этап не пройден, создание экземпляра завершается неудачей, и
   программа становится недействительной.

После успешного создания экземпляра у нас есть функция, которую можно
вызывать так же, как и любую другую функцию.

```go
f := GMin[float64]
f(3.14, 2.71) // 2.71
```)

Например, в коде выше создание экземпляра `GMin[float64]` фактически
возвращает нашу исходную функцию `Min` с плавающей запятой, и мы можем
использовать её при вызове функции.



### Параметры типа в объявлении типа

Параметры типа также можно использовать с типами (`type`):

```go
type Tree[T any] struct {
    left, right *Tree[T]
    value      T
}

func (t *Tree[T]) Lookup(x T) *Tree[T] {
    // ...
}
```)

Здесь обобщённый тип `Tree` хранит значения параметра типа `T`.
Обобщённые типы могут иметь методы, например, метод `Lookup` в этом
примере. Для использования обобщённого типа его необходимо
инстанцировать; `Tree[string]` — это пример создания экземпляра `Tree`
с аргументом типа `string`.

> **Зачем это Go-разработчику.** Параметры типа позволяют вынести
> обобщённые структуры данных — деревья, кеши, связные списки — в
> переиспользуемые пакеты, не жертвуя типобезопасностью. Один раз
> написанный `Tree[T]` работает и со `string`, и с `int`, и с любым
> пользовательским типом.

## 3. Ограничения типов и наборы типов (Type Constraints & Type Sets)

Для понимания дженериков в Go необходимо разобраться с тем, как
ограничения типов определяют, какие именно типы можно подставлять вместо
параметров типа. Когда мы пишем обычную функцию, каждый её параметр
имеет конкретный тип, например, `float64`, и этот тип определяет набор
допустимых значений, которые можно передать в функцию. В случае с
обобщёнными функциями у нас есть параметры типа, и для каждого из них
тоже существует свой набор допустимых типов, который задаётся
**ограничением типа (type constraint)**. Это ограничение выполняет роль
**метатипа** — типа, который описывает не значения, а сами типы,
которые могут быть использованы.



### Интерфейс как набор типов

В Go ограничения типов всегда представляют собой интерфейсы, но с
выходом дженериков интерфейсы обрели новое значение. Раньше интерфейс
определял набор методов: любой тип, который реализовывал все указанные
методы, считался реализующим этот интерфейс. Теперь на интерфейсы можно
смотреть иначе: **интерфейс определяет набор типов**. Если тип входит в
этот набор, он реализует интерфейс.

Оба подхода эквивалентны для интерфейсов, состоящих только из методов,
но новый подход открывает дополнительные возможности: мы можем явно
перечислять типы, которые должны входить в набор.

```go
type Stringer interface {
    String() string
}
```)

```go
type Ordered interface {
    ~int | ~int8 | ~int16 | ~int32 | ~int64 |
    ~uint | ~uint8 | ~uint16 | ~uint32 | ~uint64 |
    ~float32 | ~float64 | ~string
}
```)

Оба подхода приводят к одному и тому же результату: для каждого набора
методов мы можем представить соответствующий набор типов, реализующих
эти методы, и это набор типов, определённых интерфейсом.



### Синтаксис объединения типов

Для явного перечисления типов в интерфейсах были добавлены новые
конструкции. Например, интерфейс `interface{ int | string | bool }`
определяет набор, состоящий ровно из трёх типов: `int`, `string` и
`bool`. Такой интерфейс может использоваться только как ограничение типа
и будет принимать любой из перечисленных типов.



### Тильда и базовые типы

Если мы хотим включить в набор не только сам тип `string`, но и все
пользовательские типы, основанные на `string` (например,
`type MyString string`), используется **тильда**: `~string` означает
множество всех типов, чьим **базовым типом** (underlying type) является
`string`.

```go
type MyString string
// ~string включает MyString в constraint
```)



### Ограничение `comparable`

В Go 1.18 также введено предопределённое ограничение **`comparable`**.
Оно обозначает множество всех типов, значения которых можно сравнивать
операторами `==` и `!=`. В это множество входят:

* все числовые типы;
* строки;
* указатели;
* каналы;
* интерфейсы;
* структуры и массивы, все поля/элементы которых также `comparable`.

`comparable` необходимо указывать как ограничение для типа-ключа
обобщённой map. Без него компилятор не гарантирует, что значения
параметра типа можно использовать в качестве ключа:

```go
func Keys[K comparable, V any](m map[K]V) []K {
    keys := make([]K, 0, len(m))
    for k := range m {
        keys = append(keys, k)
    }
    return keys
}
```



### `any` — предопределённый псевдоним

Поскольку пустой интерфейс, не содержащий никаких методов и не
перечисляющий никаких типов, является очень распространённым
ограничением, обозначающим «любой тип», в Go 1.18 был введён
предопределённый идентификатор **`any`** — псевдоним для `interface{}`.
Это позволяет писать более лаконичный код: `func F[T any]()` означает,
что параметр типа `T` может быть заменён любым типом.



### Пакет `constraints` и `Ordered`

Стандартный пакет **`constraints`** предоставляет готовые полезные
ограничения. Например, ограничение **`Ordered`** объединяет в себе все
целочисленные типы, все типы с плавающей точкой и строковые типы. Это
ограничение гарантирует, что внутри обобщённой функции значения
параметра типа можно сравнивать операторами `<`, `>`, `<=` и `>=`,
поскольку эти операции определены для всех типов, входящих в набор.



### Разрешённые операции в теле функции

В теле обобщённой функции разрешены только те операции, которые
допустимы для всех типов, входящих в ограничение. Если ограничение
включает только типы, поддерживающие сложение, то операцию `+`
использовать можно. Если ограничение включает типы, у которых нет общего
набора операций, компилятор не позволит выполнить эти операции внутри
функции. Таким образом, ограничения служат своеобразным **контрактом**:
они сообщают компилятору, какие типы допустимы, и на основе этой
информации компилятор разрешает или запрещает те или иные операции.

Использование интерфейсов в качестве наборов типов — это мощный новый
механизм, имеющий ключевое значение для работы ограничений типов в Go.
На данный момент интерфейсы, использующие новые синтаксические формы,
могут использоваться только в качестве ограничений.

> **Зачем это Go-разработчику.** Ограничения типов — это контракт между
> вами и компилятором. Вы точно описываете, с какими типами работает
> ваша обобщённая функция, а компилятор гарантирует, что ни один
> неподходящий тип не просочится. `comparable` незаменим для любых
> обобщённых map и функций поиска; `Ordered` — для сортировок и
> операций сравнения.

## 4. Вывод типов (Type Inference)

Одно из важнейших изменений, связанных с дженериками, — **вывод типов
аргументов функции**. Именно он позволяет писать обобщённый код в
естественном, лаконичном стиле, без необходимости явно перечислять все
параметры типа при каждом вызове. Go поддерживает два механизма вывода:
из аргументов функции и на основе ограничений.



### Вывод типа из аргументов функции

Когда в языке появляются параметры типа, у разработчика возникает
необходимость явно указывать аргументы типа при вызове обобщённых
функций. Вернёмся к примеру обобщённой функции `GMin`, которая принимает
два значения одного параметризованного типа и возвращает меньшее из них.
Без вывода типов такой вызов выглядел бы как `GMin[int](10, 20)`, где в
квадратных скобках явно указано, что тип `T` заменяется на `int`. Такой
синтаксис работает, но делает код более громоздким.

```go
GMin[int](10, 20) // явный тип
```)

Компилятор Go способен во многих случаях определять аргументы типа
автоматически, основываясь на типах обычных аргументов, которые
передаются в функцию. Вместо явного указания типа в квадратных скобках
достаточно написать `GMin(10, 20)`:

```go
GMin(10, 20) // вывод типа — T = int
```)

Компилятор смотрит на типы значений `10` и `20`, видит, что оба имеют
тип `int`, и делает вывод, что параметр типа `T` должен быть заменён на
`int`.

Важно понимать, что вывод типов работает **только для тех параметров
типа, которые используются в объявлении обычных параметров функции**.
Если параметр типа используется только в возвращаемом значении или
только внутри тела функции, компилятор не сможет вывести его из
аргументов вызова. Например, для функции `func MakeT[T any]() T`,
которая не принимает аргументов, а только возвращает значение типа `T`,
вывод типов невозможен:

```go
func MakeT[T any]() T {
    var zero T
    return zero
}
// MakeT() — ошибка, T невыводим
// MakeT[int]() — OK
```)

В таком случае при вызове обязательно придётся указывать аргумент типа
явно: `MakeT[int]()`. Это логично, поскольку компилятору просто неоткуда
взять информацию о том, какой тип требуется подставить.



### Вывод типа на основе ограничений (constraint type inference)

Go поддерживает ещё один механизм вывода типов, который называется
**выводом типа на основе ограничений**. Рассмотрим пример с
масштабированием среза целых чисел. Предположим, у нас есть обобщённая
функция `Scale`, которая принимает срез чисел и коэффициент
масштабирования, умножая каждый элемент среза на этот коэффициент, и
возвращает новый срез:

```go
func Scale[E constraints.Integer](s []E, c E) []E {
    r := make([]E, len(s))
    for i, v := range s {
        r[i] = v * c
    }
    return r
}
```)

Функция принимает срез типа `[]E`, где `E` ограничено целочисленными
типами, и возвращает значение того же типа `[]E`. Это работает для
обычных срезов, но создаёт проблемы с пользовательскими типами,
основанными на срезах.

Представьте, что у нас есть тип `Point`, который представляет собой
просто срез целых чисел, определяющих координаты точки. У этого типа
есть методы, например, метод `String`, который красиво выводит
координаты. Если мы вызовем нашу функцию `Scale` с аргументом типа
`Point`, она вернёт значение типа `[]int32`, а не `Point`:

```go
type Point int32
points := []Point{1, 2, 3}
scaled := Scale(points, 2) // returns []int32, not []Point
```)

```go
var result []Point = scaled // compile error: []int32 != []Point
```)

В результате мы потеряем всю информацию о том, что это была точка, и не
сможем вызвать методы, определённые для типа `Point`. Код, который
ожидал получить `Point`, а получил просто срез, не скомпилируется.

Чтобы решить эту проблему, нужно изменить сигнатуру функции так, чтобы
она возвращала не базовый тип среза, а тот самый пользовательский тип,
который был передан на вход. Для этого вводится второй параметр типа
`S`, который представляет собой сам тип среза, а не его элемент.
Ограничение для `S` выглядит как `~[]E`, что означает: `S` должен быть
типом, чьим базовым типом является срез элементов типа `E`:

```go
func Scale[S ~[]E, E constraints.Integer](s S, c E) S {
    r := make(S, len(s))
    for i, v := range s {
        r[i] = v * c
    }
    return r
}
```)

Первый параметр функции теперь имеет тип `S`, а не `[]E`, и возвращаемый
тип также `S`. Теперь, когда мы вызываем `Scale` с аргументом типа
`Point`, параметр типа `S` выводится как `Point`, а параметр типа `E`
выводится как тип элемента `Point`, то есть `int32`. Функция возвращает
значение типа `Point`, и все методы этого типа остаются доступными.

Здесь работает **вывод типа на основе ограничений**: когда ограничение
одного параметра типа определено через другой параметр типа. В нашем
случае ограничение для `S` — это `~[]E`. Когда компилятор выводит, что
`S` равен `Point`, он смотрит на ограничение и понимает, что базовым
типом `Point` является `[]int32`. Из этого он делает вывод, что параметр
типа `E` должен быть равен `int32`. Тип для `E` выводится не из
аргументов функции, а из того факта, что мы уже знаем тип `S` и знаем,
как `S` связан с `E` через ограничение.

Этот механизм особенно полезен, когда один из аргументов типа выводится
по обычным правилам из аргументов функции, а второй выводится из первого
через их взаимосвязь в ограничении.

> **Зачем это Go-разработчику.** Вывод типов — то, что делает дженерики
> Go удобными в повседневной работе. Вы пишете `Scale(p, 2)` вместо
> `Scale[Point, int32](p, 2)`, а компилятор сам восстанавливает все
> параметры типа из контекста. Вывод на основе ограничений критичен для
> сохранения пользовательских типов при проходе через обобщённые
> функции — без него дженерики были бы гораздо менее выразительными.

## 5. Когда применять дженерики: рекомендации

Дженерики — мощный инструмент, но их не следует применять бездумно.
Команда Go сформулировала ряд рекомендаций, помогающих определить,
когда обобщённый код оправдан, а когда лучше обойтись обычными
интерфейсами.



### Когда стоит использовать дженерики

* **Обобщённые структуры данных.** Если вы пишете связный список,
  дерево, кеш, очередь, стек или любую другую структуру, которая должна
  хранить значения произвольного типа, — это прямое показание к
  дженерикам. Одна реализация `List[T]` заменяет `ListInt`, `ListString`
  и так далее.
* **Алгоритмы над срезами и map.** Функции `Filter`, `Map`, `Reduce`,
  `Contains`, `Keys`, `Values` — классические кандидаты на
  параметризацию типом элемента коллекции.
* **Функции, работающие с любым числовым типом.** Если алгоритм
  (например, `Sum`, `Min`, `Max`, `Clamp`) не зависит от конкретного
  числового типа, дженерики с ограничением `constraints.Ordered` или
  `constraints.Integer | constraints.Float` устраняют дублирование.



### Когда не стоит использовать дженерики

* **Простой интерфейс уже решает задачу.** Если вам нужно
  параметризовать поведение, а не тип хранимых данных, обычный
  интерфейс с одним-двумя методами — более идиоматичное решение.
* **Только один конкретный тип используется на практике.** Если вы
  пишете обобщённую функцию, но во всём проекте она вызывается ровно с
  одним типом, — дженерики не дают выигрыша, а усложняют чтение кода.
* **Дженерики не заменяют полиморфизм через интерфейсы.** Это
  принципиально разные инструменты. Интерфейсы — для абстракции
  поведения (полиморфизм времени выполнения), дженерики — для абстракции
  типов (полиморфизм времени компиляции).



### Ограничения Go-дженериков

| Ограничение                                            | Пояснение                                                                                                                                |
| ------------------------------------------------------ | ---------------------------------------------------------------------------------------------------------------------------------------- |
| Нет метапрограммирования                               | Нельзя во время компиляции итерировать по полям типа или генерировать код в зависимости от имени типа. Go-дженерики — это не шаблоны C++ |
| Нет специализации                                      | Нельзя написать отдельную реализацию обобщённой функции для конкретного типа (например, оптимизированную `Sum` для `int`)                |
| Методы типов не могут иметь собственные параметры типа | Обобщённый тип может иметь методы, но сигнатура метода не может добавлять новые параметры типа, отличные от параметров типа самого типа  |
| Нет вариативных параметров типа                        | Нельзя написать `func F[T ...any]()` — количество параметров типа всегда фиксировано                                                     |

> **Зачем это Go-разработчику.** Главный критерий: дженерики нужны,
> когда вы параметризуете *тип данных*. Когда вы параметризуете
> *поведение* — используйте интерфейсы. Смешивать эти два механизма
> можно и нужно, но начинать всегда стоит с вопроса «можно ли решить
> эту задачу без дженериков?»
