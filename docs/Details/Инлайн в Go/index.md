# Инлайн в Go

## 1. Что такое инлайн (inline) в Go?

**Инлайн (inlining)** — это оптимизация компилятора, при которой вызов функции заменяется непосредственно её телом.

Проще говоря, компилятор «копирует» код из вызываемой функции и «вставляет» его на место вызова, вместо того чтобы генерировать инструкции для перехода к другой части кода.

![](assets/7iVBBHxoDuRgxHShi3Lv6L08pfx_0zUuTWKTDUKy6hQ=.png)

Здесь при выполнении `main` происходит вызов `add`: процессор перепрыгивает в код функции `add`, выполняет сложение и возвращается назад.

![](assets/CMATSMpftZR2NxRaYvtrNaIXi5VDXkEVGIchqBWFkiQ=.png)



### Зачем это нужно?

* **Устранение накладных расходов:** Каждый вызов функции требует создания нового stack frame (запись аргументов, возвращаемого адреса и т.д.). Для очень маленьких функций эти накладные расходы могут быть сопоставимы с полезной работой самой функции — вызов пустой функции в Go стоит порядка десятков наносекунд. Инлайн убирает эти затраты полностью.
* **Открытие возможностей для других оптимизаций:** Когда код находится «в одном месте», компилятор может применить к нему более агрессивные оптимизации: **constant folding** (постоянное свёртывание), **dead code elimination** (удаление мёртвого кода), лучший анализ потоков данных.
* **Улучшение локальности кеша инструкций:** Код после инлайна лежит в памяти последовательно, без `CALL`/`RET`. Процессору не нужно прыгать в другой участок памяти и сбрасывать конвейер — это снижает количество **instruction cache misses** и **branch mispredictions**.

> **Зачем это Go-разработчику.** Инлайн напрямую влияет на производительность вашего кода. Маленькие вспомогательные функции (геттеры, простые преобразования, обёртки) — идеальные кандидаты. Пишите их компактно, и компилятор сделает остальное.



### Ограничения

* Инлайн применяется в основном к маленьким и простым функциям. Сложные функции (с циклами, сложными условиями, вызовами других функций) обычно не инлайнятся, так как это может привести к сильному «разбуханию» бинарного кода.
* **Рекурсивные функции** не инлайнятся — компилятор не может заменить рекурсивный вызов телом функции, так как это привело бы к бесконечному разворачиванию. Go также не поддерживает оптимизацию хвостовой рекурсии.
* **Методы интерфейсов** не инлайнятся при вызове через интерфейсную переменную, поскольку компилятор не знает конкретный тип на этапе компиляции. Исключение — **девиртуализация** (Go 1.21+), когда компилятор доказывает конкретный тип и заменяет интерфейсный вызов прямым.
* **Функции с&#x20;****`defer`** имеют повышенную стоимость из-за дополнительной логики разворачивания стека, что снижает шансы на инлайн.
* Решение об инлайне принимает компилятор, а не программист. Хотя можно давать подсказки с помощью комментариев `//go:noinline` (запретить) или `//go:inline` (предложить, но это слабая рекомендация). Подробнее — в разделе 4.

## 2. Когда компилятор делает инлайн

Компилятор не может инлайнить всё подряд. Ему нужна метрика, чтобы определить, будет ли инлайн выгоден и не приведёт ли он к чрезмерному росту размера программы. Этой метрикой и являются **«веса» (weights)** или **«стоимость» (cost)**.



### Что такое «вес» функции?

Это численное значение, которое оценивает «сложность» функции с точки зрения компилятора. Чем выше вес, тем «тяжелее» функция и тем меньше шансов, что она будет инлайнирована.



### Как считаются веса?

Компилятор Go анализирует **AST (Abstract Syntax Tree** — абстрактное синтаксическое дерево) функции и присваивает «стоимость» различным конструкциям. Компилятор «обходит» все узлы AST функции и суммирует их стоимость — полученная сумма и есть итоговый «вес».

Примерные значения стоимости для разных узлов AST (могут меняться в разных версиях Go):

| Конструкция                                              | Стоимость                        |
| -------------------------------------------------------- | -------------------------------- |
| Простое выражение (арифметика, присваивание)             | 1                                |
| Обращение к элементу слайса/массива                      | 1                                |
| Вызов встроенной функции (`len`, `cap`, `append` и т.д.) | 1                                |
| Конструкция `if`                                         | 2                                |
| Конструкция `for` (цикл)                                 | 3                                |
| Вызов пользовательской функции                           | 5+                               |
| Конструкция `switch`/`select`                            | 3+                               |
| `defer`                                                  | дополнительные накладные расходы |

> **Зачем это Go-разработчику.** Понимание «стоимости» конструкций помогает писать код, который компилятор охотнее инлайнит. Например, вынос тела цикла в отдельную функцию может сделать вызывающую функцию достаточно лёгкой для инлайна.



### Порог инлайнинга

У компилятора есть внутренний порог (например, 80). Если «вес» функции меньше этого порога, компилятор попытается её инлайнить. Точное значение порога зависит от версии Go и контекста: для **mid-stack inlining** (раздел 5) порог выше — компилятор более агрессивен при встраивании функций, находящихся в середине цепочки вызовов.

## 3. Как узнать, что компилятор заинлайнил функцию?

Основной инструмент — флаг `-gcflags="-m"`, который включает вывод решений компилятора об оптимизациях:

```shellscript
go build -gcflags="-m" ./...
```

Вывод покажет:

```
./main.go:12:6: can inline add
./main.go:8:6: can inline main
./main.go:9:5: inlining call to add
```

* `can inline add` — компилятор считает функцию `add` достаточно лёгкой для инлайна.
* `inlining call to add` — вызов `add` действительно был заинлайнен в месте вызова.
* `function too complex for inlining` — функция слишком сложна, инлайн не применяется.

Для более подробного вывода используйте `-gcflags="-m -m"` (двойной `-m`).

Глубокий анализ через SSA-дамп

Для полного понимания решений инлайнера можно посмотреть SSA-представление:

```shellscript
go build -gcflags="-d=ssa/inline/debug=1" 2>&1 | grep -i inline
```

Или получить полный дамп с помощью `-d=ssa/check_bce/debug=1` для анализа конкретной функции.

Также полезна утилита просмотра SSA, генерирующая HTML-отчёт:

```shellscript
GOSSAFUNC=main go build
```

Откроется `ssa.html` в браузере, где визуально видно все этапы компиляции, включая этап инлайнинга.

> **Зачем это Go-разработчику.** Всегда проверяйте `-gcflags="-m"` при оптимизации узких мест. Вы можете быть уверены, что ваша маленькая функция заинлайнится, а на деле — нет (из-за скрытой сложности вроде `defer` или вызова другого метода). Не гадайте — проверяйте.

## 4. Управление инлайнингом: директивы компилятора

Go предоставляет две директивы для управления инлайнингом — одна жёсткая, другая мягкая.



### `//go:noinline` — запрет инлайна

Категорический запрет компилятору инлайнить функцию. Применяется непосредственно перед объявлением функции:

```go
//go:noinline
func expensiveDebug(x int) int {
    return x * 2
}
```

**Когда применять:**

* **Бенчмарки:** Чтобы изолировать стоимость вызова конкретной функции и не дать компилятору «оптимизировать её из существования».
* **Отладка:** Инлайн «схлопывает» стектрейсы — с `//go:noinline` функция всегда видна в стеке вызовов.
* **Предотвращение разбухания кода:** Редкая ситуация, когда большая функция используется во многих местах и инлайн приводит к чрезмерному росту бинарного файла.



### `//go:inline` — рекомендация инлайна

Мягкая подсказка компилятору. Компилятор **может проигнорировать** эту директиву, если функция превышает порог сложности:

```go
//go:inline
func add(a, b int) int {
    return a + b
}
```

На практике эта директива редко нужна — компилятор и так инлайнит всё, что считает выгодным. Полезна в основном для функций, которые находятся на границе порога: вы говорите компилятору «я готов заплатить размером бинарного файла за скорость здесь».

> **Зачем это Go-разработчику.** `//go:noinline` — ваш основной инструмент. Используйте его в бенчмарках, чтобы получить честные цифры, и в отлаживаемых функциях, чтобы видеть полный стектрейс. `//go:inline` применяйте только после подтверждения через `-gcflags="-m"`, что без неё инлайн не происходит.

## 5. Mid-stack inlining (Go 1.12+)

До Go 1.12 компилятор инлайнил только **листовые (leaf) функции** — те, которые сами не вызывают других функций. Это серьёзно ограничивало применимость оптимизации.

Начиная с Go 1.12, появился **mid-stack inlining** — возможность инлайнить функцию, находящуюся не только на конце, но и в середине цепочки вызовов.



### Как это работает

Рассмотрим цепочку: `A → B → C → D`, где `D` — листовая функция.

* **Без mid-stack (Go < 1.12):** инлайнится только `D`.
* **С mid-stack (Go ≥ 1.12):** `D` встраивается в `C`, затем `C` (уже с учётом встроенной `D`) может быть встроена в `B`, и так далее. Инлайн идёт «волной» от листьев вверх по стеку.



### Порог для mid-stack

Для mid-stack инлайнинга порог выше, чем для leaf-инлайна — компилятор более агрессивен, потому что потенциальный выигрыш больше (устраняется не один вызов, а потенциально целая цепочка). В современных версиях Go (1.20+) порог для mid-stack может достигать значений \~200 против \~80 для leaf-инлайна.

> **Зачем это Go-разработчику.** Благодаря mid-stack inlining вы можете писать код с несколькими уровнями вспомогательных функций и быть уверенным, что компилятор «схлопнет» их в эффективный плоский код. Это снижает необходимость в ручном «разворачивании» логики ради производительности.

## 6. Связь инлайна с escape analysis и другими оптимизациями

Инлайн не существует изолированно — он тесно связан с другими механизмами компилятора и часто служит «катализатором» для более глубоких оптимизаций.



### Инлайн и escape analysis

**Escape analysis** определяет, выделяется ли переменная в куче (heap) или на стеке (stack). Инлайн может кардинально изменить его результат.

Рассмотрим пример:

```go
func newInt() *int {
    x := 42
    return &x // x «убегает» в кучу
}

func main() {
    p := newInt()
    fmt.Println(*p)
}
```

Без инлайна `x` выделяется в куче — она «убегает» из `newInt`. Но если `newInt` заинлайнится в `main`, компилятор увидит полный контекст использования и может доказать, что `x` не живёт дольше `main`. В результате `x` останется на стеке — выделения в куче не будет.



### Инлайн и девиртуализация (Go 1.21+)

**Девиртуализация (devirtualization)** — это замена вызова метода через интерфейс прямым вызовом конкретного типа. Если компилятор может доказать, какой конкретный тип скрывается за интерфейсом, он:

1. Заменяет интерфейсный вызов прямым.
2. После этого прямой вызов может быть заинлайнен.

```go
var w io.Writer = os.Stdout
w.Write([]byte("hello")) // интерфейсный вызов
```

Если компилятор докажет, что `w` — это всегда `*os.File`, вызов `Write` будет девиртуализован и потенциально заинлайнен.



### Инлайн и constant propagation

После инлайна компилятор видит фактические значения аргументов и может:

* Вычислить выражение на этапе компиляции (constant folding).
* Удалить ветки кода, которые никогда не выполняются с данными аргументами (dead code elimination).

> **Зачем это Go-разработчику.** Инлайн — это «ключ», открывающий дверь для других оптимизаций. Функция может быть тривиальной сама по себе, но её инлайн позволяет компилятору увидеть общую картину и применить каскад оптимизаций: исключить выделение в куче, вычислить константы, удалить мёртвый код.

## 7. Влияние на отладку и стектрейсы

Инлайн имеет обратную сторону: он «схлопывает» стек вызовов, что затрудняет отладку.



### Как инлайн меняет стектрейс

Рассмотрим цепочку вызовов `main → process → validate`, где `validate` заинлайнен в `process`:

**Стектрейс без инлайна:**

```
goroutine 1 [running]:
main.validate(...)
    /app/main.go:25
main.process(...)
    /app/main.go:15
main.main()
    /app/main.go:8
```

**Стектрейс с инлайном&#x20;****`validate`****:**

```
goroutine 1 [running]:
main.process(...)
    /app/main.go:15
main.main()
    /app/main.go:8
```

Строка `validate` исчезла из трейса — паника или ошибка будет указывать на `process`, хотя фактически проблема возникла в `validate`. Это усложняет диагностику.



### Что показывает pprof

При профилировании через `pprof` заинлайненная функция может быть «поглощена» вызывающей. Инструменты вроде `go tool pprof` умеют частично восстанавливать информацию об инлайне, но не всегда точно.



### Практические рекомендации по отладке

* Для функций, где важна точность стектрейса (валидация, обработка ошибок), используйте `//go:noinline`.
* При анализе падений в production всегда учитывайте, что часть цепочки может быть «схлопнута» инлайном.
* Используйте `GOTRACEBACK=system` для максимально подробного стектрейса.

> **Зачем это Go-разработчику.** Инлайн делает код быстрее, но сложнее для отладки. Это осознанный компромисс. В большинстве случаев выигрыш в производительности перевешивает — но в критичных к диагностике участках применяйте `//go:noinline`.

## 8. Практические рекомендации

| Рекомендация                           | Пояснение                                                                                                        |
| -------------------------------------- | ---------------------------------------------------------------------------------------------------------------- |
| Пишите маленькие, чистые функции       | Компилятор охотнее инлайнит функции с низким весом AST. Разбивайте сложную логику на компактные блоки            |
| Не злоупотребляйте `//go:noinline`     | Только для бенчмарков и отладки. Массовый запрет инлайна замедляет программу                                     |
| Проверяйте через `-gcflags="-m"`       | Не гадайте — смотрите реальные решения компилятора при оптимизации узких мест                                    |
| Помните про компромисс скорость/размер | Инлайн ускоряет выполнение, но увеличивает бинарный файл. В подавляющем большинстве случаев выигрыш перевешивает |
| Обновляйте Go                          | Каждая новая версия улучшает эвристики инлайнера — код, не инлайнившийся в 1.19, может инлайниться в 1.22        |
| Учитывайте эффект на стектрейсы        | В production-коде заинлайненная функция не видна в панике. Для критичных участков — `//go:noinline`              |
| Используйте `GOSSAFUNC` для анализа    | Визуализация SSA помогает понять, почему та или иная функция не была заинлайнина                                 |

> **Зачем это Go-разработчику.** Инлайн — это не магия и не то, что можно игнорировать. Понимание механизма позволяет писать код, который компилятор способен эффективно оптимизировать, и осознанно управлять этим процессом там, где это действительно нужно.
