## 1. Базовые примитивы конкурентности в Go

Прежде чем разбирать конкурентные паттерны, необходимо понимать базовые примитивы, на которых они построены. Если вы уже уверенно работаете с горутинами и каналами, этот раздел можно пропустить.

### Горутина

**Горутина** (goroutine) — это легковесный поток выполнения, управляемый планировщиком Go. Горутина не является потоком ОС: она работает в пространстве пользователя и multiplex'ируется на системные потоки рантаймом Go.

Запуск горутины:

```go
go func() {
    fmt.Println("я работаю параллельно")
}()
```

Ключевые свойства:

* Стартовый размер стека — около 2 КБ, стек растёт и сжимается динамически.
* Можно запустить сотни тысяч горутин без значительных накладных расходов.
* Горутина завершается, когда завершается её функция (или когда программа падает).
* Если основная функция `main` завершается, все горутины принудительно останавливаются.

> **Зачем это Go-разработчику.** Горутина — атомарная единица конкурентности. Любой конкурентный паттерн в Go — это способ организовать взаимодействие горутин.

### Канал

**Канал** (channel) — типизированный туннель для передачи данных между горутинами. Канал можно отправить значение (`ch <- v`), прочитать значение (`v := <-ch`) и закрыть (`close(ch)`).

```go
ch := make(chan int)      // небуферизированный: отправитель ждёт получателя
ch := make(chan int, 10)  // буферизированный: до 10 значений без блокировки отправителя
```

Ключевые свойства:

* Чтение из закрытого канала возвращает нулевое значение типа (проверка через `v, ok := <-ch`).
* Запись в закрытый канал вызывает **panic**.
* Закрытие nil-канала вызывает **panic**.
* Небуферизированный канал синхронизирует отправителя и получателя — это гарантия «встречи».
* Буферизированный канал даёт отправителю «кредит» на N сообщений.

> **Зачем это Go-разработчику.** Канал — центральный инструмент философии Go: «не общайтесь через разделяемую память, разделяйте память через общение». Практически каждый конкурентный паттерн построен на каналах.

### select

**`select`** — конструкция, позволяющая горутине ждать на нескольких канальных операциях одновременно. Выполняется та ветвь, чей канал готов первым:

```go
select {
case v := <-ch1:
    fmt.Println("из ch1:", v)
case ch2 <- 42:
    fmt.Println("отправили в ch2")
case <-ctx.Done():
    fmt.Println("отмена")
}
```

Ключевые свойства:

* Если готовы несколько ветвей — выбирается **случайная**.
* Ветвь `default` выполняется немедленно, если ни один канал не готов (неблокирующий select).
* Чтение из nil-канала блокируется навсегда — это можно использовать для динамического «отключения» ветвей.

> **Зачем это Go-разработчику.** `select` — основа паттернов Fan-in, Cancellation и таймаутов. Без `select` невозможно слушать несколько каналов одновременно.

### sync.WaitGroup

**`sync.WaitGroup`** — счётчик, позволяющий одной горутине дождаться завершения группы других горутин:

```go
var wg sync.WaitGroup
for i := 0; i < 5; i++ {
    wg.Add(1)
    go func() {
        defer wg.Done()
        // работа
    }()
}
wg.Wait() // блокируется, пока все не вызовут Done()
```

Ключевые свойства:

* `Add(n)` увеличивает счётчик на n. Должен вызываться **до** запуска горутины.
* `Done()` уменьшает счётчик на 1. Обычно через `defer`.
* `Wait()` блокируется, пока счётчик не станет нулём.
* WaitGroup нельзя копировать (передавать по значению).

> **Зачем это Go-разработчику.** WaitGroup — фундамент для Worker Pool, Fan-out и ErrGroup. Это простейший способ сказать: «дождись всех».

### sync.Mutex и sync.RWMutex

Когда каналов недостаточно, используется **мьютекс** — примитив исключительного доступа к разделяемым данным:

```go
var mu sync.Mutex
mu.Lock()
// критическая секция
mu.Unlock()
```

`sync.RWMutex` позволяет множественное чтение и эксклюзивную запись.

> **Зачем это Go-разработчику.** Не все задачи конкурентности решаются каналами. Защита `map` от конкурентной записи, разделяемый кэш, счётчики — здесь мьютексы незаменимы. Паттерны вроде Semaphore тоже строятся на мьютексах.

## 2. Введение в конкурентные паттерны

**Конкурентные паттерны** — это типовые архитектурные решения для организации взаимодействия между одновременно выполняющимися единицами работы (горутинами).

В отличие от классических паттернов GoF (банды четырёх), которые решают проблемы ООП, конкурентные паттерны решают проблемы:

* Синхронизации доступа к данным.
* Координации порядка выполнения.
* Обмена данными между горутинами.
* Управления жизненным циклом конкурентных процессов.

Паттерны в этом документе выстроены от простого к сложному. Первые паттерны (Worker Pool, Fan-in, Fan-out, Pipeline) — базовые строительные блоки. Cancellation — сквозной паттерн, пронизывающий все остальные. ErrGroup и Semaphore — надстройки над стандартными примитивами. Generator, Or-Done, Tee и Bridge — специализированные паттерны для узких сценариев.

> **Зачем это Go-разработчику.** Конкурентные паттерны — это не «красивый код», а инструмент предотвращения утечек горутин, гонок данных и неконтролируемой деградации под нагрузкой. Правильно выбранный паттерн делает код не только быстрее, но и безопаснее.

## 3. Worker Pool

**Worker Pool** — это один из фундаментальных конкурентных паттернов,
который решает классическую задачу: как эффективно обработать большое
количество задач, не создавая при этом бесконечное количество горутин.
Вместо того чтобы на каждую задачу запускать новую горутину (что привело
бы к взрывному росту потребления памяти и контекстных переключений), мы
создаем фиксированное количество "рабочих" (воркеров), которые
параллельно выбирают задачи из общего канала и выполняют их.

### Почему это важно

В Go горутины действительно очень легковесны — они весят всего
несколько килобайт, и можно запустить сотни тысяч из них без особых
проблем. Однако это не значит, что неограниченное создание горутин —
хорошая идея. Даже легковесные горутины создают нагрузку на планировщик,
и каждая из них требует определенных ресурсов. Кроме того, если каждая
горутина выполняет, скажем, сетевой запрос к внешнему API, вы можете
легко исчерпать лимиты соединений или создать чрезмерную нагрузку на
сервис, который вы вызываете.

Worker Pool находит идеальный баланс: вы контролируете уровень
параллелизма, ограничивая количество одновременно работающих горутин, но
при этом сохраняете возможность обрабатывать огромное количество задач
последовательно, складывая их в очередь.

### Как это устроено

Типичная реализация Worker Pool строится вокруг трех ключевых
компонентов. Первый — это канал задач, в который отправляются все
задания, которые нужно выполнить. Второй — это фиксированное
количество горутин-воркеров, которые запускаются при старте пула и живут
на протяжении всего времени его работы. Третий — это канал
результатов, куда воркеры отправляют результаты выполнения задач.

Когда вы отправляете задачу в пул, она попадает в канал задач. Каждый
воркер внутри себя имеет бесконечный цикл, в котором он пытается
прочитать новую задачу из этого канала. Как только задача поступает,
воркер забирает ее, выполняет, отправляет результат в канал результатов
и возвращается в состояние ожидания следующей задачи. Если задач нет,
воркер блокируется на чтении из канала и не потребляет процессорное
время — это одна из сильных сторон каналов в Go.

**Важные моменты реализации:**

Один из самых важных моментов при реализации Worker Pool — это
корректное завершение работы. Если вы просто запустите воркеры и
забудете о них, они будут висеть вечно, ожидая задач, которых уже не
будет. Это классическая утечка горутин. Правильное завершение обычно
реализуется через закрытие канала задач или через контекст.

Когда вы закрываете канал задач, все воркеры, которые читают из него,
получат сигнал о закрытии. Цикл чтения из канала в Go имеет удобную
форму: for task := range taskChannel. Этот цикл автоматически
завершится, когда канал будет закрыт. После выхода из цикла воркер может
выполнить финальную очистку и завершиться.

Еще один важный нюанс касается буферизации канала задач. Если у вас есть
канал с буфером, задачи будут накапливаться в нем до тех пор, пока буфер
не заполнится. Только после этого отправка задачи начнет блокироваться,
ожидая освобождения места. Это позволяет сглаживать пиковые нагрузки:
если воркеры временно не справляются с потоком задач, задачи будут
копиться в буфере, а отправитель не будет блокироваться до тех пор, пока
буфер не переполнится. Размер буфера нужно подбирать исходя из ожидаемой
нагрузки и доступной памяти.

### Когда полезно

Этот паттерн становится незаменимым в нескольких типичных ситуациях.
Первая — это обработка большого количества независимых задач,
например, пакетная обработка данных, где нужно прогнать миллион записей
через какую-то функцию.

Вторая — это ограничение нагрузки на внешние ресурсы, такие как базы
данных или API. Если вы делаете HTTP-запросы к внешнему сервису, который
разрешает не более 50 одновременных соединений, пул из 50 воркеров будет
идеальным решением.

Третья ситуация — это управление памятью. Хотя каждая горутина и
легковесна, каждая задача, которую она обрабатывает, может требовать
выделения памяти. Если вы запустите 100 тысяч горутин, каждая из которых
аллоцирует большие буферы, потребление памяти может стать проблемой.
Worker Pool ограничивает количество одновременно живущих в памяти
"активных" обработчиков.

### Практический пример

Представьте сервис, который получает от пользователей ссылки на
изображения и должен скачать эти изображения, изменить их размер и
сохранить в облачное хранилище. Пользователи могут отправлять тысячи
ссылок за раз. Вместо того чтобы на каждую ссылку запускать отдельную
горутину, которая скачивает, обрабатывает и загружает изображение, вы
создаете пул воркеров. Размер пула можно установить, например, в 10
воркеров, что ограничит количество одновременных сетевых подключений до
разумных 10. Это не перегрузит ни ваш сервер, ни внешние сервисы, с
которыми вы взаимодействуете. При этом обработка всех ссылок займет
ровно столько времени, сколько нужно, но в контролируемом режиме.

Такой подход позволяет также легко добавлять прогресс-бары, обработку
ошибок с повторными попытками и другие полезные функции, потому что у
вас есть централизованный канал результатов, через который проходят все
результаты обработки.

Worker Pool часто выступает первой или промежуточной стадией Pipeline. При реализации Worker Pool обязательно использовать Cancellation для остановки воркеров.

## 4. Fan-in (Объединение)

**Fan-in** — это паттерн конкурентного программирования, который
решает задачу объединения данных из нескольких источников в единый
поток. Fan-in берет несколько каналов (или горутин) и объединяет их
выходные данные в один общий канал, создавая для потребителя иллюзию
единого источника.

В мире Go этот паттерн встречается постоянно. Когда вы слушаете
несколько каналов одновременно, используя select, вы на самом деле
реализуете простейшую форму Fan-in. Когда вы запускаете несколько
горутин, каждая из которых ищет данные в своей части массива, а потом
собираете результаты в общий канал —  это тоже Fan-in. Паттерн
настолько естественен для конкурентных программ на Go, что часто
используется интуитивно, без осознания того, что это устоявшееся
архитектурное решение.

### Почему это важно

В основе философии Go лежит принцип "не общайтесь через разделяемую
память, разделяйте память через общение". Каналы — это главный
инструмент этого общения. Но когда в программе появляется несколько
источников данных (например, несколько воркеров, каждый из которых
обрабатывает свою часть задач, или несколько подписок на внешние
события), возникает вопрос: как удобно потреблять данные из всех этих
источников? Потребитель не хочет знать, сколько источников данных
существует и как они организованы. Он хочет получать данные из одного
места.

Fan-in решает эту проблему, создавая абстракцию единого источника.
Потребитель работает с одним каналом, читает из него в цикле, а
сложность агрегации данных из множества источников скрыта внутри
реализации Fan-in. Это делает код чище, уменьшает связность компонентов
и упрощает тестирование.

Кроме того, Fan-in позволяет эффективно использовать ресурсы. Вместо
того чтобы потребитель сам следил за несколькими каналами через
громоздкие конструкции с select (которые становятся все сложнее по мере
роста числа источников), Fan-in выносит эту сложность в отдельный
компонент. Потребитель получает один канал и может обрабатывать данные в
простом цикле for value := range mergedChannel.

### Как это устроено

Существует несколько способов реализации Fan-in, и выбор зависит от
конкретной задачи. Самый распространенный способ основан на
использовании отдельной горутины-мультиплексора, которая запускается
один раз и живет на протяжении всего времени объединения потоков. Эта
горутина внутри себя имеет бесконечный цикл, в котором с помощью select
она слушает все исходные каналы одновременно. Как только в одном из
каналов появляются данные, мультиплексор забирает их и отправляет в
общий выходной канал.

Важно понимать, что мультиплексор должен корректно обрабатывать закрытие
исходных каналов. Если какой-то из каналов закрывается, мультиплексор
должен перестать его слушать, чтобы не вызывать панику при попытке
чтения из закрытого канала. Самый элегантный способ —  использовать
специальный прием с nil-каналами. Когда канал закрывается, вы можете
присвоить ему значение nil, и select перестанет обрабатывать этот
вариант, так как чтение из nil-канала блокируется навсегда.

Более сложные реализации Fan-in могут включать в себя буферизацию, чтобы
сглаживать пиковые нагрузки, или приоритизацию, когда данные из одних
каналов обрабатываются раньше, чем из других. Но в большинстве случаев
простая реализация с select и nil-каналами оказывается достаточной.

### Важные моменты реализации

Один из ключевых моментов при реализации Fan-in —  это управление
жизненным циклом. Мультиплексор должен завершиться ровно в тот момент,
когда завершились все исходные каналы. Если вы забудете закрыть выходной
канал после того, как все входные каналы исчерпаны, потребитель навсегда
зависнет в ожидании данных. Это классическая утечка горутины. Поэтому в
реализации Fan-in обязательно нужно отслеживать, сколько исходных
каналов еще живы, и когда счетчик достигает нуля, закрывать выходной
канал.

Для отслеживания количества живых каналов часто используют
sync.WaitGroup. Мультиплексор запускает для каждого входного канала
отдельную горутину, которая читает из этого канала и пересылает данные в
общий канал. Когда канал закрывается, горутина завершается и уменьшает
счетчик WaitGroup. Отдельная горутина-наблюдатель ждет, когда счетчик
обнулится, и закрывает выходной канал. Этот подход называется "Fan-in
через отдельные горутины" и он особенно удобен, когда количество
источников фиксировано и известно заранее.

Другой важный момент —  это тип данных. Fan-in работает только с
каналами одного типа. Нельзя объединить канал целых чисел с каналом
строк. Если вам нужно объединить данные разных типов, вам придется
использовать общий интерфейс или создать структуру-обертку, которая
будет хранить данные вместе с информацией об источнике.

Третий нюанс касается производительности. В реализации Fan-in через
мультиплексор с select есть потенциальная проблема: если один из каналов
генерирует данные намного быстрее других, он может монополизировать
мультиплексор, и данные из медленных каналов будут обрабатываться с
задержкой. В некоторых сценариях это приемлемо, в других —  нет. Если
важна справедливость, можно использовать более сложные алгоритмы,
например, круговое обслуживание (round-robin) или приоритеты.

### Когда полезен

Первый и самый очевидный сценарий —  это объединение результатов работы
нескольких воркеров. Представьте, что вы разбили большую задачу на части
и запустили несколько горутин, каждая из которых обрабатывает свою
часть. Каждая горутина отправляет результаты в свой канал. Fan-in
объединяет эти каналы, и потребитель получает единый поток результатов в
том порядке, в котором они готовы, а не в порядке запуска воркеров.

Второй сценарий —  это обработка событий из нескольких источников.
Например, вы подписались на сообщения из нескольких очередей RabbitMQ,
или слушаете вебсокеты от нескольких клиентов, или получаете сигналы от
нескольких системных таймеров. Вместо того чтобы писать сложную логику
выбора, из какого источника читать, вы объединяете все источники в один
канал и обрабатываете события в простом цикле.

Третий сценарий —  это реализация паттерна "отмена по первому
успеху". Вы запускаете несколько горутин, которые пытаются решить одну
и ту же задачу разными способами (например, запрашивают данные из
нескольких кэшей или реплик базы данных). Первая горутина, которая
вернет результат, отправляет его в общий канал, а Fan-in позволяет
получить этот результат, после чего остальные горутины можно отменить.

Четвертый сценарий —  это агрегация данных в потоковой обработке. Вы
можете получать данные из нескольких источников (файлы, сетевые
соединения, пользовательский ввод), объединять их в один поток и
применять к нему общую логику обработки. Это делает архитектуру более
модульной: вы можете добавлять новые источники данных, просто подключая
их к Fan-in, не меняя основную логику обработки.

### Практический пример

Представьте, что вы пишете систему мониторинга, которая собирает метрики
из нескольких источников: из Kubernetes API, из Prometheus, из логов
приложений и из пользовательских метрик. Каждый источник работает в
своей горутине и отправляет метрики в свой канал. Вам нужно объединить
все эти метрики в единый поток для записи в базу данных.

Вы создаете функцию MergeMetrics, которая принимает список каналов с
метриками и возвращает один общий канал. Внутри функции вы запускаете
мультиплексор, который слушает все каналы одновременно. Когда приходит
метрика из любого источника, она немедленно отправляется в выходной
канал. Когда все источники завершили работу (их каналы закрыты),
выходной канал закрывается.

Потребитель (сервис записи в базу) работает с одним каналом. Он не
знает, сколько источников метрик существует, не знает, как они устроены,
не заботится о том, добавится ли новый источник завтра. Он просто читает
метрики из канала в цикле и пишет их в базу. Если нужно добавить новый
источник метрик, вы просто добавляете его канал в список при вызове
MergeMetrics, и все продолжает работать без единого изменения в коде
потребителя.

Более того, вы можете добавить в Fan-in дополнительные возможности.
Например, вы можете добавить буферизацию, чтобы сглаживать всплески
нагрузки. Или вы можете добавить фильтрацию, чтобы пропускать только
метрики определенного типа. Или вы можете добавить агрегацию, чтобы
собирать метрики за минуту и отправлять их одним пакетом. Все эти
возможности добавляются в одном месте, не затрагивая ни источники
метрик, ни их потребителя.

### Связь с другими паттернами

Fan-in — обратная сторона Fan-out: вместе они образуют законченную схему «разделил-обработал-собрал». Fan-in часто используется на последней стадии Pipeline для сбора результатов параллельных этапов. Мультиплексор Fan-in должен слушать `ctx.Done()` через Cancellation.

## 5. Fan-out (Разветвление)

**Fan-out** — это паттерн конкурентного программирования, который
решает задачу распределения работы между несколькими параллельными
исполнителями. Если Fan-in объединяет несколько потоков в один, то
Fan-out делает ровно противоположное: он берет один источник данных и
разветвляет его на множество потребителей, каждый из которых
обрабатывает свою часть работы независимо и параллельно.

В Go этот паттерн лежит в основе практически любой параллельной
обработки данных. Когда вы запускаете несколько горутин, каждая из
которых читает из одного канала, вы уже используете Fan-out. Стандартный
пул воркеров, о котором мы говорили ранее, — это частный случай
Fan-out, где входной канал задач распределяется между фиксированным
количеством воркеров.

### Почему важен

Одна из главных причин использовать Fan-out —  это масштабирование
производительности. В современном мире, где у нас есть многоядерные
процессоры, оставлять всю работу на одном ядре —  непозволительная
роскошь. Fan-out позволяет загрузить все доступные ядра, обрабатывая
данные параллельно. Если ваша задача может быть распараллелена, Fan-out
— это тот инструмент, который поможет вам получить линейный прирост
производительности при добавлении новых ядер.

Но дело не только в производительности. Fan-out также помогает управлять
сложностью. Вместо того чтобы писать один огромный монолитный
обработчик, который делает всё сам, вы разбиваете работу на множество
маленьких независимых воркеров. Каждый воркер делает одну простую вещь,
и все они работают параллельно. Это делает код более понятным,
тестируемым и устойчивым к ошибкам: если один воркер упал, остальные
продолжают работать.

Кроме того, Fan-out позволяет контролировать уровень параллелизма. Вы не
запускаете горутину на каждую задачу (что могло бы привести к взрывному
росту потребления ресурсов), а создаете фиксированное количество
воркеров, которые обрабатывают задачи из общей очереди. Это предсказуемо
и стабильно.

### Как устроено

В основе Fan-out лежит простая, но мощная идея: один канал с задачами
читается несколькими горутинами. Каждая горутина живет в своем цикле,
пытаясь получить следующую задачу из общего канала. С точки зрения
планировщика Go, это выглядит так: когда несколько горутин блокируются
на чтении из одного канала, планировщик выбирает одну из них, когда в
канале появляются данные. Это создает естественный механизм
распределения задач — каждая задача достанется той горутине, которая
первой "проснется".

Однако простая схема "один канал, много читателей" имеет один важный
нюанс. Если воркеры не одинаковые (например, одни обрабатывают задачи
быстро, другие медленно), и если задачи поступают неравномерно, может
возникнуть ситуация, когда медленные воркеры создают "бутылочное
горлышко". Быстрые воркеры могут обрабатывать задачи быстрее, чем они
поступают, и простаивать, пока медленные воркеры накапливают очередь.

Для решения этой проблемы часто используется комбинация Fan-out с
дополнительным каналом для обратной связи или более сложные алгоритмы
распределения, такие как "work stealing" (кража работы), когда
простаивающие воркеры могут "украсть" задачи у занятых. Однако в
большинстве случаев простая схема с общим каналом работает отлично.

### Важные детали реализации

Первый и самый важный момент — это обработка закрытия канала. Когда вы
распределяете задачи между несколькими горутинами, каждая из них читает
из одного и того же канала. Если вы просто закроете канал, все горутины
получат сигнал о закрытии. Нужно быть уверенным, что канал закрывается
только после того, как все задачи отправлены, и что все горутины
корректно завершаются после закрытия. Обычно для этого используется
sync.WaitGroup, которая позволяет дождаться завершения всех воркеров.

Второй момент — это количество воркеров. Сколько их должно быть? Ответ
зависит от характера задач. Если задачи в основном выполняют операции
ввода-вывода (сетевые запросы, чтение с диска), вы можете запустить
сотни или даже тысячи воркеров, так как большую часть времени они будут
ждать. Если же задачи требуют интенсивных вычислений (CPU-bound),
оптимальное количество воркеров примерно равно количеству ядер
процессора. Запуск большего количества воркеров приведет к лишним
переключениям контекста и даже может снизить производительность.

Третий момент — это балансировка нагрузки. В простой схеме с общим
каналом задачи распределяются между воркерами непредсказуемо. Если вам
нужно гарантировать равномерное распределение (например, когда задачи
имеют сильно разную стоимость выполнения), возможно, стоит рассмотреть
более сложные стратегии, такие как отправка задач в конкретные каналы
воркеров по кругу (round-robin) или использование очереди с
приоритетами.

Четвертый момент — это обработка ошибок. Когда один из воркеров
завершается с ошибкой, что делать с остальными? Нужно ли останавливать
весь Fan-out? Нужно ли продолжать обработку? Ответ зависит от конкретной
задачи. Иногда вы хотите, чтобы ошибка в одном воркере не влияла на
другие. Иногда, наоборот, вы хотите отменить все воркеры при первой же
ошибке (например, если вы ищете что-то и нашли). Для управления такими
сценариями часто используется контекст с отменой.

### Когда полезно

Первый сценарий — это пакетная обработка данных. Представьте, что у
вас есть миллион записей в файле, и каждую нужно преобразовать и
сохранить в базу данных. Обрабатывать их последовательно — значит
ждать часами. С помощью Fan-out вы запускаете 100 воркеров, каждый
читает записи из общего канала и обрабатывает их параллельно. Время
обработки сокращается в десятки раз.

Второй сценарий — это параллельные запросы к внешним сервисам. Вам
нужно получить данные о пользователе из пяти разных микросервисов.
Вместо того чтобы делать запросы последовательно (что займет сумму
времен всех запросов), вы запускаете пять горутин, каждая делает свой
запрос, и все они работают параллельно. Общее время обработки становится
равным времени самого медленного запроса.

Третий сценарий — это обработка событий в реальном времени. Вы
получаете поток событий (например, клики пользователей), и каждое
событие нужно обработать: обогатить данными, проверить на валидность,
сохранить в хранилище. Fan-out позволяет обрабатывать события
параллельно, не накапливая очередь.

Четвертый сценарий — это реализация паттерна "конвейер" (pipeline).
В конвейере каждая стадия обработки может быть реализована через
Fan-out. Например, первая стадия читает данные из файла и отправляет их
в канал. Вторая стадия (Fan-out) запускает несколько воркеров, которые
обрабатывают данные параллельно. Третья стадия собирает результаты и
записывает их в базу. Такая архитектура позволяет легко масштабировать
каждую стадию независимо.

### Практический пример

Представьте, что вы пишете сервис обработки видео. Пользователь
загружает видео, и его нужно конвертировать в несколько форматов: MP4
для веба, MOV для редактирования, и создать превью. Это идеальная задача
для Fan-out.

Вы создаете канал задач, куда отправляете информацию о загруженном
видео. Затем запускаете три воркера: один для конвертации в MP4, второй
для конвертации в MOV, третий для создания превью. Каждый воркер читает
из одного и того же канала, но обрабатывает задачу по-своему. Когда
видео загружено, вы отправляете одну задачу в канал, и все три воркера
получают её и начинают обработку параллельно.

Но вот в чем важный момент: если конвертация в MP4 занимает 10 минут, в
MOV — 5 минут, а создание превью — 1 минуту, все три процесса идут
параллельно. Общее время обработки видео будет равно 10 минутам (самому
долгому процессу), а не сумме всех времен. Если бы вы обрабатывали
последовательно, пользователь ждал бы 16 минут.

Более того, вы можете легко масштабировать этот подход. Если конвертация
в MP4 становится узким местом, вы можете запустить не один, а три
воркера для MP4, и они будут обрабатывать видео параллельно. Вам не
нужно менять код остальных воркеров. Каждый воркер просто читает из
канала, и добавление новых воркеров увеличивает пропускную способность.

А что если во время обработки пользователь удалил видео? Вы отменяете
контекст, и все воркеры получают сигнал об отмене. Каждый воркер может
корректно завершить свою работу (например, удалить временные файлы) и
остановиться. Без Fan-out вам пришлось бы передавать сигнал отмены
каждому воркеру вручную.

### Связь с другими паттернами

Fan-out — естественный способ распараллелить стадию внутри Pipeline. Комбинация Fan-out + Fan-in даёт map-reduce стиль обработки. Количество воркеров в Fan-out можно ограничить через Semaphore, а групповую отмену реализовать через ErrGroup.

## 6. Pipeline (Конвейер)

**Pipeline** — это один из самых элегантных и мощных паттернов
конкурентного программирования в Go, который позволяет организовать
обработку данных в виде последовательности этапов, где выход одного
этапа становится входом для следующего.

В Go Pipeline строится на основе каналов и горутин. Каждый этап
конвейера — это функция, которая принимает канал на чтение, читает из
него данные, обрабатывает их и отправляет в выходной канал. Этапы
соединяются последовательно: выходной канал первого этапа передается на
вход второму, выход второго — на вход третьему, и так далее. Благодаря
тому, что каждый этап работает в своей горутине, данные проходят через
конвейер параллельно: пока второй этап обрабатывает первый элемент,
первый этап уже может обрабатывать второй, а третий — ждать результат
от второго.

Этот паттерн настолько органично вписывается в философию Go, что его
использование стало практически стандартом де-факто для организации
сложной обработки данных. Стандартная библиотека Go активно использует
этот подход, а многие популярные библиотеки для потоковой обработки
данных построены именно на этом паттерне.

### Почему важен

Pipeline решает сразу несколько важных задач, которые возникают при
разработке сложных систем обработки данных. Первая и главная — это
разделение ответственности. Каждый этап конвейера отвечает ровно за одну
операцию. Один этап читает данные из файла, другой парсит их, третий
валидирует, четвертый обогащает данными из внешнего API, пятый сохраняет
в базу данных. Каждый из этих этапов можно разрабатывать, тестировать и
изменять независимо от остальных. Если вам нужно добавить новый этап
обработки, вы просто встраиваете его в конвейер, не меняя существующие
этапы.

Вторая важная задача — это эффективное использование ресурсов. В
традиционном последовательном подходе данные обрабатываются по принципу
"прочитал весь файл -> распарсил всё -> сохранил всё". Это требует
огромного количества памяти, потому что все данные должны быть загружены
целиком. В Pipeline данные текут маленькими порциями. Вы прочитали одну
запись, передали её на парсинг, пока парсер работает, вы уже читаете
следующую запись. Памяти требуется ровно столько, сколько нужно для
хранения небольшого буфера, а не всего датасета.

Третья задача — это автоматический параллелизм. В Pipeline каждый этап
работает в своей горутине, и они выполняются параллельно. Если у вас
четырехъядерный процессор, и конвейер состоит из четырех этапов, каждый
этап может выполняться на своем ядре. Более того, вы можете
распараллелить отдельные этапы, применив Fan-out внутри этапа, создавая
конвейеры с внутренним параллелизмом.

Четвертая задача — это устойчивость к ошибкам. В Pipeline вы можете
обрабатывать ошибки на каждом этапе, решая, останавливать ли весь
конвейер или пропускать "испорченные" данные. Вы можете добавить этап
валидации, который отбрасывает некорректные данные, и остальные этапы
будут работать только с чистыми данными.

### Как устроено

Pipeline строится из функций-этапов, каждая из которых имеет примерно
одинаковую сигнатуру: принимает канал на чтение (обычно <-chan T) и
возвращает канал на запись (обычно chan U). Типы данных на входе и
выходе могут быть разными — в этом вся гибкость конвейера. Например,
этап чтения может возвращать канал строк, этап парсинга принимает канал
строк и возвращает канал структур, этап валидации принимает канал
структур и возвращает канал отфильтрованных структур.

Каждый этап внутри себя запускает горутину, которая в цикле читает из
входного канала, обрабатывает данные и отправляет результат в выходной
канал. Когда входной канал закрывается, этап закрывает свой выходной
канал и завершается. Это создает естественный механизм распространения
сигнала завершения: когда самый первый этап закрывает свой выходной
канал (потому что данные закончились), следующий этап получает сигнал о
закрытии, завершает свою работу и закрывает свой канал, и так далее по
цепочке.

Важно понимать, что Pipeline не требует, чтобы все этапы работали с
одинаковой скоростью. Если один этап работает медленнее других, данные
будут накапливаться в буфере канала между этапами. Размер этого буфера
можно настроить, чтобы сглаживать пиковые нагрузки. Если буфер
переполняется, отправитель блокируется, автоматически создавая
backpressure — механизм обратного давления, который замедляет более
быстрые этапы, чтобы они не перегружали медленные.

### Важные моменты реализации

Первый и самый важный момент — это управление закрытием каналов. В
Pipeline есть простое правило: канал закрывает тот, кто в него пишет.
Это означает, что каждый этап отвечает за закрытие своего выходного
канала. Никто извне не должен закрывать каналы этапов. Если вы нарушите
это правило, вы рискуете получить панику при записи в закрытый канал
или, что еще хуже, утечку горутин, которые навсегда зависнут в ожидании
данных из закрытого канала.

Второй момент — это обработка ошибок. Классический Pipeline
предполагает, что все этапы всегда успешно обрабатывают данные. В
реальности это не так. Есть несколько подходов к обработке ошибок.
Первый подход — использовать отдельный канал для ошибок, который идет
параллельно основному конвейеру. Второй подход --- передавать ошибки
вместе с данными, используя структуру-обертку, которая содержит либо
данные, либо ошибку. Третий подход — останавливать весь конвейер при
первой ошибке, используя контекст с отменой. Выбор подхода зависит от
конкретной задачи.

Третий момент — это контекст. В реальных системах Pipeline должен
уметь останавливаться по сигналу отмены. Например, пользователь закрыл
приложение, или истек таймаут. Каждый этап должен слушать не только
входной канал, но и канал ctx.Done(), чтобы иметь возможность прервать
обработку досрочно. Это особенно важно для этапов, которые работают с
внешними сервисами или выполняют долгие операции.

Четвертый момент — это буферизация каналов. Выбор размера буфера для
каналов между этапами может сильно влиять на производительность и
поведение системы. Если буфер слишком маленький, быстрые этапы будут
часто блокироваться, ожидая, пока медленные этапы освободят место. Если
буфер слишком большой, вы рискуете потребить много памяти, особенно если
данные большие. Оптимальный размер буфера часто находится
экспериментально, исходя из профиля нагрузки.

### Когда полезен

Первый и самый очевидный сценарий — это обработка больших файлов или
потоков данных. Логи файлов, дампы баз данных, большие CSV-файлы — всё
это идеальные кандидаты для Pipeline. Вы читаете файл построчно (один
этап), парсите каждую строку в структуру (второй этап), валидируете
данные (третий этап), обогащаете данными из кэша или внешнего API
(четвертый этап), и сохраняете в базу данных (пятый этап). Все это
происходит в потоковом режиме, без загрузки всего файла в память.

Второй сценарий — это ETL-процессы (Extract, Transform, Load). В
современной аналитике и обработке данных Pipeline стал стандартом
де-факто. Вы извлекаете данные из источника, трансформируете их через
цепочку преобразований, и загружаете в хранилище. Каждое преобразование
— это отдельный этап, который легко тестировать и заменять.

Третий сценарий — это обработка медиафайлов. Изображения, видео, аудио
часто проходят через цепочку преобразований: загрузка, проверка формата,
конвертация, создание превью, добавление водяных знаков, сохранение.
Pipeline позволяет организовать эту обработку как последовательность
независимых этапов, которые могут выполняться параллельно.

Четвертый сценарий — это реализация сложных бизнес-процессов.
Например, обработка заказа в интернет-магазине может проходить через
этапы: проверка наличия товара, резервирование, списание средств,
формирование накладной, отправка на склад, уведомление клиента. Каждый
этап может выполняться в своей горутине, и если один этап временно
недоступен (например, складская система не отвечает), данные будут
накапливаться в буфере, не блокируя остальные операции.

### Практический пример

Представьте, что вы пишете систему для обработки логов. Логи поступают в
огромных объемах — миллионы строк в час. Вам нужно: читать логи из
файла, парсить каждую строку в структуру лога, обогащать лог геоданными
по IP-адресу, фильтровать только ошибки и предупреждения, агрегировать
их по минутам, и сохранять в базу данных.

Вы строите Pipeline из следующих этапов:

1. **Reader**. Читает файл построчно и отправляет строки в канал. Он работает быстро, так как чтение с диска — операция ввода-вывода.
2. **Parser**. Получает строки, парсит их в структуру LogEntry. Этот этап может быть CPU-интенсивным из-за парсинга, поэтому вы запускаете внутри него пул из 4 воркеров (Fan-out). Распарсенные структуры отправляются в общий канал.
3. **Enricher**. Получает LogEntry, извлекает IP-адрес, делает запрос к гео-сервису (внешнее API), добавляет информацию о местоположении. Это I/O-интенсивный этап, поэтому вы запускаете 100 воркеров, так как большая часть времени уходит на ожидание ответа от API.
4. **Filter**. Получает обогащенные записи, пропускает только те, у которых уровень лога — error или warning. Остальные просто отбрасываются (не отправляются в выходной канал).
5. **Aggregator**. Накапливает записи за 1-минутные окна. Когда окно закрывается, отправляет агрегированные данные в выходной канал.
6. **Writer**. Получает агрегированные данные и сохраняет их в базу данных. Этот этап тоже I/O-интенсивный, поэтому вы используете пул подключений к базе.

Что происходит в этой системе? Пока Writer сохраняет в базу
агрегированные данные за первую минуту, Aggregator уже собирает данные
за вторую минуту, Filter и Enricher обрабатывают новые записи, а Reader
продолжает читать файл. Все шесть этапов работают параллельно, и
производительность системы ограничена только самым медленным этапом.
Если база данных начинает тормозить, Writer замедляется, начинает
заполняться буфер перед ним, и через механизм обратного давления
замедляются и предыдущие этапы. Система саморегулируется, не допуская
перегрузки.

Более того, если гео-сервис временно недоступен, Enricher начинает
накапливать записи в своем буфере. Если буфер переполняется, Parser
начинает ждать, и чтение файла замедляется. Это позволяет системе
пережить временные сбои без потери данных. А если пользователь решает
остановить обработку, вы просто отменяете контекст, и все этапы получают
сигнал о завершении, корректно закрывая свои каналы и освобождая
ресурсы.

### Пример

```go
func gen(nums ...int) <-chan int {
    out := make(chan int)
    go func() {
        defer close(out)
        for _, n := range nums {
            out <- n
        }
    }()
    return out
}

func square(in <-chan int) <-chan int {
    out := make(chan int)
    go func() {
        defer close(out)
        for n := range in {
            out <- n * n
        }
    }()
    return out
}

func main() {
    c := gen(1, 2, 3, 4, 5)
    out := square(c)
    for v := range out {
        fmt.Println(v)
    }
}
```

### Связь с другими паттернами

Pipeline — самый композируемый паттерн: каждая стадия может быть Worker Pool, Fan-out, Fan-in или Generator. Pipeline немыслим без Cancellation — каждый этап обязан слушать `ctx.Done()`. Generator часто является первой стадией Pipeline, а Fan-in — последней.

## 7. Cancellation (Отмена)

**Cancellation** — это паттерн конкурентного программирования, который
решает одну из самых сложных проблем в асинхронных системах: как
корректно остановить выполняющуюся работу, когда результат больше не
нужен.

В Go Cancellation реализуется через механизм контекста
(context.Context), который стал стандартом де-факто для передачи
сигналов отмены и сквозных данных через границы горутин. Контекст —
это не просто флаг "остановиться", это целая инфраструктура, которая
позволяет строить деревья отмены, устанавливать таймауты, передавать
значения и, самое главное, делать всё это безопасно и согласовано на
всех уровнях системы.

Этот паттерн настолько важен, что без него невозможно представить
production-приложение на Go. Любой HTTP-сервер, любое приложение с
длительными операциями, любой код, работающий с внешними ресурсами,
обязан правильно обрабатывать отмену. Без этого ресурсы будут утекать,
горутины — зависать, а приложения — падать под нагрузкой.

### Почему важен

В Go горутины дешевы, но они не бесплатны. Каждая горутина, которая
продолжает работать после того, как ее результат стал не нужен, — это
утечка ресурсов. Она потребляет память, может держать открытыми сетевые
соединения, файловые дескрипторы, блокировки в базе данных. Если таких
горутин накапливается много, приложение начинает замедляться, а затем
падает с ошибкой "out of memory". Cancellation — это механизм,
который позволяет этого избежать.

Но проблема не только в утечках. Cancellation также влияет на
пользовательский опыт. Представьте веб-сервер, который обрабатывает
запрос 30 секунд, хотя клиент уже разорвал соединение через 5 секунд.
Сервер продолжает работать, тратя ресурсы на ответ, который никто не
получит. Если таких запросов много, сервер быстро исчерпает свои
возможности. Правильная отмена позволяет прекращать обработку сразу, как
только становится понятно, что результат не нужен.

Кроме того, Cancellation делает систему более отзывчивой. Если у вас
есть операция с таймаутом (например, запрос к внешнему API, который
должен ответить за 1 секунду), без механизма отмены вы будете ждать
ответа вечно. С отменой вы можете прервать операцию по истечении
таймаута и вернуть пользователю понятную ошибку, а не заставлять его
ждать неизвестно сколько.

### Как устроено

В основе Cancellation в Go лежит пакет context. Контекст — это
неизменяемая структура, которая может быть отменена. Когда вы создаете
корневой контекст через context.Background() или context.TODO(), вы
получаете контекст, который никогда не отменяется сам по себе. Но вы
можете создать производные контексты с помощью context.WithCancel,
context.WithTimeout или context.WithDeadline.

context.WithCancel возвращает новый контекст и функцию cancel. Когда вы
вызываете эту функцию, все горутины, которые слушают канал ctx.Done(),
получают сигнал об отмене. Важно понимать, что отмена распространяется
вниз по дереву: если вы отменили родительский контекст, все дочерние
контексты тоже отменяются. Это позволяет отменять целые группы операций
одним вызовом.

context.WithTimeout и context.WithDeadline создают контексты, которые
отменяются автоматически по истечении заданного времени. Под капотом они
используют ту же механику WithCancel, только функцию cancel вызывает
внутренний таймер.

Самый важный метод контекста — это Done(), который возвращает канал,
закрывающийся в момент отмены. В каждой горутине, которая выполняет
потенциально долгую операцию, должен быть select, который слушает этот
канал:

```go
select {
case result := <-someChan:
    return result

case <-ctx.Done():
    return ctx.Err() // context.Canceled  или  context.DeadlineExceeded
}
```

Этот паттерн позволяет горутине корректно завершиться, не дожидаясь
окончания долгой операции, если пришел сигнал отмены.

### Важные моменты реализации

Первый и самый важный момент --- контекст всегда должен передаваться
явно первым аргументом функции. Это не просто конвенция, это требование,
которое делает код читаемым и предсказуемым. Функция, которая принимает
контекст, обязана либо использовать его для отмены, либо передавать
дальше. Если вы видите функцию, которая не принимает контекст, но
выполняет долгие операции, это потенциальная проблема.

Второй момент — никогда не храните контекст в структурах. Контекст
должен передаваться по цепочке вызовов, а не сохраняться как поле
структуры. Исключение составляют структуры, которые являются частью
публичного API (например, http.Request содержит контекст), но даже в
этом случае контекст живет в рамках одного запроса. Хранение контекста в
долгоживущих структурах приводит к утечкам памяти и путанице с тем,
какой контекст сейчас активен.

Третий момент — функция отмены должна вызываться ровно один раз, и
обязательно должна быть вызвана. Если вы создали контекст через
WithCancel, вы обязаны вызвать возвращенную функцию cancel, когда работа
с этим контекстом закончена. Иначе контекст и все связанные с ним
горутины будут висеть до завершения родительского контекста, что может
привести к утечкам. Обычно cancel вызывается через defer сразу после
создания контекста:

```go
ctx, cancel := context.WithTimeout(parentCtx, 5*time.Second)
defer cancel() // гарантирует, что ресурсы будут освобождены
```

Четвертый момент — контекст не предназначен для передачи произвольных
данных. В контекст можно класть значения через context.WithValue, но это
должно использоваться только для сквозных данных, которые действительно
нужны всем уровням обработки (например, ID запроса для трассировки).
Использование контекста для передачи бизнес-данных делает код хрупким и
неочевидным.

Пятый момент — проверка отмены должна происходить регулярно. Если ваша
операция состоит из нескольких шагов, вы должны проверять ctx.Done()
между шагами. Проверка только в начале и в конце недостаточна —
операция может быть отменена в середине, и вы должны иметь возможность
прерваться как можно раньше, чтобы не тратить ресурсы.

### Когда полезен

Первый сценарий — это обработка HTTP-запросов. Каждый HTTP-запрос в Go
имеет свой контекст, который отменяется, когда клиент разрывает
соединение или когда истекает таймаут сервера. Если вы в своем
обработчике запускаете горутины или делаете запросы к внешним сервисам,
вы обязаны передавать этот контекст дальше. Тогда при отмене запроса вся
цепочка операций остановится.

Второй сценарий — это работа с базами данных. Практически все
современные драйверы баз данных в Go поддерживают контекст. Вы можете
передать контекст с таймаутом в QueryContext или ExecContext, и запрос
будет прерван, если превысит лимит времени. Это защищает вас от
"висящих" запросов, которые могут заблокировать пул соединений.

Третий сценарий — это вызовы внешних API. Когда вы делаете
HTTP-запрос, вы можете создать контекст с таймаутом и передать его в
http.NewRequestWithContext. Если внешний сервис не отвечает в течение
заданного времени, запрос будет прерван, и вы сможете обработать эту
ситуацию, не дожидаясь системного таймаута TCP, который может составлять
минуты.

Четвертый сценарий — это управление жизненным циклом горутин. В
сложных системах вы часто запускаете фоновые горутины, которые должны
работать всё время работы приложения. При завершении приложения
(например, при получении сигнала SIGTERM) вам нужно корректно остановить
все эти горутины. Корневой контекст, который отменяется при завершении,
позволяет сделать это элегантно.

Пятый сценарий — это реализация таймаутов для пользовательских
операций. Вы можете дать пользователю возможность отменить операцию
через интерфейс (например, кнопка "Отмена" в веб-интерфейсе). Эта
отмена должна распространиться через весь стек обработки вплоть до базы
данных.

### Практический пример

Представьте, что вы пишете API-шлюз, который принимает запросы от
клиентов, агрегирует данные из нескольких микросервисов и возвращает
результат. Клиент может разорвать соединение в любой момент, и каждый
микросервис имеет свой SLA с таймаутами. Без правильной отмены это был
бы кошмар.

Вы получаете HTTP-запрос. В объекте http.Request уже есть контекст,
который отменится, если клиент закроет соединение. Вы создаете дочерний
контекст с таймаутом, равным вашему собственному SLA (например, 3
секунды), и передаете его во все вызовы микросервисов.

Параллельно вы запускаете горутины для каждого микросервиса, используя
errgroup с контекстом. Если один из микросервисов вернул ошибку или
превысил таймаут, вы отменяете весь errgroup, и все остальные горутины
получают сигнал об отмене. Если клиент разорвал соединение, контекст
запроса отменяется, и вы тоже отменяете все операции.

Теперь представьте, что один из микросервисов завис и не отвечает. Через
3 секунды срабатывает ваш таймаут, вы отменяете контекст, и все
горутины, которые ждут ответа от этого микросервиса, видят ctx.Done() и
завершаются. Вы возвращаете клиенту ошибку 504 Gateway Timeout. Если бы
не было отмены, горутины висели бы на этом запросе гораздо дольше
(возможно, минуты), и под нагрузкой вы бы исчерпали все ресурсы.

А что, если микросервис, который завис, — это не единственный вызов? У
вас их 5. Без отмены вы ждали бы все 5, даже если один уже точно не
ответит. С отменой вы прерываете все ожидания сразу, как только
понимаете, что результат уже не может быть успешным.

### Пример

```go
func doWork(ctx context.Context) error {
    for {
        select {
        case <-ctx.Done():
            return ctx.Err()
        default:
            // полезная работа
            time.Sleep(100 * time.Millisecond)
        }
    }
}

func main() {
    ctx, cancel := context.WithTimeout(context.Background(), 2*time.Second)
    defer cancel()

    go func() {
        if err := doWork(ctx); err != nil {
            fmt.Println("work stopped:", err)
        }
    }()

    <-ctx.Done()
    fmt.Println("main: context done")
}
```

### Связь с другими паттернами

Cancellation — сквозной паттерн. Он пронизывает все остальные: Worker Pool отменяется через контекст, Fan-in слушает `ctx.Done()` в мультиплексоре, Pipeline прерывает цепочку этапов. ErrGroup — это Cancellation + WaitGroup в одном флаконе. Любой паттерн без Cancellation — это потенциальная утечка горутин.

## 8. ErrGroup (Группа ошибок)

**ErrGroup** — это надстройка над `sync.WaitGroup` из пакета `golang.org/x/sync/errgroup`, которая добавляет две критически важные возможности: отслеживание первой ошибки и автоматическую отмену всех горутин в группе при возникновении ошибки. Это один из самых полезных паттернов для сценариев, где вы запускаете несколько параллельных операций и хотите либо дождаться всех, либо остановиться при первой же неудаче.

### Почему это важно

`sync.WaitGroup` решает задачу «дождаться всех», но не отвечает на два вопроса: что делать, если одна из горутин упала? И как отменить остальные, если одна уже точно не справилась? Без ErrGroup разработчик вынужден вручную городить каналы для ошибок, флаги отмены и синхронизацию — код разрастается, становится хрупким.

ErrGroup даёт ответ из коробки: первая ошибка автоматически отменяет контекст для всех горутин группы, а `Wait()` возвращает эту ошибку. Это радикально упрощает код параллельных операций с единой точкой отказа.

### Как устроено

Пакет `golang.org/x/sync/errgroup` предоставляет лаконичный API:

```go
import "golang.org/x/sync/errgroup"

g, ctx := errgroup.WithContext(ctx)

g.Go(func() error {
    // работа с ctx — если другая горутина вернула ошибку, ctx отменён
    return doWork(ctx)
})

g.Go(func() error {
    return doOtherWork(ctx)
})

if err := g.Wait(); err != nil {
    // первая ошибка из любой горутины
}
```

Ключевая механика: `errgroup.WithContext(ctx)` возвращает группу `g` и дочерний контекст `ctx`. Когда любая из горутин, запущенных через `g.Go()`, возвращает ненулевую ошибку, ErrGroup:

1. Запоминает эту ошибку (первую — остальные игнорируются).
2. Вызывает `cancel()` для дочернего контекста.
3. Все остальные горутины группы, если они слушают `ctx.Done()`, получают сигнал отмены.
4. `g.Wait()` дожидается завершения всех горутин и возвращает запомненную ошибку.

Важно: `g.Wait()` ждёт **всех** горутин, даже после отмены. Если горутина игнорирует `ctx.Done()`, она продолжит работу — ErrGroup не убивает горутины принудительно.

### Важные моменты реализации

**Всегда слушайте контекст.** Горутины, запущенные через `g.Go()`, обязаны проверять `ctx.Done()` и завершаться при отмене. Если горутина игнорирует контекст, она продолжит работу даже после ошибки в соседней горутине — это бесполезная трата ресурсов и потенциальная утечка.

**Ограничение параллелизма.** ErrGroup сам по себе не ограничивает количество одновременно запущенных горутин. Если вы в цикле вызываете `g.Go()` тысячи раз, вы запустите тысячи горутин. Для ограничения используйте `errgroup.SetLimit(n)` (доступен с Go 1.22 в `golang.org/x/sync`):

```go
g.SetLimit(10) // не более 10 горутин одновременно
for _, item := range items {
    item := item
    g.Go(func() error {
        return process(ctx, item)
    })
}
```

**Паника в горутине.** ErrGroup перехватывает панику и превращает её в ошибку, возвращаемую из `Wait()`. Приложение не падает, но это не повод игнорировать паники — они сигнализируют о багах.

**Wait вызывается один раз.** Повторный вызов `Wait()` после завершения всех горутин вернёт `nil`, даже если первая попытка вернула ошибку. Не пытайтесь переиспользовать группу — создавайте новую.

### Когда полезен

* **Параллельные запросы к нескольким источникам.** Агрегация данных из микросервисов, баз данных, API. Если один источник недоступен, нет смысла ждать остальные — ErrGroup отменит их.
* **Запуск фоновых сервисов при старте приложения.** База данных, кэш, очередь сообщений должны стартовать все. Если один не смог — приложение не должно запускаться.
* **Конкурентная обработка с досрочным выходом.** Поиск по нескольким индексам: первый нашедший результат отменяет остальных.
* **Пакетные операции с ограничением параллелизма.** `SetLimit` + ErrGroup — это Worker Pool с автоматической отменой.

### Практический пример

Представьте API-шлюз, который при получении запроса должен собрать данные из трёх микросервисов: профиль пользователя, список заказов и рекомендации. Каждый вызов — отдельная горутина. Если сервис заказов лежит и возвращает ошибку, рекомендации уже не нужны — вы всё равно вернёте клиенту ошибку.

```go
func aggregateUserData(ctx context.Context, userID string) (*UserData, error) {
    g, ctx := errgroup.WithContext(ctx)
    g.SetLimit(3)

    var profile *Profile
    var orders []Order
    var recs []Recommendation

    g.Go(func() error {
        var err error
        profile, err = fetchProfile(ctx, userID)
        return err
    })
    g.Go(func() error {
        var err error
        orders, err = fetchOrders(ctx, userID)
        return err
    })
    g.Go(func() error {
        var err error
        recs, err = fetchRecommendations(ctx, userID)
        return err
    })

    if err := g.Wait(); err != nil {
        return nil, fmt.Errorf("aggregate: %w", err)
    }
    return &UserData{Profile: profile, Orders: orders, Recs: recs}, nil
}
```

Без ErrGroup вам пришлось бы вручную создавать канал для ошибок, `sync.WaitGroup`, отдельный контекст с `cancel` и логику «кто первый упал — отменяем всех». ErrGroup делает это в 15 строк.

### Связь с другими паттернами

ErrGroup — это Cancellation + WaitGroup в одном флаконе. С `SetLimit` он превращается в Worker Pool с автоматической отменой. В Pipeline ErrGroup удобен для параллельных стадий. При старте приложения ErrGroup часто комбинируется с Semaphore для ограничения ресурсов.

## 9. Semaphore (Семафор)

**Семафор** — это паттерн для ограничения количества одновременно выполняющихся операций. В отличие от Worker Pool, который ограничивает количество воркеров, обрабатывающих задачи из очереди, семафор ограничивает количество одновременно активных операций в **любом месте кода**. Это более гибкий механизм, который можно применять точечно, независимо от того, как организованы ваши горутины.

### Почему это важно

Worker Pool хорош, когда у вас однородные задачи и единая точка входа (канал задач). Но в реальном приложении «тяжёлые» операции разбросаны по кодовой базе: один обработчик шлёт запрос к API, другой читает большой файл, третий выполняет ресурсоёмкое вычисление. Все они работают в разных горутинах, запущенных из разных мест. Вы не можете загнать их в один канал задач. Семафор решает именно эту проблему — он действует как глобальный «счётчик пропускной способности», через который проходят **все** операции, независимо от их происхождения.

### Как устроено

**Канальный семафор (простая реализация):**

Буферизированный канал размера N выступает как «пул токенов»:

```go
sem := make(chan struct{}, 10) // не более 10 одновременных операций

sem <- struct{}{}        // занять токен (блокируется, если все 10 заняты)
defer func() { <-sem }() // освободить токен
// ... выполняем операцию
```

Плюсы: ноль зависимостей, очевидный код. Минусы: нельзя захватить больше одного токена за раз, нет таймаута на захват.

**Взвешенный семафор (golang.org/x/sync/semaphore):**

Пакет `golang.org/x/sync/semaphore` предоставляет `semaphore.Weighted`:

```go
import "golang.org/x/sync/semaphore"

sem := semaphore.NewWeighted(100) // общий «бюджет» — 100 единиц

if err := sem.Acquire(ctx, 10); err != nil {
    // контекст отменён до того, как токен получен
    return err
}
defer sem.Release(10)
// ... выполняем операцию, «весящую» 10 единиц
```

Каждая операция захватывает столько единиц, сколько ей нужно. Маленький HTTP-запрос — 1 единица. Загрузка файла на 100 МБ — 50 единиц. Это даёт точный контроль над потреблением ресурсов.

### Важные моменты реализации

**Всегда освобождайте токен через&#x20;****`defer`****.** Если вы захватили токен, но из-за паники или раннего `return` не освободили его, семафор со временем «иссякнет», и все новые операции навсегда заблокируются. Это deadlock на уровне приложения.

**Выбор размера семафора.** Слишком маленький размер душит производительность; слишком большой сводит защиту на нет. Оптимальный размер определяется эмпирически:

* Для внешних API — читайте документацию сервиса (rate limit).
* Для базы данных — размер пула соединений.
* Для CPU-bound операций — `runtime.NumCPU()`.

**Семафор ≠ мьютекс.** Семафор ограничивает **количество параллельных доступов**, мьютекс даёт **эксклюзивный доступ**. Не используйте семафор размера 1 вместо `sync.Mutex` — это разные инструменты с разной семантикой.

**Таймаут на захват.** Взвешенный семафор принимает `context.Context`, поэтому захват токена можно ограничить по времени:

```go
ctx, cancel := context.WithTimeout(ctx, 5*time.Second)
defer cancel()
if err := sem.Acquire(ctx, 1); err != nil {
    return fmt.Errorf("не дождались токена: %w", err)
}
```

### Когда полезен

* **Rate limiting внешних API.** Сторонний сервис разрешает не более 50 одновременных соединений. Семафор на 50 гарантирует, что вы никогда не превысите лимит, независимо от того, из скольких мест кода идут запросы.
* **Защита от перегрузки.** Ваше приложение под нагрузкой создаёт слишком много горутин, каждая из которых аллоцирует память. Семафор ограничивает количество одновременно активных «тяжёлых» операций.
* **Управление ограниченными ресурсами.** Пул соединений с базой данных, количество открытых файловых дескрипторов, лимит на одновременные записи на диск.
* **Контроль смешанных нагрузок.** Когда в одном приложении сосуществуют лёгкие запросы (быстро) и тяжёлые (медленно), семафор не даёт «тяжёлым» операциям задавить «лёгкие».

### Практический пример

Представьте сервис конвертации документов. Пользователи загружают файлы, сервис конвертирует их в PDF. Конвертация — CPU-тяжёлая операция: каждый запуск `libreoffice --headless` потребляет \~500 МБ RAM и 100% ядра на время работы. На сервере 8 ядер и 16 ГБ RAM.

Без семафора при пиковой нагрузке в 100 одновременных конвертаций приложение уйдёт в swap и упадёт по OOM. С семафором:

```go
sem := semaphore.NewWeighted(6) // 6 конвертаций одновременно (оставляем 2 ядра системе)

func convertHandler(w http.ResponseWriter, r *http.Request) {
    if err := sem.Acquire(r.Context(), 1); err != nil {
        http.Error(w, "server overloaded", http.StatusServiceUnavailable)
        return
    }
    defer sem.Release(1)

    pdf, err := convertToPDF(r.Context(), r.Body)
    if err != nil {
        http.Error(w, err.Error(), http.StatusInternalServerError)
        return
    }
    w.Write(pdf)
}
```

Теперь сервис обрабатывает не более 6 конвертаций одновременно. Седьмой запрос либо ждёт освобождения токена, либо получает 503 при превышении таймаута. Потребление памяти не превышает \~3 ГБ, приложение стабильно.

### Связь с другими паттернами

Semaphore — более гибкая альтернатива Worker Pool для неоднородных операций. Внутри Pipeline семафор может ограничивать параллелизм конкретной стадии. ErrGroup с `SetLimit` — это частный случай семафора для групп горутин. Семафор часто комбинируется с Cancellation: захват токена через `ctx` автоматически прерывается при отмене.

## 10. Generator (Генератор)

**Generator** — это паттерн, при котором функция возвращает канал «только для чтения» и запускает горутину, которая генерирует значения в этот канал. Потребитель читает из канала в цикле, не зная, как именно значения производятся.

### Почему это важно

Generator превращает «выталкивание» данных (потребитель дёргает источник) в «вытягивание» (источник сам отправляет данные, когда они готовы). Это даёт:

* **Ленивость:** значения генерируются только тогда, когда потребитель готов их читать (благодаря блокировке канала).
* **Инкапсуляцию:** потребитель не знает, откуда берутся данные — из файла, сети, базы данных или вычислений.
* **Композируемость:** Generator часто является первой стадией Pipeline.

### Как устроено

```go
func generateNumbers(ctx context.Context, n int) <-chan int {
    out := make(chan int)
    go func() {
        defer close(out)
        for i := 0; i < n; i++ {
            select {
            case out <- i:
            case <-ctx.Done():
                return
            }
        }
    }()
    return out
}

// использование:
for num := range generateNumbers(ctx, 100) {
    fmt.Println(num)
}
```

Ключевые свойства:

* Функция возвращает `<-chan T` — канал только для чтения, потребитель не может в него писать.
* Горутина-генератор владеет каналом и закрывает его, когда данные исчерпаны.
* Канал создаётся внутри функции и не передаётся извне — это предотвращает случайную запись.

### Важные моменты реализации

**Всегда передавайте контекст.** Generator должен уметь останавливаться по сигналу отмены. Без контекста генератор, который читает из бесконечного источника (например, поток событий), никогда не завершится.

**Буферизация.** Небуферизированный канал гарантирует, что генератор не «убежит» далеко вперёд потребителя. Если потребитель медленный, генератор блокируется — это естественный backpressure. Если нужна буферизация, используйте `make(chan T, n)`.

**Закрытие канала.** Канал закрывает генератор, и только он. Никогда не закрывайте канал снаружи — это приведёт к панике в генераторе.

### Когда полезен

* **Чтение больших файлов построчно.** Вместо загрузки всего файла в память генератор читает строку и отправляет в канал.
* **Генерация последовательностей.** Числа Фибоначчи, случайные значения, идентификаторы — бесконечные или конечные.
* **Обёртка над внешним API с пагинацией.** Генератор скрывает логику пагинации: потребитель просто читает объекты из канала.
* **Первая стадия Pipeline.** Generator превращает сырой источник данных (файл, сокет, базу) в канал, с которым работают дальнейшие стадии.

### Связь с другими паттернами

Generator — естественная первая стадия Pipeline. В комбинации с Fan-out несколько воркеров читают из одного генератора. Generator с контекстом реализует Cancellation на уровне источника данных.

## 11. Or-Done (Или-готово)

**Or-Done** — это паттерн, который объединяет несколько каналов `done` (сигналов завершения) в один. Результирующий канал закрывается, как только закрывается **любой** из входных каналов.

### Почему это важно

В сложных системах горутина может зависеть от нескольких источников сигнала остановки: контекст запроса, глобальный контекст приложения, таймаут, явный сигнал отмены. Вместо того чтобы писать громоздкий `select` с множеством `case <-done1; case <-done2; ...`, Or-Done сворачивает все сигналы в один канал.

### Как устроено

```go
func orDone(channels ...<-chan struct{}) <-chan struct{} {
    switch len(channels) {
    case 0:
        return nil
    case 1:
        return channels[0]
    }
    or := make(chan struct{})
    go func() {
        defer close(or)
        switch len(channels) {
        case 2:
            select {
            case <-channels[0]:
            case <-channels[1]:
            }
        default:
            select {
            case <-channels[0]:
            case <-channels[1]:
            case <-channels[2]:
            case <-orDone(append(channels[3:], or)...):
            }
        }
    }()
    return or
}
```

Рекурсивная реализация создаёт дерево горутин: каждая горутина ждёт 3 канала + рекурсивный вызов для остальных. Как только любой канал закрывается, сигнал распространяется вверх по дереву и результирующий канал закрывается.

### Важные моменты реализации

**Рекурсия создаёт горутины.** Для N каналов создаётся примерно N/2 горутин. Для небольших N (до нескольких десятков) это нормально. Для сотен каналов лучше использовать итеративную реализацию с `reflect.Select`.

**Каналы только сигнальные.** Or-Done работает с `chan struct{}` — каналами, которые не передают данные, а только сигнализируют закрытием. Для каналов с данными используйте Fan-in.

### Когда полезен

* **Graceful shutdown.** Приложение получает `SIGTERM`, `SIGINT` и имеет внутренний таймаут — Or-Done объединяет их в один сигнал «пора останавливаться».
* **Множественные контексты.** Горутина зависит от контекста запроса и контекста соединения — Or-Done объединяет их `Done()`.
* **Мониторинг нескольких источников.** Ожидание готовности нескольких сервисов: любой упал — реагируем.
*

### Связь с другими паттернами

Or-Done — вспомогательный паттерн для Cancellation, когда источников отмены несколько. Часто используется в Pipeline и Fan-in для объединения сигналов остановки.

***

## 12. Tee (Тройник)

**Tee** — это паттерн, который разветвляет один входной канал на два независимых выходных. Каждое значение из входного канала отправляется в оба выходных. Название отсылает к Unix-команде `tee`.

### Почему это важно

Бывает, что один и тот же поток данных нужно обработать двумя разными способами параллельно. Например, логи пишутся и в файл, и в систему мониторинга. Без Tee потребители будут «бороться» за канал: кто первый прочитал — того и данные. Tee дублирует поток, и каждый потребитель получает полную копию.

### Как устроено

```go
func tee(ctx context.Context, in <-chan int) (<-chan int, <-chan int) {
    out1 := make(chan int)
    out2 := make(chan int)
    go func() {
        defer close(out1)
        defer close(out2)
        for val := range orDoneCh(ctx, in) {
            var o1, o2 = out1, out2
            for i := 0; i < 2; i++ {
                select {
                case o1 <- val:
                    o1 = nil // блокируем повторную отправку в out1
                case o2 <- val:
                    o2 = nil // блокируем повторную отправку в out2
                case <-ctx.Done():
                    return
                }
            }
        }
    }()
    return out1, out2
}
```

Ключевая техника: после отправки значения в канал переменная канала обнуляется (`o1 = nil`), чтобы `select` не отправил то же значение повторно. nil-канал в `select` игнорируется.

### Важные моменты реализации

**Медленный потребитель тормозит всех.** Если `out1` никто не читает, отправка в него блокируется, и `out2` тоже не получит значение. Оба потребителя должны работать с примерно одинаковой скоростью. Для асинхронной развязки используйте буферизированные каналы.

**Nil-каналы.** Техника с `o1 = nil` после отправки — идиоматична для Go. Она позволяет динамически исключать ветви из `select`.

### Когда полезен

* **Дублирование потока логов.** Одни и те же логи пишутся в файл и отправляются в Kafka/Elasticsearch.
* **Параллельная обработка с разной логикой.** Один поток данных обрабатывается двумя независимыми пайплайнами: один агрегирует, другой валидирует.
* **Снятие метрик.** Основной поток данных идёт в бизнес-логику, а его копия — в систему мониторинга для подсчёта throughput.
*

### Связь с другими паттернами

Tee — это специализированный Fan-out ровно на два потребителя с гарантией, что оба получат все данные. Может быть первой стадией двух параллельных Pipeline.

## 13. Bridge (Мост)

**Bridge** — это паттерн, который «выравнивает» канал каналов (`<-chan <-chan T`) в плоский канал (`<-chan T`). Потребитель читает значения так, как будто все вложенные каналы были объединены в один последовательный поток.

### Почему это важно

Канал каналов возникает, когда каждый элемент обработки сам по себе является потоком. Например, вы обходите директории: каждая директория возвращает канал файлов в ней. Или вы опрашиваете несколько сервисов, и каждый возвращает поток событий. Потребитель хочет видеть один непрерывный поток значений, а не вложенные каналы. Bridge решает эту задачу.

> **Зачем это Go-разработчику.** Bridge устраняет один уровень вложенности каналов. Без него потребитель вынужден писать вложенные циклы: `for ch := range chanOfChans { for val := range ch { ... } }`.

### Как устроено

```go
func bridge(ctx context.Context, chanStream <-chan <-chan int) <-chan int {
    out := make(chan int)
    go func() {
        defer close(out)
        for {
            var stream <-chan int
            select {
            case maybeStream, ok := <-chanStream:
                if !ok {
                    return
                }
                stream = maybeStream
            case <-ctx.Done():
                return
            }
            for val := range orDoneCh(ctx, stream) {
                select {
                case out <- val:
                case <-ctx.Done():
                    return
                }
            }
        }
    }()
    return out
}
```

Алгоритм: читаем очередной вложенный канал из `chanStream`, затем вычитываем все значения из этого вложенного канала в `out`, затем переходим к следующему вложенному каналу.

### Важные моменты реализации

**Последовательное вычитывание.** Bridge не перемешивает значения из разных вложенных каналов — он полностью вычитывает один, потом переходит к следующему. Если нужно перемешивание — используйте Fan-in.

**Отмена между вложенными каналами.** Если контекст отменён, пока мы ждём следующий вложенный канал или читаем из текущего, Bridge корректно завершается.

### Когда полезен

* **Обход файловой системы.** Каждая директория возвращает канал файлов. Bridge объединяет их в один поток файлов.
* **Агрегация потоков событий от нескольких сервисов, обнаруженных динамически.** Service discovery возвращает список сервисов, каждый сервис возвращает поток событий.
* **Жизненный цикл воркеров.** Каждый воркер пишет в свой канал; канал каналов передаёт их Bridge для единого потребления.

### Связь с другими паттернами

Bridge — это надстройка над Fan-in для случая, когда количество входных каналов неизвестно заранее и они поступают динамически. Часто комбинируется с Generator (источник каналов — генератор) и Cancellation.

## 14. Как выбирать паттерн

Выбор конкурентного паттерна определяется тремя факторами: характером источника данных, требуемым уровнем параллелизма и стратегией обработки ошибок.

| Ситуация                                                                  | Паттерн      | Ключевой примитив                             |
| ------------------------------------------------------------------------- | ------------ | --------------------------------------------- |
| Много однотипных задач, нужно ограничить одновременное выполнение         | Worker Pool  | Канал + горутины                              |
| Несколько источников данных → один потребитель                            | Fan-in       | `select` + горутины                           |
| Один источник → много параллельных обработчиков                           | Fan-out      | Общий канал между горутинами                  |
| Последовательность этапов обработки, каждый этап может быть распараллелен | Pipeline     | Каналы между этапами                          |
| Операция должна прерваться по таймауту или внешнему сигналу               | Cancellation | `context.Context`                             |
| Параллельные операции, остановка при первой ошибке                        | ErrGroup     | `errgroup.Group`                              |
| Ограничение параллелизма разнородных операций из разных мест кода         | Semaphore    | Буферизированный канал / `semaphore.Weighted` |
| Ленивая генерация последовательности значений                             | Generator    | Канал + одна горутина                         |
| Несколько сигналов остановки → единый сигнал                              | Or-Done      | Рекурсивный `select` на `chan struct{}`       |
| Один поток данных → два независимых потребителя                           | Tee          | Два канала + nil-каналы в `select`            |
| Канал каналов → плоский канал                                             | Bridge       | Вложенный цикл с контекстом                   |

### Комбинирование паттернов

Реальные системы редко используют один паттерн изолированно. Типичные комбинации:

* **Worker Pool + Fan-in:** пул воркеров обрабатывает задачи, результаты собираются в один канал.
* **Generator + Pipeline:** генератор — первая стадия конвейера.
* **Pipeline + Fan-out:** конкретная стадия конвейера распараллелена на несколько воркеров.
* **ErrGroup + Semaphore:** параллельные операции с ограничением по количеству и отменой при первой ошибке.
* **Cancellation + любой паттерн:** каждый паттерн должен поддерживать отмену через контекст.

> **Зачем это Go-разработчику.** Карта паттернов — это не жёсткая классификация, а инструмент навигации. Реальная система почти всегда собирается из нескольких паттернов, как из конструктора. Понимание связей между ними позволяет строить предсказуемые, тестируемые конкурентные архитектуры.

## 15. Типичные ошибки и антипаттерны

Конкурентный код в Go коварен: ошибки часто не детерминированы и проявляются только под нагрузкой. Ниже — наиболее распространённые проблемы.

### Утечка горутин

**Проблема:** горутина запущена, но никогда не завершится, потому что навсегда заблокировалась на канале или мьютексе.

```go
// УТЕЧКА: горутина навсегда заблокируется на отправке в небуферизированный канал,
// который никто не читает
ch := make(chan int)
go func() {
    ch <- 42 // никто не читает — горутина висит вечно
}()
```

**Как избежать:** всегда обеспечивать путь к завершению горутины — через контекст, закрытие канала или `select` с `default`.

### Забытый cancel()

**Проблема:** контекст создан через `WithCancel`/`WithTimeout`, но `cancel()` не вызван. Контекст и все связанные с ним ресурсы висят до завершения родительского контекста.

```go
// УТЕЧКА: cancel не вызван — таймер и ресурсы контекста не освобождаются
ctx, _ := context.WithTimeout(context.Background(), 10*time.Second)
doSomething(ctx)
```

**Как избежать:** всегда `defer cancel()` сразу после создания контекста.

### Гонка данных

**Проблема:** несколько горутин читают и пишут в одну переменную без синхронизации. Go race detector (`go test -race`) обнаруживает гонки, но только если они воспроизводятся в тесте.

```go
var counter int
for i := 0; i < 1000; i++ {
    go func() { counter++ }() // ГОНКА
}
```

**Как избежать:** использовать `sync.Mutex`, `sync/atomic` или каналы для синхронизации доступа. Запускать тесты с `-race`.

### Запись в закрытый канал

**Проблема:** запись в закрытый канал вызывает **panic**.

```go
ch := make(chan int)
close(ch)
ch <- 1 // panic: send on closed channel
```

**Как избежать:** канал закрывает только та горутина, которая в него пишет. Использовать `sync.Once` или флаг, чтобы закрыть канал ровно один раз.

### Чтение из закрытого канала без проверки

**Проблема:** чтение из закрытого канала возвращает нулевое значение типа. Если не проверять флаг `ok`, можно молча получить некорректные данные.

```go
v := <-ch // если канал закрыт, v == 0 (для int). Это корректное значение или признак закрытия?

// правильно:
v, ok := <-ch
if !ok {
    // канал закрыт
}
```

**Как избежать:** всегда проверять `ok` при чтении из канала, если закрытие возможно.

### Дедлок

**Проблема:** все горутины заблокированы и ждут друг друга. Рантайм Go детектирует deadlock только когда **все** горутины спят — если хотя бы одна работает, паники не будет.

```go
ch := make(chan int)
ch <- 1 // deadlock: небуферизированный канал, никто не читает, main спит
```

**Как избежать:** следить за тем, кто и когда читает/пишет в каналы. Небуферизированный канал требует одновременной готовности отправителя и получателя.

### time.After в цикле select

**Проблема:** `time.After` создаёт новый таймер при каждом вызове. В цикле это приводит к утечке памяти, потому что старые таймеры не собираются сборщиком мусора, пока не истечёт их время.

```go
for {
    select {
    case <-ch:
        // работа
    case <-time.After(5 * time.Second): // УТЕЧКА: новый таймер на каждой итерации
        fmt.Println("timeout")
    }
}
```

**Как избежать:** использовать `time.NewTicker` или `time.NewTimer` и переиспользовать их:

```go
timer := time.NewTimer(5 * time.Second)
for {
    select {
    case <-ch:
        // работа
    case <-timer.C:
        fmt.Println("timeout")
        timer.Reset(5 * time.Second)
    }
}
```

### Паника в горутине без recover

**Проблема:** паника в любой горутине (кроме main) **крашит всё приложение**, а не только эту горутину.

```go
go func() {
    panic("oops") // крашит весь процесс
}()
```

**Как избежать:** всегда ставить `defer recover()` в горутинах верхнего уровня. ErrGroup делает это автоматически.

### Избыточная буферизация

**Проблема:** буферизированный канал неоправданно большого размера скрывает проблемы с производительностью и потребляет память.

**Как избежать:** начинать с небуферизированного канала и добавлять буфер только после профилирования и подтверждения необходимости.

> **Зачем это Go-разработчику.** Конкурентные баги — самые дорогие в отладке. Знание типичных ошибок позволяет предотвратить их на этапе написания кода, а не искать под нагрузкой в продакшене.
