# Escape Analysis в Go

**Escape Analysis** — это статический анализ потока данных, выполняемый компилятором Go для определения того, может ли переменная быть размещена на стеке функции или должна быть выделена в куче. В Go нет ключевого слова или функции для прямого управления этим решением — оно принимается исключительно на основе анализа исходного кода.

***

## 1. Что такое Escape Analysis и зачем он нужен

### Определение

Escape analysis решает задачу: для каждой переменной (включая неявные аллокации через `new`, `make` и составные литералы) определить — стек или куча. Это не просто оптимизация, а фундаментальный этап компиляции, обязательный для корректной работы замыканий и эффективного управления памятью.

Точка входа в компиляторе (`cmd/compile/internal/gc/main.go`):

```go
// Escape-анализ.
// Необходим для перемещения heap-аллокаций на стек,
// что, в свою очередь, требуется для реализации замыканий,
// которые сохраняют адреса стековых переменных в замыкание.
// Если замыкание не утекает, оно должно быть на стеке,
// иначе копировщик стека не сможет его обновить.
// Большие значения также перемещаются со стека в ходе escape-анализа;
// поскольку большие значения могут содержать указатели,
// это должно происходить на раннем этапе.
base.Timer.Start("fe", "escapes")
escape.Funcs(typecheck.Target.Funcs)

```

> **Зачем это Go-разработчику.** Понимание escape analysis позволяет писать код с меньшим давлением на GC: стековая переменная не участвует в сборке мусора и освобождается простым сдвигом указателя стека.

### Stack vs Heap: цена размещения

| Характеристика      | Stack                                                          | Heap                                          |
| ------------------- | -------------------------------------------------------------- | --------------------------------------------- |
| Время выделения     | \~1 инструкция (сдвиг SP)                                      | десятки-сотни тактов (поиск свободного блока) |
| Освобождение        | автоматическое при возврате из функции                         | через GC (mark-sweep, требует остановки мира) |
| Кеш-локальность     | высокая (все фреймы рядом)                                     | зависит от фрагментации                       |
| Максимальный размер | \~1 ГБ (до 1ТБ с версии 1.22) на горутину (растёт динамически) | ограничен доступной памятью                   |
| Участие в GC        | нет (стек — корень сканирования)                               | да (каждый указатель в куче сканируется)      |

Размещение на стеке всегда предпочтительнее. Компилятор стремится разместить переменную на стеке всегда, когда это безопасно.

### Роль escape analysis в модели памяти Go

Escape analysis — ключевой элемент модели памяти Go, связывающий три подсистемы:

* **Компилятор** (принимает решение stack/heap)
* **Рантайм** (аллоцирует в куче при вызове `runtime.newobject` / `runtime.mallocgc`)
* **GC** (освобождает кучу; сканирует стек как множество корней)

Без escape analysis каждое `new(T)` или `&T{}` требовало бы аллокации в куче — что сделало бы GC-нагрузку неприемлемой для идиоматичного Go-кода.

## 2. Инварианты и модель анализа

### Два фундаментальных инварианта

Исходный код (`cmd/compile/internal/escape/escape.go`) формулирует их так:

```go
// Escape analysis.
//
// Here we analyze functions to determine which Go variables
// (including implicit allocations such as calls to "new" or "make",
// composite literals, etc.) can be allocated on the stack. The two
// key invariants we have to ensure are: (1) pointers to stack objects
// cannot be stored in the heap, and (2) pointers to a stack object
// cannot outlive that object (e.g., because the declaring function
// returned and destroyed the object's stack frame, or its space is
// reused across loop iterations for logically distinct variables).

```

1. **Указатели на стек-объекты не могут храниться в куче.** Если бы такой указатель существовал, GC при сканировании кучи обнаружил бы указатель на стек, который уже может быть переиспользован или невалиден после перемещения стека горутины.
2. **Указатель на стек-объект не может пережить сам объект.** Например, если функция возвращает `&localVar`, то после выхода из функции её stack frame уничтожается, и указатель становится висячим. Аналогично — переменная в цикле, чей адрес «утекает» на следующую итерацию.

### Что такое «пережить объект»: границы фрейма стека и время жизни

Фрейм стека или stack frame функции существует, пока функция не вернула управление. После этого память фрейма может быть переиспользована следующей вызываемой функцией. Поэтому:

* Возврат `&x` (где `x` — локальная переменная) → `x` перемещается в кучу.
* Сохранение `&x` в глобальную переменную → `x` перемещается в кучу.
* Отправка `&x` в канал → `x` перемещается в кучу (получатель в другой горутине может жить дольше).
* Захват `x` замыканием, которое запускается в горутине → `x` перемещается в кучу.

С другой стороны, передача `&x` в функцию, которая не сохраняет указатель нигде «снаружи», безопасна — `x` остаётся на стеке.

### Какие конструкции языка подвергаются анализу

Анализу подвергаются **все** выражения, порождающие переменные:

* Явные объявления: `var x T`, `x := ...`
* Неявные аллокации: `new(T)`, `make([]T, n)`, `make(map[K]V)`
* Составные литералы: `&T{...}`, `[]T{...}`
* Замыкания: `func() { ... }` — само замыкание как объект
* Конверсии: `string([]byte{...})`, `[]byte("...")`

Не анализируются (всегда на стеке):

* Параметры функций — они уже размещены вызывающей стороной
* Константные строки
* Переменные, размер которых (с учётом элементов) превышает `MaxStackVarSize`

## 3. Взвешенный граф присваиваний

В основе escape-анализа лежит **ориентированный граф**, моделирующий поток данных
между всеми местами хранения значений в функции.

**Узлами этого графа** выступают как реальные переменные программы, так и
**псевдопеременные** — искусственные узлы для операций, не имеющих явного
получателя (запись в кучу, мутация через указатель, вызов замыкания,
отбрасывание значения).

**Рёбра графа** отражают присваивания. Их вес равен количеству разыменований
указателя на пути от источника к получателю: прямое присваивание `y = x`
даёт ребро веса 0, запись через указатель `*p = x` — веса -1,
а чтение `x = *p` — веса 1.

Имея такой граф, анализатор обходит его в обратном направлении — от
потребителей к источникам, распространяя атрибуты (`escapes`, `persists`,
`mutates`, `calls`). Когда атрибут `attrEscapes` достигает переменной, это
означает, что её адрес прямо или косвенно покинул стековый контекст функции,
и переменная должна быть размещена в куче. Если же ни один из опасных
атрибутов до переменной не добрался, она остаётся на стеке.

### Ноды: переменные и псевдопеременные

Каждая переменная, которую анализатор обнаруживает в теле функции — будь то локальная переменная, параметр, результат или захваченная замыканием переменная внешней функции — получает собственный узел в графе. Этот узел хранит ссылку на соответствующий узел IR-дерева (Intermediate Representation — промежуточное представление, которое компилятор строит после синтаксического анализа и семантической проверки, но до генерации машинного кода; каждый `ir.Node` описывает конкретный элемент программы: имя переменной, вызов функции, литерал и т.д.; для псевдопеременных это поле пусто — им не соответствует никакое выражение в исходном коде). Помимо ссылки на IR, узел накапливает всю информацию, необходимую для принятия решения о размещении в памяти: входящие рёбра от источников данных, битовую маску накопленных атрибутов, признаки взятия адреса и переприсваивания, а для параметров — детальный набор утечек, показывающий, какие именно части составного параметра (поля структуры, элементы массива и т.д.) утекают в кучу. Вся эта информация упакована в структуру location (cmd/compile/internal/escape/graph.go):

```go
// location представляет абстрактное местонахождение, в котором хранится Go-переменная.
type location struct {
    n         ir.Node  // представленная переменная или выражение, если есть
    curfn     *ir.Func // функция, в которой объявлена эта переменная (для замыканий — внешняя функция)
    edges     []edge   // входящие рёбра
    loopDepth int      // глубина цикла на момент объявления

    // resultIndex записывает индекс кортежа (начиная с 1) для
    // PPARAMOUT-переменных внутри результирующего типа их функции.
    resultIndex int

    // derefs и walkgen используются во время walkOne для отслеживания
    // минимального количества разыменований от корня обхода.
    derefs  int // >= -1
    walkgen uint32

    // dst и dstEdgeIdx отслеживают следующее непосредственное место назначения
    // присваивания во время walkOne, а также индекс ребра, указывающего обратно на это место.
    dst        *location
    dstEdgeIdx int

    // attrs — это битовая маска атрибутов местонахождения.
    attrs locAttr

    // paramEsc записывает набор утечек (leak set) представленного параметра.
    paramEsc leaks

    captured   bool // захватило ли это переменную замыкание?
    reassigned bool // была ли эта переменная переприсвоена?
    addrtaken  bool // был ли взят адрес этой переменной?
    param      bool // является ли эта переменная параметром (ONAME с классом ir.PPARAM)?
    paramOut   bool // является ли эта переменная выходным параметром (ONAME с классом ir.PPARAMOUT)?
}

```

Атрибуты (`locAttr`) определяют, куда «утекает» переменная:

```go
type locAttr uint8

const (
    attrEscapes   locAttr = 1 << iota // адрес переменной утёк в кучу → heap allocate
    attrPersists                      // переменная должна пережить выражение, создавшее её
    attrMutates                       // достижимые указатели могут быть мутированы
    attrCalls                         // достижимые замыкания могут быть вызваны без отслеживания результата
)

```

Кроме обычных переменных, в графе присутствуют **псевдопеременные** (специальные `location`, создаваемые в `batch`):

```go
type batch struct {
    allLocs         []*location
    closures        []closure
    reassignOracles map[*ir.Func]*ir.ReassignOracle

    heapLoc    location   // представляет кучу — сток для escaping-значений
    mutatorLoc location   // представляет операцию мутации через указатель
    calleeLoc  location   // представляет вызов замыкания
    blankLoc   location   // представляет отбрасывание значения (_)
}

```

**Псевдопеременные** — это искусственные узлы графа потокового анализа, которые не соответствуют реальным переменным исходного кода. Они моделируют специальные точки потребления данных, для которых невозможно завести обычную переменную-получатель: запись в динамическую память, мутацию через неизвестный указатель, вызов замыкания и сознательное игнорирование значения. Без таких узлов анализ был бы неполным — терялись бы целые классы потоков данных, уходящих за пределы имён.

> **Пояснение автора: Мутация через указатель** — это изменение значения переменной не напрямую по её имени, а через указатель, который на неё ссылается.

Инициализация атрибутов псевдопеременных в `Batch`:

```go
b.heapLoc.attrs = attrEscapes | attrPersists | attrMutates | attrCalls
b.mutatorLoc.attrs = attrMutates
b.calleeLoc.attrs = attrCalls

```

Атрибуты псевдопеременных задаются при создании batch и далее не меняются — это константные свойства самих узлов, определяющие их семантику. Когда в графе строится ребро от какой-либо переменной к псевдопеременной, эти атрибуты автоматически распространяются обратно на источник, помечая его соответствующими свойствами.

`heapLoc` уже имеет `attrEscapes` — любое значение, «присвоенное» куче (через ребро с `derefs < 0`), автоматически наследует этот атрибут.

### Рёбра: присваивания и их веса (derefs)

Каждое присваивание в программе порождает ориентированное **ребро (edge)** от переменной-источника к переменной-получателю. Вес этого ребра, называемый `derefs`, равен разности между количеством операций разыменования (\*) и количеством операций взятия адреса (&), через которые проходит значение на пути от источника к получателю.

```go
// An edge represents an assignment edge between two Go variables.
type edge struct {
    src    *location
    derefs int // >= -1
    notes  *note
}

```

Вес ребра — **`derefs`** — равен **количеству разыменований минус количество взятий адреса**:

| Присваивание  | derefs | Смысл                                           |
| ------------- | ------ | ----------------------------------------------- |
| `p = &q`      | -1     | адрес `q` течёт в `p`                           |
| `p = q`       | 0      | значение `q` течёт в `p` без изменений          |
| `p = *q`      | 1      | значение, на которое указывает `q`, течёт в `p` |
| `p = **q`     | 2      | двойное разыменование                           |
| `p = **&**&q` | 2      | комбинация сводится к net derefs                |

Прямое присваивание значения без каких-либо операций над указателями даёт ребро веса 0 — значение перетекло из одной переменной в другую в точности так, как оно хранилось. Если же значение проходит через оператор взятия адреса, вес становится отрицательным: присваивание p = \&q порождает ребро веса -1, поскольку значение q не покинуло своего местоположения, а в p попадает лишь его адрес. Напротив, разыменование увеличивает вес: p = \*q даёт вес 1, так как значение извлекается из памяти по указателю, хранящемуся в q. Двойное разыменование p = \*\*q даёт вес 2, и так далее.

Операции взятия адреса и разыменования могут комбинироваться, и анализатор вычисляет итоговый вес, сокращая взаимно обратные операции. Например, выражение p = **&**\&q после сокращения даёт чистый вес 2, поскольку каждая пара \*& взаимно уничтожается. При этом вес никогда не может опуститься ниже -1: оператор & применим только к адресуемым выражениям, а само выражение \&x уже не является адресуемым, поэтому дважды взять адрес невозможно, и накопить вес -2 нельзя.

Различие между отрицательным, нулевым и положительным весом принципиально для анализа. Ребро веса 0 означает, что источник и получатель разделяют одно и то же значение — если получатель утекает в кучу, то и источник должен быть размещён там же. Ребро веса -1 означает, что утекает не само значение, а лишь его адрес — это более слабая форма утечки, которая, тем не менее, также заставляет переместить источник в кучу, поскольку стековый адрес станет недействительным после возврата из функции. Ребро с положительным весом означает, что значение было извлечено из памяти по указателю, и для анализа важно отследить, откуда взялся сам указатель.

### Семантика derefs и транзитивное замыкание

**Ключевой механизм анализа: транзитивное замыкание графа**

Анализатор не ограничивается рассмотрением одиночных рёбер — он вычисляет **транзитивное замыкание графа**, отслеживая все возможные пути между узлами и накапливая вдоль них суммарный вес. Если в графе существует путь от переменной A к переменной B с суммарным весом d, это означает, что значение A достигает B, пройдя через d операций разыменования. Именно так анализатор узнаёт, что переменные связаны не только прямыми присваиваниями, но и длинными цепочками промежуточных передач данных.

При обходе графа (функция `walkOne` в `solve.go`) компилятор вычисляет минимальное количество разыменований от корневого узла до всех достижимых узлов. Минимизация принципиально важна: если от A до B ведут два пути, один с суммарным весом 0, а другой с весом 2, анализатор выбирает путь с весом 0 как более прямой. Именно по кратчайшему в смысле веса пути и распространяются атрибуты — это гарантирует, что утечка будет обнаружена по самому непосредственному каналу, а не замаскирована длинной цепочкой разыменований.

**Особый случай: отрицательный суммарный вес пути**

Если от корневого узла root до переменной l накопленный суммарный вес оказывается отрицательным, это означает, что по данному пути передаётся не само значение l, а её адрес: операций взятия адреса на пути оказалось больше, чем разыменований. Иными словами, root получил указатель на l. Из этого следуют два правила, которые анализатор обязан соблюсти.

**Первое правило:** если root — это куча (или переменная, уже признанная размещаемой в куче), то l также должна быть в куче. Причина в том, что указатель на стековую переменную, сохранённый в куче, станет висячим после возврата из функции — стековый кадр будет уничтожен, а куча продолжит хранить ссылку на уже несуществующую память.

**Второе правило:** если root переживает l по времени жизни (например, root — переменная внешней функции, а l — локальная переменная замыкания), то l также должна быть в куче. Стековая переменная внутренней функции существует только во время её выполнения, и указатель на неё не может быть сохранён в переменной внешней функции, которая живёт дольше. Единственный способ обеспечить корректность — разместить l в куче, уровняв времена жизни.

**Важный нюанс: ограничение отрицательного веса нулём**

При накоплении derefs вдоль пути отрицательные значения принудительно зажимаются в ноль перед продолжением обхода на соседние узлы. Это необходимо для корректности анализа. Рассмотрим пример: root = \&l, и далее l = x. От root к l суммарный вес равен -1 — адрес l течёт в root. Но означает ли это, что адрес x также течёт в root? Нет: root указывает на l, а l содержит значение x, поэтому root до x не добирается. Если бы анализатор продолжил обход от l к x, сохраняя отрицательный вес -1, он ошибочно заключил бы, что адрес x тоже достиг root. Чтобы избежать этого, при переходе к соседям через ребро от узла, уже достигнутого с отрицательным весом, накопленный вес предварительно зажимается в 0. Таким образом, адрес распространяется ровно на один уровень косвенности и не проходит сквозь присваивания значений.

### Модель передачи данных через AST

При обходе AST анализатору нужно перевести каждую конструкцию языка в термины графа — создать рёбра между узлами, соответствующими переменным, с правильными весами. Для этого используется вспомогательная абстракция \*\*hole («дырка») \*\*— контекст, который накапливает информацию о том, куда именно и через сколько разыменований должно быть доставлено значение выражения. По сути, hole представляет собой ещё не завершённое ребро: получатель (dst) уже известен, а источник будет определён, когда анализатор дойдёт до конкретной переменной или выражения, чей результат «заполнит дырку».

```go
// hole представляет контекст вычисления Go-выражения.
// Например, при вычислении p в "x = **p" мы имеем hole,
// у которого dst==x и derefs==2.
type hole struct {
    dst       *location
    derefs    int // >= -1
    notes     *note
    addrtaken bool
}

```

Поле `dst` указывает на узел-получатель, `derefs` хранит текущее количество разыменований, которое потребуется применить к источнику, а `addrtaken` фиксирует факт взятия адреса источника на каком-либо из этапов. Когда обход выражения завершён и источник определён, вызывается функция `flow`, которая добавляет готовое ребро с итоговым весом от источника к получателю.

Методы манипуляции с hole позволяют наращивать контекст по мере прохождения по AST-дереву:

```go
func (k hole) shift(delta int) hole {
    k.derefs += delta
    if k.derefs < -1 {
        base.Fatalf("derefs underflow: %v", k.derefs)
    }
    k.addrtaken = delta < 0
    return k
}

func (k hole) deref(where ir.Node, why string) hole { return k.shift(1).note(where, why) }
func (k hole) addr(where ir.Node, why string) hole  { return k.shift(-1).note(where, why) }

```

Метод `deref` увеличивает derefs на 1 и соответствует применению оператора \* к текущему выражению в AST: значение, которое заполнит дырку, будет дополнительно разыменовано перед доставкой получателю. Метод `addr` уменьшает derefs на 1, соответствует оператору &, а также устанавливает флаг `addrtaken`, сигнализируя, что у источника был взят адрес. Базовый метод `shift` выполняет изменение веса и дополнительно проверяет, что derefs не опустился ниже -1 — как уже обсуждалось, двойное взятие адреса в Go невозможно, и такая ситуация указывала бы на ошибку в самом анализаторе.

Функция `flow` — центральная точка добавления рёбер в граф. Она получает заполненный контекст `hole` и источник `src`, после чего принимает решение о создании ребра:

```go
func (b *batch) flow(k hole, src *location) {
    if k.addrtaken {
        src.addrtaken = true
    }

    dst := k.dst
    if dst == &b.blankLoc {
        return
    }
    if dst == src && k.derefs >= 0 { // dst = dst, dst = *dst, ...
        return
    }
    if dst.hasAttr(attrEscapes) && k.derefs < 0 { // dst = &src
        // Немедленное обнаружение: адрес src течёт в escaping-локацию
        src.attrs |= attrEscapes | attrPersists | attrMutates | attrCalls
        return
    }

    dst.edges = append(dst.edges, edge{src: src, derefs: k.derefs, notes: k.notes})
}

```

Логика функции включает несколько важных проверок. Если в ходе построения контекста был взят адрес источника, флаг `addrtaken` немедленно выставляется у самого источника — это ключевой признак для последующего анализа, поскольку переменная, чей адрес был взят, потенциально может утечь. Если получателем выступает `blankLoc` (отбрасывание значения в \_), ребро не создаётся вовсе — значение сознательно игнорируется, и отслеживать его дальнейшую судьбу не нужно. Если источник и получатель совпадают, а вес неотрицательный (например, x = x или x = \*x), ребро также пропускается как бесполезный цикл, не несущий новой информации.

Особого внимания заслуживает раннее обнаружение утечки: если получатель уже имеет атрибут `attrEscapes` (например, это `heapLoc` или другая переменная, уже признанная escaping), а вес ребра отрицательный — то есть получателю передаётся адрес источника — то источник немедленно помечается всеми четырьмя атрибутами (attrEscapes, attrPersists, attrMutates, attrCalls). Это оптимизация, избавляющая от необходимости ждать полного обхода графа: факт утечки адреса в кучу или другой escaping-контекст очевиден уже в момент построения ребра, и анализатор фиксирует его сразу. Во всех остальных случаях ребро добавляется в список входящих рёбер получателя и будет обработано позже, на этапе обхода графа и распространения атрибутов.

> **Зачем это Go-разработчику.** Модель графа с derefs объясняет, почему компилятор «понимает» разницу между `return &x` (прямой адрес, derefs = -1 → escape) и `return x` (значение, derefs = 0 → нет escape). Аналогично: почему `p := &x; return *p` может не вызвать escape — цепочка `result = *p; p = &x` даёт путь с derefs = (-1) + 1 = 0, и компилятор видит, что в результат течёт значение, а не адрес.

### Пример построения графа

Рассмотрим функцию:

```go
func example() *int {
    x := 1
    p := &x
    return p
}

```

AST-обход строит локации и рёбра:

1. `x := 1` → локация `Lx`, значение: константа 1
2. `p := &x` → локация `Lp`, ребро: `Lx → Lp` с derefs = -1 (`p = &x`)
3. `return p` → ребро: `Lp → result` с derefs = 0 (`result = p`)

Транзитивный путь: `Lx → Lp → result` с derefs = -1 + 0 = -1. `result` переживает `Lx` (фрейм уничтожается), derefs < 0 → адрес `x` течёт в результат → `x` перемещается (escapes) в кучу.

## 4. Алгоритм обхода графа (solve.go)

### Постановка задачи

После построения графа необходимо для каждой переменной ответить:

1. Может ли её адрес оказаться в куче (прямо или транзитивно)?
2. Переживает ли какая-либо достижимая локация саму переменную?

Формально: для каждой пары `(root, l)` найти минимальное количество разыменований на пути от `root` к `l`. Если `derefs < 0` и `root` переживает `l` — `l` должна быть в куче.

### Алгоритм: Беллман-Форд с нижней границей

Алгоритм реализован в `walkOne` (`cmd/compile/internal/escape/solve.go`) и представляет собой модифицированный алгоритм Беллмана-Форда:

```go
// walkOne вычисляет минимальное количество разыменований от корня до
// всех остальных location.
func (b *batch) walkOne(root *location, walkgen uint32, enqueue func(*location)) {
    // Граф потока данных имеет отрицательные рёбра (от операций взятия адреса),
    // поэтому мы используем алгоритм Беллмана–Форда. Однако нам не нужно
    // беспокоиться о бесконечных отрицательных циклах, поскольку мы
    // ограничиваем промежуточные значения счётчика разыменований нулём.

    root.walkgen = walkgen
    root.derefs = 0
    root.dst = nil

```

Выбор Беллмана-Форда обусловлен наличием отрицательных рёбер (`derefs = -1`). Однако благодаря нижней границе `derefs >= 0` (при продолжении обхода), отрицательные циклы невозможны.

**Основной цикл обхода:**

```go
    todo := newQueue(1)
    todo.pushFront(root)

    for todo.len() > 0 {

        l := todo.popFront()
        derefs := l.derefs
        var newAttrs locAttr

        // Если l.derefs < 0, то адрес l течёт в root.
        addressOf := derefs < 0
        if addressOf {
            derefs = 0  // нижняя граница

            // Если адрес l течёт туда, что переживает l → heap allocate
            if b.outlives(root, l) {
                newAttrs |= attrEscapes | attrPersists | attrMutates | attrCalls
            } else if root.hasAttr(attrPersists) {
                newAttrs |= attrPersists
            }
        }

        if derefs == 0 {
            newAttrs |= root.attrs & (attrMutates | attrCalls)
        }

        // Запись утечек параметров (для межпроцедурного анализа)
        if l.param {
            if b.outlives(root, l) {
                l.leakTo(root, derefs)
            }
            if root.hasAttr(attrMutates) {
                l.paramEsc.AddMutator(derefs)
            }
            if root.hasAttr(attrCalls) {
                l.paramEsc.AddCallee(derefs)
            }
        }

        // Применяем новые атрибуты
        if newAttrs&^l.attrs != 0 {
            l.attrs |= newAttrs
            enqueue(l)
            if l.attrs&attrEscapes != 0 {
                continue  // дальше не идём — уже escaping
            }
        }

        // Релаксация рёбер (Bellman-Ford)
        for i, edge := range l.edges {
            if edge.src.hasAttr(attrEscapes) {
                continue
            }
            d := derefs + edge.derefs
            if edge.src.walkgen != walkgen || edge.src.derefs > d {

                edge.src.walkgen = walkgen
                edge.src.derefs = d
                edge.src.dst = l
                edge.src.dstEdgeIdx = i
                if edge.src.queuedWalkOne != walkgen {
                    edge.src.queuedWalkOne = walkgen
                    todo.pushBack(edge.src)
                }
            }
        }
    }
}

```

**Пошагово алгоритм работает так:**

1. **Инициализация:** root получает `derefs = 0`, ставится в очередь.
2. **Извлечение ноды** `l` из очереди.
3. **Проверка addressOf:** если `derefs < 0`, адрес `l` течёт в `root`. Если `root` переживает `l` — `l` escaping. Значение `derefs` зажимается в 0 для дальнейшего обхода.
4. **Запись утечек параметров:** если `l` — параметр, записываем, куда он течёт.
5. **Релаксация рёбер:** для каждого входящего ребра `edge.src → l`, новый путь к `root` через `l` имеет derefs = `derefs + edge.derefs`. Если он короче известного, обновляем `edge.src` и ставим в очередь.
6. **Повтор до опустошения очереди.**

### Классификация нод и walkAll

`walkOne` — это один обход от одного корня. `walkAll` запускает `walkOne` от **каждой** локации (включая `heapLoc`, `mutatorLoc`, `calleeLoc`):

```go
func (b *batch) walkAll() {
    // Мы используем рабочую очередь для отслеживания location, которые нужно
    // посетить, и повторяем обход до достижения фиксированной точки.
    //
    // Мы выполняем обход один раз от каждого location (включая кучу),
    // а затем ставим в очередь повторно каждый location при его переходе
    // из !persists в persists и из !escapes в escapes, что может произойти
    // не более одного раза для каждого. Таким образом, мы выполняем Θ(len(b.allLocs)) обходов.

    todo := newQueue(len(b.allLocs) + 3)

    enqueue := func(loc *location) {
        if !loc.queuedWalkAll {
            loc.queuedWalkAll = true
            if loc.hasAttr(attrEscapes) {
                todo.pushFront(loc)  // убегающие локации — в приоритете
            } else {
                todo.pushBack(loc)
            }
        }
    }

    for _, loc := range b.allLocs {
        todo.pushFront(loc)
        loc.queuedWalkAll = true
    }
    todo.pushFront(&b.mutatorLoc)
    todo.pushFront(&b.calleeLoc)
    todo.pushFront(&b.heapLoc)

    var walkgen uint32
    for todo.len() > 0 {

        root := todo.popFront()
        root.queuedWalkAll = false
        walkgen++
        b.walkOne(root, walkgen, enqueue)
    }
}

```

Каждая локация — корень одного обхода. При обнаружении новых `attrPersists` или `attrEscapes` нода повторно ставится в очередь — так достигается неподвижная точка. Сложность: `Θ(n)` обходов, где `n` — количество локаций.

Финальная классификация в `finish` (`escape.go`):

```go
if loc.hasAttr(attrEscapes) {
    if n.Op() == ir.ONAME {
        base.WarnfAt(n.Pos(), "moved to heap: %v", n)
    } else {
        base.WarnfAt(n.Pos(), "%v escapes to heap", n)
    }
    n.SetEsc(ir.EscHeap)
} else {
    n.SetEsc(ir.EscNone)
    if !loc.hasAttr(attrPersists) {
        // transient: можно переиспользовать память немедленно
        n.SetTransient(true)
    }
}

```

### Точка входа: Funcs → Batch → walkAll

Цепочка вызовов:

```

escape.Funcs(all []*ir.Func)
  └─ ir.VisitFuncsBottomUp(all, func(list, recursive) { Batch(list, ...) })
       └─ Batch:
           1. initFunc — создать локации для переменных и параметров
           2. walkFunc — обойти AST каждого fn, построить рёбра
           3. flowClosure — обработать замыкания (по значению/ссылке)
           4. HeapAllocReason — проверить размеры, слишком большой размер → принудительно в heapHole
           5. walkAll — основной алгоритм (Беллман-Форд от всех корней)
           6. finish — классификация и вывод диагностики

```

`VisitFuncsBottomUp` гарантирует, что листовые функции (которые никого не вызывают) анализируются первыми — так их теги параметров (parameter tags) доступны при анализе вызывающих функций.

## 5. Межпроцедурный анализ

### Теги параметров: что это и как вычисляются

Когда компилятор завершает анализ тела функции, он знает судьбу каждого её параметра — куда именно и через сколько разыменований перетекли значения. Чтобы эта информация была доступна при анализе вызывающего кода (где тело вызываемой функции может быть уже недоступно, например, если она из другого пакета), компилятор упаковывает её в компактные теги и сохраняет в экспортируемых данных пакета (export data). При анализе места вызова (call site) эти теги читаются, и анализатор понимает, как вызов повлияет на переданные аргументы — не заглядывая в тело вызываемой функции.

Информация о каждом параметре хранится в структуре leaks — массиве из восьми байтов, по одному на каждый возможный «сток» данных:

* куча (heap) — значение параметра или его часть попала в динамическую память;
* мутатор (mutator) — значение параметра или его часть была мутирована через указатель;
* замыкание (callee) — значение параметра или его часть была использована при вызове замыкания;
* результирующие параметры — значение параметра или его часть перетекла в возвращаемые результаты функции (до пяти штук, отсюда и 8 слотов: 3 фиксированных стока + 5 результатов).

```go
// leaks представляет множество потоков присваивания от параметра
// к куче, мутатору, замыканию или к любому из результирующих
// параметров функции (первые numEscResults результатов).
type leaks [8]uint8

const (
    leakHeap = iota  // утечка в кучу
    leakMutator      // утечка в мутатор (мутация через указатель)
    leakCallee       // утечка в замыкание (вызов)
    leakResult0      // утечка в нулевой результирующий параметр
)

const numEscResults = len(leaks{}) - leakResult0  // = 5 (до пяти результирующих параметров)

```

Каждый байт в массиве leaks хранит минимальное количество разыменований, с которым значение параметра достигло соответствующего стока. Значение 0 означает, что до стока дошло само значение параметра напрямую; 1 означает, что до стока дошло значение, на которое параметр указывал; 2 — двойное разыменование, и так далее. Значение 255 (то есть -1 в беззнаковом представлении) означает отсутствие потока — значение параметра до этого стока не добралось ни по какому пути.

Например, если параметр p \*int был передан в функцию, которая записывает \*p в кучу, то в слоте leakHeap для этого параметра окажется значение 1: в кучу утекло не само p, а значение, на которое p указывает. Если же параметр напрямую сохраняется в результат функции (return p), то в слоте leakResult0 будет 0. А если параметр вообще не используется, все слоты будут содержать 255 — никаких потоков нет.

Эта система позволяет анализатору на стороне вызывающего кода точно воспроизвести эффект вызова: он берёт теги параметров вызываемой функции, применяет их к переданным аргументам с учётом веса рёбер на месте вызова (например, если аргумент передаётся как \&x, добавляется лишнее взятие адреса), и получает итоговые атрибуты для переменных вызывающей стороны — всё без повторного анализа тела вызываемой функции.

### Потоки: param → heap, param → result

На этапе обхода графа, когда анализатор распространяет атрибуты, он попутно собирает информацию о том, куда утекает каждый параметр функции. Для этого в коде walkOne есть специальная проверка: если текущий узел l является параметром (l.param == true), то при достижении какого-либо стока root фиксируется факт утечки с текущим накопленным весом derefs:

```go
if l.param {
    if b.outlives(root, l) {
        l.leakTo(root, derefs)
    }
    if root.hasAttr(attrMutates) {
        l.paramEsc.AddMutator(derefs)
    }
    if root.hasAttr(attrCalls) {
        l.paramEsc.AddCallee(derefs)
    }
}

```

Здесь три независимых проверки. Первая проверяет, переживает ли root параметр l по времени жизни; если да — вызывается leakTo, который разбирается, в какой именно сток утекло значение (куча или результат). Вторая проверяет атрибут attrMutates у стока — если стоком оказался mutatorLoc, значит, параметр участвовал в мутации через указатель. Третья ловит атрибут attrCalls — параметр был использован при вызове замыкания.

Метод `leakTo` различает два случая в зависимости от того, куда именно утекло значение:

```go
func (l *location) leakTo(sink *location, derefs int) {
    // Если sink — result parameter, который сам не escaping...
    if !sink.hasAttr(attrEscapes) && sink.isName(ir.PPARAMOUT) && sink.curfn == l.curfn {
        ri := sink.resultIndex - 1
        if ri < numEscResults {
            l.paramEsc.AddResult(ri, derefs)
            return
        }
    }
    // Иначе — heap leak.
    l.paramEsc.AddHeap(derefs)
}

```

Если сток — это результирующий параметр той же функции, и сам он не признан escaping (то есть результат остаётся на стеке вызывающей стороны), утечка записывается в соответствующий слот результата. Во всех остальных случаях — включая утечку в `heapLoc`, в escaping-результат или в переменную, переживающую параметр — утечка считается утечкой в кучу (`AddHeap`). Логика здесь в том, что если результат сам уходит в кучу, то значение параметра, добравшееся до этого результата, в конечном счёте тоже оказывается в куче.

Когда все утечки собраны, вызывается `Optimize` для сокращения хранимой информации:

```go
func (l *leaks) Optimize() {
    // Если есть путь в кучу с derefs = x,
    // нет смысла хранить пути в результаты с derefs >= x.
    if x := l.Heap(); x >= 0 {
        for i := 1; i < len(*l); i++ {
            if l.get(i) >= x {
                l.set(i, -1)
            }
        }
    }
}

```

Смысл оптимизации в том, что утечка в кучу — самый сильный вид утечки. Если параметр уже утекает в кучу напрямую (скажем, с `derefs = 0`), то информация о том, что он также утекает в результат с `derefs = 2`, становится избыточной: куча «поглощает» более слабые утечки. Поэтому все слоты результатов, где вес не меньше (то есть утечка не более прямая, чем уже зафиксированная в кучу), обнуляются — это уменьшает размер тегов, сохраняемых в export data, без потери значимой информации.

### Применение тегов в месте вызова (call site)

Когда компилятор анализирует вызов `foo(&x)`, он проверяет тег параметра `foo`:

1. **Параметр "утекает в кучу"** **(heap leak)** — `&x` должен быть размещён в куче, так как вызываемая функция делает указатель доступным глобально.
2. \*\*Параметр "утекает в результат" (result leak) \*\*— результат вызова `foo(&x)` может содержать указатель на `x`, так что `x` должна пережить вызов. Если результат, в свою очередь, никуда не убегает (escaping), аллокации может не быть.
3. **Параметр "не утекает"** \*\*(no leak) \*\*— `x` может оставаться на стеке; `foo` не сохраняет указатель. Такие параметры в выводе `-m` помечаются как `"x does not escape"`.
4. **Параметр "содержимое утекает" (content leak)**(derefs > 0) — сама переменная-параметр не убегает (escaping), но данные, на которые она указывает, утекают. В выводе: `"leaking param content: x"`.

Именно для этого нужны теги: без них компилятор был бы вынужден считать **любой** вызов внешней функции убегающим (escaping) для её аргументов.

### Особый случай: взаимная рекурсия

Если функции взаимно рекурсивны, они анализируются в одном общем Batch — это принципиально, потому что анализировать их по отдельности невозможно: каждая ссылается на другую, и готовых тегов для вызываемой функции ещё не существует. Для проверки принадлежности к текущему пакету анализа используется метод `inMutualBatch`:

```go
func (b *batch) inMutualBatch(fn *ir.Name) bool {
    if fn.Defn != nil && fn.Defn.Esc() < escFuncTagged {
        if fn.Defn.Esc() == escFuncUnknown {
            base.FatalfAt(fn.Pos(), "graph inconsistency: %v", fn)
        }
        return true
    }
    return false
}

```

Метод проверяет статус анализа вызываемой функции. Константа `escFuncTagged` означает, что теги параметров функции уже вычислены и сохранены; `escFuncUnknown` — что функция ещё не участвовала в анализе. Если статус меньше escFuncTagged, но не равен escFuncUnknown, значит, функция находится в том же Batch и прямо сейчас анализируется вместе с текущей — это и есть случай взаимной рекурсии.

Когда inMutualBatch возвращает true, анализатор не может воспользоваться готовыми тегами параметров вызываемой функции (их ещё нет), поэтому он действует иначе: узлы параметров и результатов вызываемой функции напрямую включаются в граф вызывающей, а на месте вызова строятся рёбра от переданных аргументов к этим узлам. После завершения совместного анализа обе функции получат свои теги одновременно, и взаимные вызовы окажутся корректно разрешены. Если бы анализатор попытался обработать их последовательно, он бы зациклился или получил неполную картину утечек.

### Директива `//go:noescape`

Директива `//go:noescape` — это псевдо-комментарий, который **подсказывает** компилятору, что помеченная функция не допускает утечки переданных аргументов в кучу. Директива не верифицируется компилятором — если разработчик ошибся и функция на самом деле утекает, корректность программы нарушается, и ответственность за это полностью лежит на разработчике. Применяется она в основном для низкоуровневых runtime-функций и ассемблерных вставок, тело которых escape-анализатор не может проанализировать напрямую.

Обработка директивы происходит в функции `paramTag` (escape.go) при вычислении тегов параметров для внешних функций — тех, чьё тело недоступно (например, объявленных в других пакетах или написанных на ассемблере):

```go
// Внешние функции по умолчанию считаются небезопасными,
// кроме случаев, когда перед объявлением указана //go:noescape.
if fn.Pragma&ir.Noescape != 0 {
    if diagnose && f.Sym != nil {
        base.WarnfAt(f.Pos(), "%v не утекает", name())
    }
    esc.AddMutator(0)
    esc.AddCallee(0)
} else {
    if diagnose && f.Sym != nil {
        base.WarnfAt(f.Pos(), "утечка параметра: %v", name())
    }
    esc.AddHeap(0)
}

```

Логика здесь отражает консервативный подход анализатора: не имея доступа к телу внешней функции, он обязан предположить худшее. Поэтому для обычных внешних функций вызывается AddHeap(0) — параметр считается утекающим в кучу напрямую (с нулём разыменований), что заставляет вызывающую сторону размещать соответствующие аргументы в куче. Если же функция помечена `//go:noescape`, анализатор доверяет этой пометке и вместо кучи записывает утечку только в мутатор и замыкание (AddMutator(0) и AddCallee(0)) — это означает, что функция, возможно, будет читать или изменять значение по переданному указателю, но не сохранит его в кучу и не позволит ему покинуть стековый контекст. Отсутствие AddHeap при этом и есть ключевое отличие: параметр не считается убегающим (escaping), и вызывающая сторона может оставить аргумент на стеке.

> **Зачем это Go-разработчику.** `//go:noescape` используется в стандартной библиотеке (например, в `runtime`) для функций, которые принимают указатели, но гарантированно не сохраняют их после возврата. В пользовательском коде лучше полагаться на автоматический анализ, так как некорректная `//go:noescape` может привести к undefined behavior.

## 6. Роль инлайнинга в точности анализа

### Почему инлайнинг критичен для escape analysis

Escape analysis и инлайнинг тесно связаны. Без инлайнинга компилятор вынужден полагаться на теги параметров (или консервативные предположения) для каждого вызова функции. Инлайнинг устраняет границу вызова — встраивает тело функции в вызывающую, позволяя escape-анализу работать напрямую с кодом.

**Пример:**

```go
func newInt() *int {
    x := 42
    return &x  // x escapes
}

func caller() {
    p := newInt()  // без инлайнинга: вызов; x (внутри newInt) уже в куче
    _ = p
}

```

Без инлайнинга `caller` видит только сигнатуру `newInt() *int` и знает, что возвращаемый указатель — escaping, поэтому `x` внутри `newInt` размещается в куче. Сама переменная `p` (указатель) при этом остаётся на стеке `caller`. С инлайнингом тело `newInt` встраивается в `caller`, и `x` напрямую анализируется в контексте `caller`. Если `p` не используется escaping-образом, `x` может остаться на стеке.

### Порог сложности инлайнинга

Инлайнинг в Go ограничен предельной сложностью функции — примерно 80 узлов AST. Сложность здесь — это не цикломатическая сложность и не количество строк, а просто суммарное количество узлов в IR-дереве тела функции: каждая операция, вызов, присваивание, ветвление добавляют узлы, и когда их набирается больше порога, компилятор отказывается от инлайнинга такой функции. Функции, не прошедшие по лимиту, инлайниться не будут.

Для escape-анализа это имеет прямое следствие. Если функция не инлайнится, её тело недоступно анализатору в месте вызова, и он вынужден полагаться на теги параметров, сохранённые в export data. А теги, как уже обсуждалось, консервативны: они фиксируют лишь минимальный вес для каждого стока и не отражают контекст конкретного места вызова. Инлайнинг же подставляет тело функции прямо в вызывающий код, и анализатор работает с фактическими потоками данных, что даёт более точный результат.

Практическое следствие: разбиение больших функций на более мелкие может улучшить точность escape-анализа. Если из большой функции выделить небольшие вспомогательные, они с большей вероятностью окажутся под порогом инлайнинга, их тела будут подставлены в месте вызова, и анализатор увидит полную картину потоков данных без консервативных приближений. Разумеется, это работает только если выделенные функции действительно проходят по бюджету сложности — иначе разбиение ничего не даст.

### Взаимодействие с `//go:noinline`

Директива `//go:noinline` запрещает инлайнинг. Для escape-анализа это означает:

* Аргументы-указатели такой функции анализируются через теги
* Если теги показывают утечку — аргументы уйдут в кучу
* Функция `//go:noinline` с `//go:noescape` — исключение: аргументы останутся на стеке

> **Зачем это Go-разработчику.** Если вы видите неожиданный escape в production-коде, проверьте: инлайнится ли вызываемая функция? Команда `go build -gcflags="-m -m"` покажет решения об инлайнинге (двойной `-m`). Неинлайненный вызов с указателем часто вызывает heap-аллокацию аргумента.

## 7. Типичные случаи, вызывающие побег в кучу

### Возврат указателя на локальную переменную

```go
func f() *int {
    x := 1
    return &x  // перемещено в кучу: x
}

```

Причина: `result = &x` → путь с derefs = -1. `result` переживает `x` (фрейм уничтожается) → `x` должна быть в куче.

```go
func f() *int {
    x := 1
    return &x // x escapes to heap
}

```

### Боксинг в `interface{}`

```go
func f() {
    var i interface{} = 42  // 42 убегает в кучу
    _ = i
}

```

Причина: `interface{}` хранит указатель на данные (eface). Компилятор создаёт неявную копию значения в куче и сохраняет указатель на неё. Для типов, которые помещаются в указатель напрямую (direct iface — указатели, каналы, мапы, функции), аллокации не происходит.

> **Примечание автора:** Direct iface — способ хранения значения в интерфейсе без дополнительной аллокации. Если значение само является указателем (или имеет размер указателя — каналы, мапы, функции, указатели), компилятор помещает его прямо во второй слот интерфейса. Для всего остального (числа, строки, структуры) создаётся скрытая копия в куче, и в интерфейс кладётся уже указатель на неё.

### Захват переменной замыканием

```go
func f() func() int {
    x := 0  // перемещено в кучу: x
    return func() int {
        x++
        return x
    }
}

```

Причина: замыкание — структура с указателями на захваченные переменные. Если замыкание возвращается или запускается в горутине, захваченные переменные должны пережить фрейм.

### Хранение в heap-контейнерах

```go
var globalMap = make(map[int]*int)

func f() {
    x := 42
    globalMap[0] = &x  // перемещено в кучу: x
}

```

Причина: `globalMap` в куче → сохранение `&x` в ней означает указатель на стек-объект в куче → нарушение инварианта 1 → `x` должно быть перемещено в кучу.

### Взятие адреса элемента слайса/мапы

```go
func f() *int {
    s := []int{1, 2, 3}
    return &s[0]  // moved to heap: s
}

```

Причина: `result = &s[0]` порождает путь с derefs = -1 от элементов `s` (индексация даёт неявное разыменование) к результату. Компилятор видит утечку адреса → весь слайс в кучу.

### Каналы: send и receive

```go
func f() {
    ch := make(chan *int)
    x := 42
    ch <- &x  // moved to heap: x
}

```

Причина: отправка в канал делает значение доступным другой горутине, время жизни которой не привязано к текущему фрейму → `x` должна пережить фрейм → отправляется в кучу.

### Размер переменной неизвестен или превышает лимит

Компилятор проверяет размеры в `HeapAllocReason` (`cmd/compile/internal/escape/utils.go`):

```go
func HeapAllocReason(n ir.Node) string {
    if n == nil || n.Type() == nil {
        return ""
    }

    // Параметры всегда на стеке
    if n.Op() == ir.ONAME {
        n := n.(*ir.Name)
        if n.Class == ir.PPARAM || n.Class == ir.PPARAMOUT {
            return ""
        }
    }

    if n.Type().Size() > ir.MaxStackVarSize {

        return "too large for stack"
    }

    if (n.Op() == ir.ONEW || n.Op() == ir.OPTRLIT) && n.Type().Elem().Size() > ir.MaxImplicitStackVarSize {

        return "too large for stack"
    }

    if n.Op() == ir.OMAKESLICE {
        n := n.(*ir.MakeExpr)
        r := n.Cap
        if n.Cap == nil {
            r = n.Len
        }
        if !ir.IsSmallIntConst(r) {
            return "" // гибридный подход: до K на стеке, больше — в куче
        }
        elem := n.Type().Elem()
        if ir.Int64Val(r) > ir.MaxImplicitStackVarSize/elem.Size() {

            return "too large for stack"
        }
    }
    return ""
}

```

Ключевые пороги (`gc/main.go`):

```go
ir.MaxStackVarSize = 64 * 1024          // 64 КБ
ir.MaxImplicitStackVarSize = 16 * 1024  // 16 КБ

```

* `MaxStackVarSize` (64 КБ) — максимальный размер любой переменной на стеке
* `MaxImplicitStackVarSize` (16 КБ) — максимальный размер для неявных аллокаций (`new`, `&T{}`)

Для `make([]int, n)` с неконстантным `n` применяется гибридный подход: компилятор генерирует условную аллокацию — до порога `MaxImplicitStackVarSize` (16 КБ) на стеке, выше — в куче через `runtime.makeslice`.

### Неразличимость полей структуры (Granularity)

```go
type User struct {
    Name *string
    Age  int
}

func f() *string {
    u := User{Age: 30}
    s := "hello"
    u.Name = &s
    return u.Name // перемещено в кучу: s → u тоже в куче
}

```

Причина: escape analysis оперирует всей структурой как единым `location`. Поля не различаются — обращение к `u.Name` моделируется как обращение ко всей `u`. Документировано в `escape.go`:

```go
// Every Go language construct is lowered into this representation,
// generally without sensitivity to flow, path, or context; and
// without distinguishing elements within a compound variable. For
// example:
//
//     var x struct { f, g *int }
//     var u []*int
//
//     x.f = u[0]
//
// is modeled simply as
//
//     x = *u

```

Если одно поле структуры убегает — вся структура в кучу.

### Вызов метода через интерфейс без статического разрешения

```go
type Stringer interface {
    String() string
}

func f(s Stringer) {
    _ = s.String()  // s может быть перемещён в кучу
}

```

Когда компилятор видит вызов s.String(), он не знает, какой конкретный тип скрывается за интерфейсом s. Разные типы могут реализовывать String() по-разному, и в некоторых реализациях метод может сохранить ресивер (то есть сам s) в кучу — например, записать его в глобальную переменную или передать в запущенную горутину. Поскольку тело конкретного метода компилятору в точке вызова недоступно (оно будет определено только во время исполнения), он обязан предположить худшее: ресивер утекает. Поэтому s помечается как escaping и размещается в куче. Если бы тип был известен статически, компилятор мог бы проанализировать тело метода и, возможно, оставить s на стеке.

> **Практическое следствие:** вызов методов на конкретном типе (не через интерфейс) даёт компилятору больше информации — он может проанализировать тело метода напрямую и часто оставляет переменную на стеке. Там, где важна производительность, предпочтите конкретные типы интерфейсам — особенно в горячих путях.

### Передача указателя в неанализируемую функцию и leaking param

```go
func f() {
    x := 42
    fmt.Println(&x)  // moved to heap: x (без инлайнинга)
}

```

Когда компилятор встречает вызов `fmt.Println(&x)`, он передаёт адрес `x` в функцию из другого пакета. `fmt.Println` — вариативная функция с сигнатурой `...interface{}`, и без инлайнинга её тело анализатору недоступно. В этой ситуации он смотрит на теги параметров `fmt.Println`, сохранённые в экспортируемые данные пакета `fmt`, и видит там `leaking param` — параметр помечен как утекающий в кучу. Поскольку в функцию передаётся адрес `x` (вес ребра -1), анализатор делает вывод: адрес стековой переменной попадает в утекающий параметр, а значит, и сама `x` должна быть в куче.

Если бы компилятор мог инлайнить `fmt.Println`, он подставил бы её тело прямо в `f` и увидел бы, что `x` на самом деле никуда не утекает за пределы `f` — форматирование и печать происходят внутри. Тогда `x` осталась бы на стеке. Но `fmt.Println` слишком сложна для инлайнинга, поэтому срабатывает консервативный сценарий.

Общее правило для внешних функций таково: по умолчанию все параметры считаются leaking, если нет директивы `//go:noescape`. В коде вычисления тегов это выглядит так:

```go
if fn.Pragma&ir.Noescape != 0 {
    // параметр не убегает
} else {
    base.WarnfAt(f.Pos(), "leaking param: %v", name())
    esc.AddHeap(0)
}

```

К «неанализируемому поведению» относятся:

* Функции из других пакетов без `//go:noescape`
* Вариативные функции (`fmt.Printf`, `append`)
* Вызовы через `unsafe` или `syscall`
* Функции, которые компилятор не может инлайнить

Во всех этих случаях escape-анализатор перестраховывается и помечает соответствующие переменные как escaping, даже если фактической утечки не происходит.

> **Общая причина:** и вызов через интерфейс (предыдущий раздел), и вызов неанализируемой функции сводятся к одному корню — компилятор не может заглянуть в тело вызываемой функции и потому обязан предположить худшее.

### `runtime.SetFinalizer` принудительно отправляет в кучу

**`runtime.SetFinalizer`** — функция рантайма, которая связывает с объектом **финализатор** (колбек-функцию). Сигнатура: `func SetFinalizer(obj interface{}, finalizer interface{})`. Когда GC обнаруживает, что `obj` стал недостижим, он вызывает `finalizer(obj)` перед освобождением памяти. Типичные сценарии использования: закрытие файловых дескрипторов, освобождение C-памяти, сброс буферов — то есть очистка внешних ресурсов, не управляемых GC напрямую.

Вызов `runtime.SetFinalizer(obj, fn)` всегда вызывает heap-аллокацию `obj`, даже если escape-анализ признал её стековой. Причина — финализатор должен сработать после того, как объект станет недостижимым, что возможно только при размещении в куче (стековая переменная освобождается детерминированно при выходе из функции и не участвует в GC).

```go
func f() {
    x := new(int)
    runtime.SetFinalizer(x, func(obj *int) {})
    // x escapes to heap, даже если new(int) мог бы быть на стеке
}

```

> **Зачем это Go-разработчику.** Если вы используете `runtime.SetFinalizer`, не пытайтесь удержать объект на стеке — он всё равно уйдёт в кучу. Учитывайте это при профилировании аллокаций.

## 8. Диагностика escape analysis

### Флаги компилятора

```shellscript
go build -gcflags="-m"      # базовый вывод решений escape analysis

go build -gcflags="-m -m"   # расширенный: инлайнинг + пути утечек

go build -gcflags="-m -l"   # с отключением инлайнинга (-l)

```

### Как читать вывод

Три основных типа сообщений (формируются в `finish` и `reportLeaks`, `escape.go`):

```go
// moved to heap — переменная (ir.ONAME) размещена в куче
if n.Op() == ir.ONAME {
    base.WarnfAt(n.Pos(), "moved to heap: %v", n)
} else {
    base.WarnfAt(n.Pos(), "%v escapes to heap", n)
}

// leaking param — параметр утекает (derefs = 0 — напрямую, >0 — содержимое)
if x := esc.Heap(); x >= 0 {
    if x == 0 {
        base.WarnfAt(pos, "leaking param: %v", name)
    } else {
        base.WarnfAt(pos, "leaking param content: %v", name)
    }
}

// leaking param to result — параметр течёт в результат с уровнем косвенности
if x := esc.Result(i); x >= 0 {
    res := sig.Result(i).Nname.Sym().Name
    base.WarnfAt(pos, "leaking param: %v to result %v level=%d", name, res, x)
}

```

**Сводка сообщений:**

| Сообщение                              | Значение                                        |
| -------------------------------------- | ----------------------------------------------- |
| `moved to heap: x`                     | Локальная переменная `x` размещена в куче       |
| `x escapes to heap`                    | Выражение (new, make, литерал) размещено в куче |
| `x does not escape`                    | `x` остаётся на стеке                           |
| `leaking param: x`                     | Параметр `x` доступен снаружи после возврата    |
| `leaking param content: x`             | Содержимое, на которое указывает `x`, утекает   |
| `leaking param: x to result y level=N` | Параметр течёт в результат с derefs = N         |
| `inlining call to foo`                 | Вызов инлайнится — см. раздел 6                 |

### Полная диагностика: `-gcflags="-m -m"`

С двойным `-m` компилятор показывает трассировку путей утечки. Вывод формируется в `explainPath` и `explainFlow` (`solve.go`):

```go
func (b *batch) explainPath(root, src *location) []*logopt.LoggedOpt {
    // идёт по цепочке src → src.dst → ... → root,
    // печатая рёбра с derefs и контекстом
}

func (b *batch) explainFlow(pos string, dst, srcloc *location, derefs int, ...) {
    ops := "&"
    if derefs >= 0 {
        ops = strings.Repeat("*", derefs)
    }
    flow := fmt.Sprintf("   flow: %s ← %s%v:", b.explainLoc(dst), ops, b.explainLoc(srcloc))
    // ...
}

```

Пример вывода с `-m -m`:

```

./main.go:5:2: x escapes to heap:
./main.go:5:2:   flow: {heap} ← &x:
./main.go:5:2:     from &x (address-of) at ./main.go:6:9
./main.go:5:2:     from return (return) at ./main.go:7:2

```

### Типичные ошибки интерпретации

1. **«x does not escape» не гарантирует размещение на стеке.** `HeapAllocReason` может отправить переменную в кучу из-за размера (> 64 КБ).

2. **`moved to heap`**\*\* vs **`escapes to heap`**:\*\* `moved to heap` — для уже объявленных переменных (`ir.ONAME`), `escapes to heap` — для выражений, порождающих аллокации (`new`, `&T{}`, `make`).
3. **Вывод зависит от&#x20;****`-l`****.** Без `-l` инлайнинг устраняет мнимые побеги. Проверяйте с `-l` и без.

> **Зачем это Go-разработчику.** Регулярный прогон `go build -gcflags="-m"` в CI позволяет отслеживать регрессии по heap-аллокациям. Неожиданное `moved to heap` — ранний сигнал о проблеме в коде.

## 9. Ограничения текущей реализации

### Нечувствительность к потоку управления

Escape analysis не учитывает ветвления. Если в одной ветке `if` переменная escaping, а в другой — нет, она считается escaping:

```go
func f(b bool) *int {
    x := 1
    if b {
        return &x  // escaping
    }
    return nil    // не escaping
    // Итог: x moved to heap (консервативно)
}

```

Явно указано в исходном коде:

```go
// Every Go language construct is lowered into this representation,
// generally without sensitivity to flow, path, or context;

```

### Отсутствие контекстной чувствительности вызовов

Теги параметров вычисляются **один раз** для функции и не зависят от контекста вызова. Если `foo(p *int)` иногда вызывается с escaping-аргументом, а иногда — нет, тег будет консервативным (leaking). Инлайнинг — единственный способ обойти это ограничение для конкретного call site.

### Гранулярность на уровне целой переменной

Escape analysis не различает поля структуры или элементы слайса (см. «Неразличимость полей структуры» в разделе 7). Обращение к `s.f` моделируется как ко всей `s`. Следствие: любое escaping-поле «тянет» за собой всю структуру.

### Консервативность для внешних пакетов

Для функций из других пакетов без инлайнинга и `//go:noescape` все параметры-указатели считаются leaking в кучу. Это гарантирует безопасность ценой лишних аллокаций.

### Циклические структуры данных

Escape analysis не может доказать, что указатели образуют чисто внутренний цикл в пределах одного фрейма. Любое ребро с `derefs < 0`, достигающее escaping-локации, вызывает heap-аллокацию всей компоненты связности.

## 10. Практические рекомендации

### Когда стоит заботиться о heap-аллокациях

* **Горячие пути** (высокочастотные вызовы) — каждая аллокация увеличивает GC-нагрузку
* **Высоконагруженные сервисы** — GC-паузы влияют на хвостовую задержку (tail latency)
* **Слайсы в циклах** — `make` с неконстантным размером в цикле порождает поток аллокаций
* **Большие структуры** — копирование в кучу + GC-сканирование

Не оптимизируйте преждевременно: однократная аллокация при старте или в редко вызываемом коде не стоит усилий.

### Приёмы удержания переменных на стеке

1. **Избегайте возврата указателей на локальные переменные.** Возвращайте значения:
   ```go
   // Плохо: x moved to heap
   func f() *int { x := 1; return &x }
   // Хорошо: нет аллокации
   func f() int { return 1 }
   ```

2. **Передавайте слайсы по значению, а не указатели на них.** Заголовок слайса (`SliceHeader`) копируется, внутренний массив — нет.
3. **Используйте&#x20;****`[]byte`****&#x20;вместо&#x20;****`string`****&#x20;в горячих путях.** Конверсия `string([]byte{...})` или `[]byte("...")` часто вызывает аллокацию.
4. **Выносите&#x20;****`make`****&#x20;за пределы цикла** и переиспользуйте слайс:
   ```go
   buf := make([]byte, 0, 1024)
   for _, item := range items {
       buf = process(item, buf[:0])
   }
   ```

5. **Структуры:** передавайте по указателю (большие) или по значению (маленькие, если нужно избежать побега).
6. \*\*Используйте \*\***`sync.Pool`** для часто создаваемых/уничтожаемых объектов — снижает давление на GC через переиспользование.

### Инструменты профилирования

```shellscript

# Просмотр решений escape analysis

go build -gcflags="-m" ./...

# Бенчмарк с подсчётом аллокаций

go test -bench=. -benchmem ./...

# Профилирование кучи

go test -bench=. -memprofile=mem.out
go tool pprof -alloc_space mem.out

```

Флаг `-benchmem` показывает количество аллокаций и байт на операцию — прямой индикатор влияния escape analysis на производительность.

> **Зачем это Go-разработчику.** Умение читать вывод `-gcflags="-m"` и `-benchmem` — ключевой навык для performance-инжиниринга в Go. В сочетании с пониманием внутреннего устройства анализа это позволяет не гадать, а точно знать, почему переменная ушла в кучу и как это исправить.
