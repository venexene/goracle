# Выравнивание в Go

## 1. Что такое выравнивание

**Выравнивание (alignment)** — это способ размещения данных в памяти, при котором адрес каждой переменной обязан быть кратен некоторому числу, называемому **выравниванием типа (type alignment)**. Для большинства типов это число равно размеру самого типа.

Процессор не читает память побайтово — он работает с **кэш-линиями** (обычно 64 байта) и **машинными словами** (4 или 8 байт). Если переменная пересекает границу машинного слова, процессору приходится делать два чтения вместо одного и склеивать результат. Выравнивание гарантирует, что переменная всегда помещается в одно машинное слово.

Дополнительная причина — **атомарность**. На большинстве архитектур атомарные операции (например, `sync/atomic`) требуют, чтобы операнд был выровнен по своему естественному выравниванию. Невыровненный доступ к `int64` на 32-битных ARM-системах, например, приведёт к панике.

> **Зачем это Go-разработчику.** Понимание выравнивания позволяет уменьшить потребление памяти структурами без изменения их логики — только за счёт перестановки полей. В высоконагруженных системах с миллионами экземпляров структуры экономия даже 8 байт на экземпляр даёт гигабайты памяти.

***

## 2. Как работает выравнивание в Go

### Правила для базовых типов

Каждый тип в Go имеет гарантированное выравнивание, которое определяется на этапе компиляции:

| Тип                                       | Размер (байт) | Выравнивание (байт) |
| ----------------------------------------- | ------------- | ------------------- |
| `bool`, `byte`, `int8`, `uint8`           | 1             | 1                   |
| `int16`, `uint16`                         | 2             | 2                   |
| `int32`, `uint32`, `float32`, `rune`      | 4             | 4                   |
| `int64`, `uint64`, `float64`, `complex64` | 8             | 8                   |
| `complex128`                              | 16            | 16                  |
| `string`                                  | 16            | 8                   |
| `slice` (любой)                           | 24            | 8                   |
| `map`, `chan`, `pointer`                  | 8             | 8                   |
| `interface`                               | 16            | 8                   |

На 32-битных архитектурах размер указателя и выравнивание `map`/`chan`/`pointer` уменьшаются до 4 байт.

### Правила для структур

Для **структуры** (`struct`) действуют три правила:

1. **Выравнивание структуры** равно максимальному выравниванию среди её полей.
2. **Каждое поле** размещается со смещением, кратным его собственному выравниванию. Если предыдущее поле закончилось на адресе, не кратном нужному выравниванию, компилятор вставляет **отступ (padding) **— неиспользуемые байты.
3. **Размер структуры** должен быть кратен её выравниванию. Если суммарный размер полей (с учётом отступа между ними) не дотягивает до кратности, компилятор добавляет **завершающий отступ (trailing padding)**.

> **Зачем это Go-разработчику.** Компилятор не переупорядочивает поля — он сохраняет их в том порядке, в котором вы объявили. Значит, выбор порядка полей напрямую влияет на размер структуры.

## 3. Инструменты для анализа выравнивания

Go предоставляет три функции из пакета `unsafe` для интроспекции памяти:

* `unsafe.Sizeof(x)` — размер значения `x` в байтах, включая padding.
* `unsafe.Alignof(x)` — выравнивание типа значения `x`.
* `unsafe.Offsetof(x.f)` — смещение поля `f` в структуре `x` от начала структуры.

Пример анализа структуры:

```go
type Example struct {
    a bool
    b int32
    c bool
}

var e Example
fmt.Println(unsafe.Sizeof(e))    // 12
fmt.Println(unsafe.Alignof(e))   // 4
fmt.Println(unsafe.Offsetof(e.a)) // 0
fmt.Println(unsafe.Offsetof(e.b)) // 4 (после 3 байт padding)
fmt.Println(unsafe.Offsetof(e.c)) // 8

```

Для автоматического анализа можно использовать утилиту `fieldalignment` из `golang.org/x/tools`:

```

fieldalignment -fix ./...

```

> **Зачем это Go-разработчику.** `unsafe.Offsetof` — единственный способ точно узнать, где компилятор разместил поле и сколько padding добавлено. `fieldalignment` позволяет автоматически найти структуры с неоптимальным порядком полей во всём проекте.

## 4. Примеры выравнивания структур

### Неоптимальный порядок полей (чередование малых и больших типов)

Рассмотрим структуру, где поля идут «как пришло в голову»:

```go
type BadOrder struct {
    a byte   // 1 байт, alignment 1
    b int32  // 4 байта, alignment 4
    c byte   // 1 байт, alignment 1
}

```

Компилятор размещает поля следующим образом (адреса условные):

| Смещение | Содержимое                 |
| -------- | -------------------------- |
| 0        | `a` (byte)                 |
| 1–3      | padding (3 байта)          |
| 4–7      | `b` (int32)                |
| 8        | `c` (byte)                 |
| 9–11     | trailing padding (3 байта) |

* Поле `a` занимает 1 байт.
* После `a` добавляется 3 байта padding, чтобы `b` (требует выравнивания 4) оказалось на адресе, кратном 4.
* Поле `c` занимает 1 байт.
* Структура дополняется 3 байтами в конце, чтобы общий размер был кратен 4.

**Итог:** `unsafe.Sizeof(BadOrder{}) == 12` при полезных данных всего 6 байт. Половина памяти — padding.

```go
package main

import (
    "fmt"
    "unsafe"
)

type Example struct {
    a bool  // 1 байт
    b int32 // 4 байта
    c int8  // 1 байт
}

func main() {
    var e Example
    fmt.Printf("Размер Example: %d байт\n", unsafe.Sizeof(e))
    fmt.Printf("Выравнивание Example: %d байт\n", unsafe.Alignof(e))

    // Смещения полей
    fmt.Printf("a: offset=%d, размеp=%d\n", unsafe.Offsetof(e.a), unsafe.Sizeof(e.a))
    fmt.Printf("b: offset=%d, размеp=%d\n", unsafe.Offsetof(e.b), unsafe.Sizeof(e.b))
    fmt.Printf("c: offset=%d, размеp=%d\n", unsafe.Offsetof(e.c), unsafe.Sizeof(e.c))
}

```

Этот же пример иллюстрирует и диаграмма ниже — каждый блок соответствует байту, пустые блоки обозначают отступ:

```

Вывод (на 64-битной системе):

Размер Example: 12 байт
Выравнивание Example: 4 байта
a: offset=0, размер=1
b: offset=4, размер=4
c: offset=8, размер=1

```

### Оптимальный порядок полей (от большего alignment к меньшему)

Если переставить поля, сгруппировав их по убыванию размера, отступ резко сокращается:

```go
type GoodOrder struct {
    b int32  // 4 байта, alignment 4
    a byte   // 1 байт, alignment 1
    c byte   // 1 байт, alignment 1
}

```

| Смещение | Содержимое                 |
| -------- | -------------------------- |
| 0–3      | `b` (int32)                |
| 4        | `a` (byte)                 |
| 5        | `c` (byte)                 |
| 6–7      | trailing padding (2 байта) |

* Поля расположены в порядке убывания размера.
* После `a` и `c` добавляется 2 байта завершающего отступа, чтобы выровнять структуру до 4 байт.

**Итог:** `unsafe.Sizeof(GoodOrder{}) == 8`. Экономия 4 байта (33%) без изменения логики.

```go
type Optimized struct {
    b int32 // 4 байта
    a bool  // 1 байт
    c bool  // 1 байт
}

func main() {
    var o Optimized
    fmt.Printf("Размер Optimized: %d байт\n", unsafe.Sizeof(o)) // 8 байт
}

```

### Вложенные структуры

Вложенная структура наследует выравнивание от своего самого строгого поля, и внешняя структура обязана соблюдать это выравнивание:

```go
type Inner struct {
    x int64   // 8 байт, alignment 8
    y byte    // 1 байт, alignment 1
}

type Outer struct {
    z byte    // 1 байт, alignment 1
    i Inner   // выравнивание 8
}

```

* `Inner` имеет размер 16 байт: 8 (`x`) + 1 (`y`) + 7 trailing padding (кратность 8).
* `Inner` требует выравнивания 8.
* В `Outer` после `z` добавляется 7 байт отступа, чтобы `i` оказалось на адресе, кратном 8.

**Итог:** `unsafe.Sizeof(Outer{}) == 24`.

```go
type Inner struct {
    x int64 // 8 байт
    y bool  // 1 байт
}

type Outer struct {
    z int16 // 2 байта
    i Inner // 16 байт (8 + 1 + 7 padding)
}

func main() {
    var o Outer
    fmt.Printf("Размер Outer: %d байт\n", unsafe.Sizeof(o)) // 24 байта
}

```

> **Зачем это Go-разработчику.** При проектировании структур, которые будут вложены в другие, стоит выносить «тяжёлые» поля (с большим выравниванием) наверх, а «лёгкие» — группировать внизу. Это минимизирует отступ как внутри, так и снаружи.

## 5. Оптимизация порядка полей

### Методика

Правило простое: **располагайте поля в порядке убывания выравнивания (alignment)** — от наибольшего к наименьшему.

Порядок групп:

1. `complex128` (alignment 16)
2. `int64`, `uint64`, `float64`, `complex64` (alignment 8)
3. Указатели, `map`, `chan`, `func`, `slice`, `string`, `interface` (alignment 8 на 64-бит)
4. `int32`, `uint32`, `float32` (alignment 4)
5. `int16`, `uint16` (alignment 2)
6. `byte`, `bool`, `int8`, `uint8` (alignment 1)

Если два поля имеют одинаковое выравнивание, порядок между ними не влияет на размер структуры.

### Пример сравнения «до и после»

```go
// До: 40 байт
type Before struct {
    flag    bool        // 1
    counter int64       // 8 → после 7 байт padding
    ready   bool        // 1
    data    [3]byte     // 3
    ptr     *int        // 8 → после 4 байт padding
} // + 7 байт trailing padding

// После: 24 байта
type After struct {
    counter int64       // 8
    ptr     *int        // 8
    data    [3]byte     // 3
    flag    bool        // 1
    ready   bool        // 1
} // trailing padding не нужен (24 кратно 8)

```

Экономия: 16 байт на экземпляр (40%). При 10 миллионах экземпляров — около 152 МБ.

### Когда оптимизация оправдана

* Структура создаётся в больших количествах (миллионы экземпляров в памяти).
* Структура хранится в долгоживущих коллекциях (кэш, долгосрочная мапа).
* Вы разрабатываете библиотеку, которой будут пользоваться другие.

### Когда оптимизация избыточна

* Локальная переменная функции (живёт на стеке, сборщик мусора её не сканирует).
* Структура с одним-двумя полями.
* Код, критичный к читаемости: группировка полей по **логическому смыслу** может быть важнее экономии пары байт.

> **Зачем это Go-разработчику.** Перестановка полей — один из немногих способов оптимизации памяти, который не меняет семантику кода, безопасен и легко автоматизируется (`fieldalignment -fix`). Это «бесплатная» оптимизация, которую можно применить к уже работающему проекту.

## 6. Особые случаи

### Пустая структура `struct{}`

Пустая структура `struct{}` имеет размер **0 байт** и выравнивание **1**. Это означает, что два поля `struct{}` могут иметь одинаковый адрес.

Использование `struct{}` в качестве поля структуры **не влияет на размер** (zero-sized field), но может вызывать неожиданное поведение, если такая структура — **последнее поле**, за которым идёт поле большего выравнивания:

```go
type Quirky struct {
    a byte
    b struct{}
    c int64  // потребует 7 байт padding после b
}

```

Здесь `b` (размер 0) не создаёт полезной нагрузки, но `c` всё равно требует выравнивания 8, поэтому после `b` добавляется отступ. При этом `unsafe.Sizeof(Quirky{}) == 24`.

### Поле размера ноль в финальной позиции

Если поле размера ноль (zero-sized) (`struct{}` или `[0]byte`) стоит **последним** в структуре, компилятор добавляет финальный отступ, как если бы поля не было вовсе. Размер структуры от этого не меняется.

Но с точки зрения `unsafe.Offsetof` поле размера ноль может разделять адрес со следующим за структурой объектом в памяти — это нормально и не является ошибкой.

### Выравнивание в массивах структур

Элементы массива структур упаковываются вплотную — без дополнительных зазоров между элементами. Каждый следующий элемент начинается сразу за предыдущим, ровно через `unsafe.Sizeof(StructType)` байт:

```go
type S struct {
    a byte
    b int32
}
// unsafe.Sizeof(S{}) = 8 (с завершающим отступом)
// Массив [2]S: элементы по адресам 0 и 8

```

> **Зачем это Go-разработчику.** При работе со слайсами структур важно понимать, что неоптимальный порядок полей умножается на длину слайса. Слайс из миллиона `BadOrder` тратит на 4 МБ больше, чем слайс из миллиона `GoodOrder`.

## 7. Практическая значимость

### Влияние на сборщик мусора

GC сканирует каждое поле структуры. Чем больше размер структуры, тем больше памяти нужно обойти сборщику при маркировке живых объектов. Уменьшение размера структуры на 30–40% прямо пропорционально ускоряет фазу сканирования.

### Кэш-промахи

Процессор загружает данные в кэш порциями — кэш-линиями по 64 байта. Компактная структура с большей вероятностью поместится в одну кэш-линию, уменьшая количество кэш-промахов. Это особенно важно для структур, по которым происходит итерация (например, в горячих циклах).

### Ложное разделение (false sharing)

Когда две горутины работают с разными полями одной структуры, но эти поля попадают в одну кэш-линию, процессорные ядра вынуждены синхронизировать кэш-линии, даже если логического конфликта нет. Это называется **false sharing**. Правильное выравнивание и, при необходимости, явный padding до размера кэш-линии решают эту проблему:

```go
type Counter struct {
    value int64
    _     [56]byte // padding до 64 байт (размер кэш-линии)
}

```

> **Зачем это Go-разработчику.** В высоконагруженных параллельных системах false sharing может снизить производительность на порядок. Явный padding — стандартный приём, используемый в стандартной библиотеке Go (например, `sync.Pool`).

***

## 8. Краткие рекомендации

* Располагайте поля структуры от наибольшего alignment к наименьшему.
* Используйте `fieldalignment -fix ./...` для автоматического поиска и исправления неоптимальных структур в проекте.
* Не оптимизируйте порядок полей в ущерб читаемости, если структура существует в единичных экземплярах.
* Помните о `struct{}`: будучи последним полем, она не увеличивает размер структуры, но может влиять на адреса следующего поля с большим выравниванием.
* При работе со слайсами структур перестановка полей даёт умноженный эффект — экономия на каждом элементе умножается на длину слайса.
* Для параллельных структур, разделяемых между горутинами, рассматривайте явный padding до размера кэш-линии для предотвращения false sharing.
