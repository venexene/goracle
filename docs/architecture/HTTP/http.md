# HTTP: протокол и реализация в Go

## 1. Введение

Протокол передачи гипертекста (Hypertext Transfer Protocol — HTTP) — это прикладной протокол для передачи гипертекстовых документов, таких как HTML. Он создан для связи между веб-браузерами и веб-серверами, хотя может использоваться и для других целей. Протокол следует классической клиент-серверной модели: клиент открывает соединение, создаёт запрос и ждёт ответа. HTTP — протокол без сохранения состояния: сервер не хранит данные между двумя парами «запрос-ответ». Основан на TCP/IP, но может использовать любой транспортный протокол с гарантированной доставкой.

В этом документе разобраны:

* эволюция протокола от HTTP/0.9 до HTTP/3 и ключевые изменения на каждом этапе
* семантика HTTP: ресурсы, методы, статус-коды, заголовки
* устройство HTTP/1.1 и HTTP/2 на уровне протокола
* архитектура пакета `net/http` в Go
* HTTP-сервер: от `Accept` до вызова handler'а
* `ServeMux` и маршрутизация (включая паттерны Go 1.22+)
* конкурентность в сервере
* HTTP-клиент и `Transport`
* middleware, тестирование, graceful shutdown
* HTTP/2 и продвинутые темы

### Что такое HTTP

HTTP — это протокол для получения ресурсов, например HTML-документов. Он лежит в основе обмена данными в Интернете и является протоколом клиент-серверного взаимодействия: запросы инициирует получатель, обычно веб-браузер. Готовый документ состоит из разнородных ресурсов — текст, стили, изображения, видео, скрипты.

![](assets/img1.svg)

Клиент и сервер обмениваются отдельными сообщениями, а не потоком данных. Сообщения от клиента называются **запросами** (request), от сервера — **ответами** (response).

![](assets/img2.svg)

HTTP разработан в начале 1990-х и является расширяемым протоколом. Это протокол прикладного уровня, использующий TCP (или TLS-защищённый TCP) для пересылки сообщений. Благодаря расширяемости HTTP применяется не только для гипертекста, но и для изображений, видео, отправки HTML-форм, частичного обновления страниц.

> **Зачем это Go-разработчику.** `net/http` реализует обе стороны протокола — клиентскую и серверную. Каждый тип в пакете (`Server`, `Client`, `Transport`, `ServeMux`) отражает одну из ролей в этой модели. Понимание ролей упрощает навигацию по API.

### Составляющие HTTP-систем

HTTP — клиент-серверный протокол: запросы отправляет пользовательский агент (браузер, робот, утилита). Каждый запрос обрабатывается сервером и возвращается ответ. Между ними — многочисленные прокси: кеширующие, фильтрующие, балансирующие нагрузку.

![](assets/img3.svg)

Прокси на уровне приложений могут быть прозрачными (пересылают запросы без изменений) и непрозрачными (модифицируют запрос). Функции прокси: кеширование, фильтрация, балансировка, аутентификация, протоколирование.

> **Зачем это Go-разработчику.** `Transport` в `net/http` умеет работать через HTTP-прокси (`Proxy`, `ProxyConnectHeader`). `ReverseProxy` из `net/http/httputil` — это готовый HTTP-прокси на Go.

#### Клиент: пользовательский агент

Пользовательский агент (user agent) — инструмент, действующий от лица пользователя. Чаще всего браузер, но также `curl`, боты, приложения. Браузер всегда инициирует запрос; сервер обычно этого не делает.

Чтобы отобразить страницу, браузер отправляет запрос за HTML, затем за дополнительными ресурсами (CSS, скрипты, изображения) и собирает их в единый документ.

#### Веб-сервер

Сервер обслуживает запросы клиента. Физически это может быть группа серверов за балансировщиком. Начиная с HTTP/1.1 заголовок `Host` позволяет нескольким серверам использовать один IP-адрес.

#### Прокси

Прокси — промежуточные узлы на уровне приложений. Основные функции:

- **кеширование** — публичный или приватный кеш
- **фильтрация** — антивирус, родительский контроль
- **балансировка нагрузки** — распределение запросов между серверами
- **аутентификация** — контроль доступа
- **протоколирование** — история операций

### Ключевые свойства HTTP

**Простота.** HTTP-сообщения читаемы человеком, что упрощает отладку. Даже в HTTP/2 с бинарным фреймингом семантика сообщений осталась прежней.

**Расширяемость.** HTTP-заголовки, введённые в HTTP/1.0, сделали протокол легко расширяемым. Новая функциональность добавляется соглашением о семантике заголовка.

**Отсутствие состояния, наличие сессий.** HTTP не хранит состояние между запросами. HTTP-куки и заголовки решают эту проблему, добавляя сессии поверх протокола. В Go за куки отвечает `http.Cookie` и `http.CookieJar`.

**Соединения.** HTTP требует надёжного транспорта (TCP). В HTTP/1.0 каждое соединение — одна пара запрос-ответ. HTTP/1.1 добавил keep-alive и конвейерную обработку. HTTP/2 — мультиплексирование в одном соединении. HTTP/3 переходит на QUIC поверх UDP.

> **Зачем это Go-разработчику.** Эволюция модели соединений напрямую влияет на `Transport`: пул соединений, `MaxIdleConns`, `DisableKeepAlives`, `Protocols` — все эти настройки отражают переход от HTTP/1.0 к современным версиям.

### Чем можно управлять через HTTP

Расширяемость HTTP позволяет управлять разными аспектами взаимодействия:

- **Кеширование** — сервер указывает, что и как долго кешировать. Заголовки `Cache-Control`, `ETag`, `Last-Modified`.
- **Ослабление ограничений источника** — заголовки `CORS` (`Access-Control-Allow-Origin`) разрешают междоменные запросы. В Go `CrossOriginProtection` защищает от CSRF.
- **Аутентификация** — `WWW-Authenticate`, `Authorization`, сессии через куки.
- **Прокси и туннелирование** — HTTP-запросы через прокси для обхода сетевых ограничений.
- **Сессии** — HTTP-куки связывают запросы с состоянием сервера.

### HTTP-поток

Открытие TCP-соединения → отправка HTTP-сообщения → чтение ответа → закрытие или переиспользование соединения.

Пример запроса:

```
GET / HTTP/1.1
Host: developer.mozilla.org
Accept-Language: fr
```

Пример ответа:

```
HTTP/1.1 200 OK
Date: Sat, 09 Oct 2010 14:28:02 GMT
Server: Apache
Content-Length: 29769
Content-Type: text/html

<!doctype html>…
```

### HTTP-сообщения

Сообщения HTTP/1.1 человекочитаемы. В HTTP/2 они встроены в бинарные кадры, но семантика та же.

Существует два типа сообщений: запросы и ответы.

#### Запрос

![](assets/img4.svg)

Элементы запроса:

- **Метод** — глагол (`GET`, `POST`) или существительное (`OPTIONS`, `HEAD`), определяющий операцию
- **Путь к ресурсу** — без протокола, домена и порта
- **Версия протокола**
- **Заголовки** — дополнительная информация
- **Тело** — для методов типа `POST` содержит передаваемые данные

#### Ответ

![](assets/img5.svg)

Элементы ответа:

- **Версия протокола**
- **Код состояния** — успех или причина неудачи
- **Сообщение состояния** — краткое описание кода
- **Заголовки**
- **Тело** — получаемый ресурс (опционально)

> **Зачем это Go-разработчику.** В `net/http` запрос представлен типом `*http.Request`, ответ — `http.ResponseWriter` (для сервера) или `*http.Response` (для клиента). Заголовки — `http.Header` (map[string][]string). Коды состояния — константы `http.StatusOK`, `http.StatusNotFound` и т.д.

## 2. Эволюция протокола

HTTP (HyperText Transfer Protocol) — это базовый протокол Всемирной паутины. Разработанный Тимом Бернерсом-Ли и его командой в период с 1989 по 1991 год, HTTP претерпел множество изменений, которые помогли сохранить его простоту, одновременно повысив его гибкость.

### Изобретение Всемирной паутины

В 1989 году, работая в ЦЕРНе, Тим Бернерс-Ли написал предложение о создании гипертекстовой системы через интернет. Первоначально она называлась Mesh, но позже, во время её внедрения в 1990 году, была переименована во Всемирную паутину. Построенная на основе существующих протоколов TCP и IP, она состояла из 4 компонентов:
- Текстовый формат для представления гипертекстовых документов — язык разметки гипертекста (HTML).
П- ротокол для обмена этими документами — протокол передачи гипертекста (HTTP).
- Клиент для отображения (и редактирования) этих документов — первый веб-браузер, называемый WorldWideWeb.
- Сервер для предоставления доступа к документу — ранняя версия httpd.

Эти четыре компонента были завершены к концу 1990 года, а первые серверы начали работать за пределами ЦЕРН к началу 1991 года. 6 августа 1991 года Тим Бернерс-Ли опубликовал сообщение в общедоступной новостной группе alt.hypertext. Сейчас это считается официальным началом Всемирной паутины как публичного проекта.

Протокол HTTP, использовавшийся на ранних этапах, был очень простым. Позже он получил название HTTP/0.9 и иногда называется однострочным протоколом.

### HTTP/0.9 – протокол в одну строку

Первоначальная версия HTTP не имела номера версии; позже она получила название 0.9, чтобы отличать её от более поздних версий. HTTP/0.9 был чрезвычайно прост: запросы состояли из одной строки и начинались с единственно возможного метода GET, за которым следовал путь к ресурсу. Полный URL-адрес не включался, поскольку протокол, сервер и порт не были необходимы после подключения к серверу.

```
GET /my-page.html
```

Ответ тоже оказался предельно простым: он состоял всего лишь из самого файла.
```
<html>
  A text-only web page
</html>
```

В отличие от последующих этапов развития технологий, HTTP-заголовки отсутствовали. Это означало, что передавались только HTML-файлы. Отсутствовали коды состояния или ошибок. В случае возникновения проблемы генерировался специальный HTML-файл, содержащий описание проблемы для удобства восприятия человеком.

### HTTP/1.0 – Расширение возможностей

Протокол HTTP/0.9 был очень ограничен, но браузеры и серверы быстро сделали его более универсальным:

- Информация о версиях отправлялась в каждом запросе (к строке GET добавлялся HTTP/1.0).
- В начале ответа также отправлялась строка с кодом состояния. Это позволяло браузеру распознавать успех или неудачу запроса и соответствующим образом адаптировать свое поведение. Например, обновлять или использовать свой локальный кэш определенным образом.
- Была введена концепция HTTP-заголовков как для запросов, так и для ответов. Метаданные могли передаваться, и протокол стал чрезвычайно гибким и расширяемым.
- Благодаря заголовку Content-Type можно было передавать документы, отличные от обычных HTML-файлов.

На тот момент типичный запрос и ответ выглядели так:
```
GET /my-page.html HTTP/1.0
User-Agent: NCSA_Mosaic/2.0 (Windows 3.1)

HTTP/1.0 200 OK
Date: Tue, 15 Nov 1994 08:12:31 GMT
Server: CERN/3.0 libwww/2.17
Content-Type: text/html

<HTML>
A page with an image
  <IMG SRC="/my-image.gif">
</HTML>
```

Затем последовало второе соединение и запрос на получение изображения (с соответствующим ответом):
```
GET /my-image.gif HTTP/1.0
User-Agent: NCSA_Mosaic/2.0 (Windows 3.1)

HTTP/1.0 200 OK
Date: Tue, 15 Nov 1994 08:12:32 GMT
Server: CERN/3.0 libwww/2.17
Content-Type: text/gif

(image content)
```

В период с 1991 по 1995 год эти технологии внедрялись методом проб и ошибок. Сервер и браузер добавляли ту или иную функцию и смотрели, насколько она востребована. Проблемы совместимости были обычным явлением. В попытке решить эти проблемы в ноябре 1996 года был опубликован информационный документ, описывающий общепринятые методы. Он получил название RFC 1945 и определял протокол HTTP/1.0.

### HTTP/1.1 – стандартизированный протокол

Тем временем шла надлежащая стандартизация. Это происходило параллельно с различными реализациями HTTP/1.0. Первая стандартизированная версия HTTP, HTTP/1.1, была опубликована в начале 1997 года, всего через несколько месяцев после HTTP/1.0.

HTTP/1.1 устранил неоднозначности и внес множество улучшений:
- Соединение можно было использовать повторно, что экономило время. Больше не нужно было открывать его несколько раз для отображения ресурсов, встроенных в один исходный документ.
- Была добавлена ​​конвейерная обработка. Это позволяло отправлять второй запрос до того, как ответ на первый был полностью передан. Это снижало задержку связи.
- Также поддерживались ответы по частям.
- Были введены дополнительные механизмы управления кэшированием.
- Была введена возможность согласования контента, включая язык, кодировку и тип. Теперь клиент и сервер могли согласовывать, какой контент обмениваться.
- Благодаря заголовку Host появилась возможность размещать разные домены с одного IP-адреса, что позволило использовать совместное размещение серверов.

Следующий пример иллюстрирует типичную последовательность HTTP/1.1 запросов, отправляемых по одному постоянному TCP-соединению, демонстрируя, как клиенты могут повторно использовать соединения для более эффективной загрузки ресурсов. Первый запрос получает веб-страницу, и сервер отвечает HTML-документом. Затем клиент отправляет дополнительные запросы последовательно по мере обнаружения ресурсов CSS и JavaScript в HTML:

Установление TCP-соединения — дорогостоящая часть взаимодействия клиента и сервера, а медленный старт TCP означает, что более длительные соединения работают быстрее, чем вновь созданные. HTTP/1.1 позволяет повторно использовать TCP-соединение для нескольких запросов и ответов, поэтому не нужно создавать новое соединение для каждого запроса. Однако клиентам по-прежнему приходилось ждать загрузки каждого ресурса, прежде чем запрашивать следующий (блокировка «начала очереди» или head-of-line blocking). Чтобы обойти это, большинство браузеров разрешают до шести TCP-соединений на один веб-сайт (или источник). С шестью параллельными соединениями браузеры могут одновременно получать несколько ресурсов, используя модель HTTP/1.1, что значительно повысило производительность.

HTTP/1.1 был впервые опубликован как RFC 2068 в январе 1997 года.

### Более десяти лет развития

Эволюция спецификаций. HTTP/1.1, опубликованный в 1999 году (RFC 2616), просуществовал почти без изменений более 15 лет. В 2014 вышли RFC 7230–7235, в 2022 — полное обновление: семантика (RFC 9110), кеширование (RFC 9111), HTTP/1.1 (RFC 9112), HTTP/2 (RFC 9113), HTTP/3 (RFC 9114). Спецификация впервые получила статус Internet Standard (STD 97).

SSL/TLS. В конце 1994 года Netscape добавил поверх TCP шифрованный слой — SSL. Позже он стал TLS. По мере превращения веба из академической сети в коммерческую среду шифрование стало необходимостью.

REST и WebDAV. В 1996 HTTP расширили для удалённого редактирования (WebDAV, CardDAV, CalDAV), но требовалась поддержка сервером. В 2000 появился REST — тот же HTTP/1.1 без новых методов, просто доступ к URI. Минус: каждый сайт делал свой RESTful API. С 2005 добавились Server-Sent Events и WebSocket (апгрейд HTTP-соединения).

Ослабление same-origin. Веб-модель безопасности появилась позже HTTP. Со временем часть ограничений сняли через заголовки: CORS — для междоменных запросов, CSP — для контроля источников контента. Появились DNT, X-Frame-Options, Upgrade-Insecure-Requests.

### HTTP/2 — протокол для повышения производительности.

С годами веб-страницы становились всё сложнее. Некоторые из них даже представляли собой самостоятельные приложения. Появлялось всё больше визуального контента, а также увеличивался объём и размер скриптов, добавляющих интерактивность. Значительно больше данных передавалось по гораздо большему количеству HTTP-запросов, что создавало дополнительную сложность и накладные расходы для соединений HTTP/1.1. Чтобы компенсировать это, Google в начале 2010-х годов внедрила экспериментальный протокол SPDY. Этот альтернативный способ обмена данными между клиентом и сервером привлёк внимание разработчиков, работающих как с браузерами, так и с серверами. SPDY обеспечил повышение скорости отклика и решил проблему дублирования передачи данных, став основой для протокола HTTP/2.

Протокол HTTP/2 отличается от HTTP/1.1 несколькими способами:
- Это бинарный протокол, а не текстовый. Его нельзя прочитать и создать вручную. Несмотря на это препятствие, он позволяет внедрять улучшенные методы оптимизации.
- Это мультиплексный протокол. Параллельные запросы могут выполняться по одному и тому же соединению, снимая ограничения протокола HTTP/1.x.
- Это мультиплексный протокол. Он сжимает заголовки. Поскольку они часто похожи в наборе запросов, это устраняет дублирование и накладные расходы на передаваемые данные.

Официально стандартизированный в мае 2015 года, HTTP/2 достиг пика использования в январе 2022 года, составив 46,9% всех веб-сайтов. Веб-сайты с высокой посещаемостью продемонстрировали наиболее быстрое внедрение в стремлении сэкономить на накладных расходах на передачу данных и, соответственно, бюджеты.

Это быстрое внедрение, вероятно, произошло потому, что HTTP/2 не требовал изменений в веб-сайтах и ​​приложениях. Для его использования был необходим только обновленный сервер, взаимодействующий с современным браузером. Для запуска внедрения требовалось лишь ограниченное количество групп, и по мере обновления устаревших версий браузеров и серверов использование естественным образом увеличивалось без значительных усилий со стороны веб-разработчиков.

### Эволюция после HTTP/2

Расширяемость HTTP по-прежнему используется для добавления новых функций. В частности, можно привести примеры новых расширений протокола HTTP, появившихся в 2016 году:

- Поддержка Alt-Svc позволила разделить идентификацию и местоположение данного ресурса. Это означало более интеллектуальный механизм кэширования CDN.
- Введение клиентских подсказок позволило браузеру или клиенту заблаговременно передавать серверу информацию о своих требованиях и аппаратных ограничениях.
- Введение префиксов, связанных с безопасностью, в заголовок Cookie помогло гарантировать, что защищенные файлы cookie не могут быть изменены.

### HTTP/3 - HTTP поверх QUIC

Следующая основная версия HTTP, HTTP/3, имеет ту же семантику, что и более ранние версии HTTP, но использует QUIC вместо TCP для транспортного уровня. К октябрю 2022 года 26% всех веб-сайтов использовали HTTP/3.

QUIC разработан для обеспечения гораздо меньшей задержки для HTTP-соединений. Как и HTTP/2, это мультиплексированный протокол, но HTTP/2 работает поверх одного TCP-соединения, поэтому обнаружение потери пакетов и повторная передача, обрабатываемые на уровне TCP, могут блокировать все потоки. QUIC работает с несколькими потоками поверх UDP и реализует обнаружение потери пакетов и повторную передачу независимо для каждого потока, так что в случае ошибки блокируется только поток, содержащий данные в этом пакете.

Определенный в RFC 9114, HTTP/3 поддерживается большинством основных браузеров, включая Chromium (и его варианты, такие как Chrome и Edge) и Firefox.

> **Зачем это Go-разработчику.** Эволюция HTTP отражена в `net/http` напрямую:
> 
> * `http.Server` по умолчанию включает HTTP/2 при TLS. `Protocols` управляет HTTP/1, HTTP/2 и h2c.
> * `Transport` кэширует TCP-соединения (`MaxIdleConns`) — это и есть keep-alive из HTTP/1.1.
> * HTTP/2-фичи: `Pusher` для server push, `HTTP2Config` для настройки фреймов и потоков.
> * HTTP/3 пока не в stdlib, но доступен через `golang.org/x/net/quic`.

## 3. Семантика HTTP

Семантика HTTP — это набор правил, определяющих смысл каждого элемента протокола: что означают методы, коды состояния и заголовки. В 2022 году семантика выделена в отдельный RFC 9110, общий для HTTP/1.1, HTTP/2 и HTTP/3. Это значит: независимо от версии протокола, `GET /users` означает одно и то же.

***

### Ресурсы и URI

Центральное понятие HTTP — **ресурс**. Ресурсом может быть HTML-страница, изображение, результат поиска, запись в базе данных — всё, что можно идентифицировать. Каждый ресурс идентифицируется **URI** (Uniform Resource Identifier).

**URL** (Uniform Resource Locator) — подмножество URI, которое не только идентифицирует, но и указывает *местоположение* ресурса:

```
https://api.example.com:443/users/42?lang=en#profile
\____/  \____________/ \_/ \_______/ \_____/ \_____/
схема     хост         порт  путь     запрос   фрагмент
```

- **Схема** — `http` или `https`
- **Хост** — домен или IP
- **Порт** — по умолчанию 80 для HTTP, 443 для HTTPS
- **Путь** — иерархический идентификатор ресурса на сервере
- **Запрос (query)** — `?key=value&key2=value2`, опциональные параметры
- **Фрагмент** — `#section`, не отправляется на сервер, используется браузером для прокрутки

В Go URL представлен типом `*url.URL` из пакета `net/url`. Парсинг: `url.Parse(rawURL)`. В HTTP-обработчике URL доступен через `r.URL`.

> **Зачем это Go-разработчику.** `r.URL.Path` — путь без query-параметров, `r.URL.Query()` — разобранные параметры запроса. Никогда не склеивайте URL руками через `+` — используйте `url.JoinPath` или `url.URL.String()`.

***

### HTTP-методы

Метод — это глагол, определяющий желаемое действие над ресурсом. В Go методы представлены константами `http.MethodGet`, `http.MethodPost` и т.д.

#### Свойства методов

| Свойство | Значение |
|---|---|
| **Безопасный (safe)** | Не изменяет состояние сервера. GET, HEAD, OPTIONS |
| **Идемпотентный (idempotent)** | Повторный запрос даёт тот же результат. GET, PUT, DELETE, HEAD, OPTIONS |
| **Кешируемый (cacheable)** | Ответ можно сохранить и переиспользовать. GET, HEAD |

#### Основные методы

**GET** — получить ресурс. Безопасный, идемпотентный, кешируемый. Не имеет тела запроса. Параметры передаются в URL: `/search?q=golang`.

```go
resp, err := http.Get("https://api.example.com/users/42")
```

**HEAD** — то же, что GET, но без тела ответа. Используется для проверки существования ресурса или получения метаданных (заголовков).

```go
resp, err := http.Head("https://example.com/file.pdf")
// resp.ContentLength, resp.Header доступны, resp.Body пуст
```

**POST** — создать новый ресурс или выполнить действие. Не безопасный, не идемпотентный. Тело запроса содержит данные создаваемого ресурса.

```go
resp, err := http.Post(url, "application/json", body)
```

**PUT** — заменить ресурс целиком (или создать, если не существует). Идемпотентный: два одинаковых PUT дают тот же результат.

```go
req, _ := http.NewRequest(http.MethodPut, url, body)
req.Header.Set("Content-Type", "application/json")
resp, err := client.Do(req)
```

**PATCH** — частичное обновление ресурса. В отличие от PUT, не требует передачи всего объекта. Не идемпотентный.

**DELETE** — удалить ресурс. Идемпотентный: повторное удаление не меняет состояния.

**OPTIONS** — запросить поддерживаемые методы. Ответ содержит заголовок `Allow` со списком методов, доступных для данного URL. Автоматически обрабатывается `ServeMux` (если не задан `DisableGeneralOptionsHandler`).

**CONNECT** — установить туннель (обычно для HTTPS через прокси). Используется `Transport` для проксирования.

**TRACE** — эхо-запрос для диагностики. Возвращает то, что получил сервер, позволяя увидеть изменения, внесённые промежуточными узлами.

> **Зачем это Go-разработчику.** Выбор метода влияет на поведение `Transport`: только GET, HEAD, OPTIONS, TRACE автоматически повторяются при сетевых ошибках. POST/PATCH требуют явного `GetBody` для повторной отправки. Идемпотентность — ключевое свойство при проектировании API.

***

### Коды состояния

Код состояния — трёхзначное число в ответе сервера, сообщающее результат обработки запроса. В Go все коды — константы вида `http.StatusOK` (200), `http.StatusNotFound` (404) и т.д.

#### 1xx — Informational (информационные)

Промежуточный ответ, не финальный. Сервер продолжает обработку.

| Код | Значение | Когда используется |
|---|---|---|
| `100 Continue` | Продолжай | Клиент может отправлять тело запроса |
| `101 Switching Protocols` | Смена протокола | Апгрейд на WebSocket |
| `103 Early Hints` | Предварительные подсказки | Сервер сообщает заголовки до финального ответа |

#### 2xx — Success (успех)

Запрос успешно обработан.

| Код | Значение | Когда используется |
|---|---|---|
| `200 OK` | Успешно | Стандартный ответ на GET, PUT, PATCH |
| `201 Created` | Создано | Результат успешного POST, заголовок `Location` указывает URL нового ресурса |
| `202 Accepted` | Принято | Запрос принят, но ещё не обработан (асинхронная операция) |
| `204 No Content` | Нет содержимого | Успех без тела ответа (часто после DELETE) |

#### 3xx — Redirection (перенаправление)

Клиенту нужно выполнить дополнительное действие.

| Код | Значение | Особенность |
|---|---|---|
| `301 Moved Permanently` | Перемещено навсегда | Меняет метод на GET |
| `302 Found` | Найдено | Исторически меняет метод на GET |
| `303 See Other` | Смотри другое | Всегда GET |
| `307 Temporary Redirect` | Временно | Сохраняет метод и тело |
| `308 Permanent Redirect` | Навсегда | Сохраняет метод и тело |

> **Зачем это Go-разработчику.** `Client` по умолчанию следует 10 редиректам. 301/302/303 превращают POST в GET, а 307/308 сохраняют метод. `http.Redirect(w, r, url, http.StatusFound)` — стандартный способ редиректа на стороне сервера. `ErrUseLastResponse` запрещает следовать редиректам.

#### 4xx — Client Error (ошибка клиента)

Запрос некорректен.

| Код | Значение | Типичная причина |
|---|---|---|
| `400 Bad Request` | Плохой запрос | Невалидный синтаксис |
| `401 Unauthorized` | Не авторизован | Требуется аутентификация |
| `403 Forbidden` | Запрещено | Доступ запрещён независимо от аутентификации |
| `404 Not Found` | Не найдено | Ресурс не существует |
| `405 Method Not Allowed` | Метод не разрешён | Неправильный HTTP-метод |
| `408 Request Timeout` | Таймаут запроса | Сервер не дождался полного запроса |
| `409 Conflict` | Конфликт | Конфликт состояния (например, при PUT) |
| `429 Too Many Requests` | Слишком много запросов | Rate limiting |

#### 5xx — Server Error (ошибка сервера)

Сервер не смог обработать валидный запрос.

| Код | Значение | Типичная причина |
|---|---|---|
| `500 Internal Server Error` | Внутренняя ошибка | Паника в обработчике, необработанная ошибка |
| `502 Bad Gateway` | Плохой шлюз | Прокси получил некорректный ответ от upstream |
| `503 Service Unavailable` | Сервис недоступен | Сервер перегружен или на обслуживании |
| `504 Gateway Timeout` | Таймаут шлюза | Прокси не дождался ответа от upstream |

```go
// Отправка ошибки в Go
http.Error(w, "Resource not found", http.StatusNotFound)

// Ответ с произвольным кодом
w.WriteHeader(http.StatusCreated)
```

> **Зачем это Go-разработчику.** Не придумывайте свои коды. Весь Интернет понимает 429 как rate limit, а 503 как временную недоступность. `net/http` отправляет `400 Bad Request` при ошибке парсинга запроса и `100 Continue` при `Expect: 100-continue` автоматически. Коды-константы (`http.StatusOK`, а не `200`) делают код читаемым.

### HTTP-заголовки

Заголовки — это метаданные запроса или ответа в формате `Имя: Значение`. Регистр имени не важен, но `net/http` каноникализирует имена через `CanonicalHeaderKey`.

В Go заголовки представлены типом `http.Header` — это `map[string][]string`. Один ключ может иметь несколько значений.

```go
// Чтение заголовка запроса
auth := r.Header.Get("Authorization")
// Установка заголовка ответа
w.Header().Set("Content-Type", "application/json")
// Добавление (без перезаписи существующего)
w.Header().Add("Set-Cookie", "session=abc")
```

#### Категории заголовков

**Общие (General)** — применимы к запросам и ответам:
- `Cache-Control` — директивы кеширования
- `Connection` — управление соединением (`keep-alive`, `close`)
- `Date` — дата и время создания сообщения
- `Transfer-Encoding` — способ кодирования тела (`chunked`)

**Запроса (Request)** — только от клиента:
- `Host` — домен сервера (единственный обязательный в HTTP/1.1)
- `User-Agent` — информация о клиенте
- `Accept`, `Accept-Encoding`, `Accept-Language` — согласование контента
- `Authorization` — учётные данные
- `Cookie` — куки, сохранённые для этого домена
- `Content-Type`, `Content-Length` — описание тела запроса
- `If-Modified-Since`, `If-None-Match` — условные запросы

**Ответа (Response)** — только от сервера:
- `Content-Type`, `Content-Length`, `Content-Encoding` — описание тела
- `Set-Cookie` — установка куки у клиента
- `Location` — URL для редиректа (с 3xx или 201)
- `WWW-Authenticate` — схема аутентификации
- `ETag`, `Last-Modified` — версия ресурса для условных запросов
- `Access-Control-Allow-Origin` — CORS

```go
// Установка cookie
http.SetCookie(w, &http.Cookie{
    Name:     "session",
    Value:    "abc123",
    HttpOnly: true,
    Secure:   true,
    SameSite: http.SameSiteLaxMode,
})
```

> **Зачем это Go-разработчику.** `r.Header.Get("X-Forwarded-For")` ненадёжен — любой клиент может его подделать. Используйте `r.RemoteAddr` для IP клиента. `w.Header().Set()` должен вызываться до `w.WriteHeader()` или `w.Write()` — после записи заголовки менять нельзя. Автоматические заголовки (`Date`, `Content-Type` при sniffing) добавляются сервером, но их можно подавить, установив в `nil`.
