# Строки в Go

## 1. Введение

**Строка** — один из самых часто используемых типов в Go. Каждый запрос к API, каждая запись в лог, каждый конфигурационный параметр — это строка. При этом строки в Go устроены иначе, чем во многих других языках: они неизменяемы, хранятся в кодировке UTF-8, а их внутреннее представление напрямую влияет на производительность операций с ними.

В этом документе мы разберём:

* внутреннее устройство строки в памяти Go;
* как строки связаны с байтами и рунами;
* кодировку UTF-8 и её влияние на длину и итерацию;
* преобразования между `string`, `[]byte` и `[]rune` — и их цену;
* механику сравнения строк и оптимизации компилятора;
* ключевые пакеты стандартной библиотеки: `strings`, `bytes`, `strconv`;
* эффективную конкатенацию через `strings.Builder`;
* типичные ошибки и лучшие практики.

> **Зачем это Go-разработчику.** Понимание внутреннего устройства строк позволяет писать код, который не создаёт лишних аллокаций, корректно работает с не-ASCII текстом и не содержит трудноуловимых багов, связанных с памятью.

## 2. Внутреннее устройство строк

### Структура string

Для стандартного компилятора Go внутренняя структура строкового типа объявляется следующим образом:

```go
type stringStruct struct {
    str unsafe.Pointer
    len int
}

```

Из объявления видно, что **строка** — это, по сути, обёртка над последовательностью байтов. Фактически строку можно рассматривать как неизменяемый по элементам срез байтов. В Go **`byte`** — это встроенный псевдоним типа `uint8`.

Строка состоит из двух полей:

* указатель на базовый массив байтов в памяти;
* длина (количество байтов).

Нулевое значение строкового типа — пустая строка, которая может быть представлена как `""` или ` ` \`\` в литерале.

### Неизменяемость

Содержимое (базовые байты) строковых значений **неизменяемо**. Нельзя изменить отдельный байт строки — выражение `s[i]` не является адресуемым. Адресуемое строковое значение можно перезаписать только целиком, присвоив ему другое строковое значение.

### Разделение базового массива

Для стандартного компилятора Go строковая переменная назначения и исходное строковое значение при присваивании используют **одну и ту же базовую последовательность байтов** в памяти. Копирования данных не происходит — копируется только сама структура (указатель и длина).

Результат выражения подстроки `s[start:end]` также использует ту же базовую последовательность байтов, что и исходная строка `s`. Это эффективно по памяти, но имеет важное следствие: подстрока удерживает в памяти **весь** родительский массив байтов, даже если нужна лишь малая его часть.

### Отсутствие методов у типа string

Встроенный тип `string` не имеет собственных методов (как и большинство других встроенных типов в Go). Однако мы можем:

* использовать функции из стандартного пакета `strings` для выполнения операций со строками;
* вызывать встроенную функцию `len` для получения длины строки в байтах;
* использовать синтаксис доступа к элементу `s[i]` для получения `i`-го байта (только чтение);
* использовать синтаксис подсреза `s[start:end]` для получения подстроки.

> **Зачем это Go-разработчику.** Строка — структура из указателя на массив байтов и длины. Неизменяемость гарантирует безопасность при конкурентном доступе. `len(s)` выполняется за O(1). Подстрока через `s[1:3]` не копирует данные, а создаёт новый заголовок на тот же массив — это дёшево, но удержание маленькой подстроки не даёт GC освободить большой исходный массив. Для принудительного копирования используйте `strings.Clone(s)`.

## 3. Строковые литералы и базовые операции

### Литералы

Go поддерживает два стиля строковых литералов:

* **Интерпретируемые литералы** (в двойных кавычках `"..."`) — поддерживают escape-последовательности (`\n`, `\t`, `\"` и другие). Компилятор проверяет их на валидность UTF-8.
* **Необработанные литералы** (в обратных кавычках `` `...` ``) — всё содержимое воспринимается буквально, без обработки escape-последовательностей. Удобны для регулярных выражений, JSON-строк, многострочного текста.

Строковые значения могут использоваться в качестве констант — наряду с логическими и числовыми значениями.

### Конкатенация

Строки можно объединять с помощью операторов `+` и `+=`. При каждой конкатенации создаётся новая строка с новым базовым массивом, в который копируются данные обеих исходных строк.

### Сравнение операторами

Все строковые типы можно сравнивать с помощью операторов `==` и `!=`. Также, подобно целым числам, две строки можно сравнивать операторами `>`, `<`, `>=`, `<=`. При сравнении сравниваются базовые байты **побайтово**. Если одна строка является префиксом другой и вторая длиннее, то вторая считается большей.

> **Зачем это Go-разработчику.** Строковые литералы бывают интерпретируемые (`"..."`, с escape-последовательностями) и raw (`` `...` ``, без них — удобны для регулярок и путей). Конкатенация через `+` создаёт новую строку; для массовой сборки используйте `strings.Builder`. Сравнение через `==` и `!=` побайтовое; `<`, `>` — лексикографическое по байтам. Для сравнения без учёта регистра — `strings.EqualFold`.

## 4. Строки, байты и руны: кодировка UTF-8

### Кодовые точки Unicode и руны

**Стандарт Unicode** определяет уникальное значение для каждого символа во всех языках мира. Базовой единицей Unicode является не символ, а **кодовая точка** (code point). Большинству символов соответствует одна кодовая точка, но некоторые символы состоят из нескольких кодовых точек.

В Go кодовые точки представлены типом **`rune`** — встроенным псевдонимом `int32`. Каждая руна хранит одно 32-битное значение кодовой точки.

### Кодировка UTF-8

Go хранит строки в кодировке **UTF-8**. В этой кодировке каждая кодовая точка занимает от 1 до 4 байтов:

| Диапазон кодовых точек | Байт на символ | Пример                                         |
| ---------------------- | -------------- | ---------------------------------------------- |
| U+0000–U+007F          | 1              | Английские буквы, цифры, базовые символы ASCII |
| U+0080–U+07FF          | 2              | Кириллица, греческий, арабский                 |
| U+0800–U+FFFF          | 3              | Китайские, японские, корейские иероглифы       |
| U+10000–U+10FFFF       | 4              | Эмодзи, редкие иероглифы                       |

Во время компиляции недопустимые строковые константы UTF-8 приведут к ошибке компиляции. Однако во время выполнения среда Go не может гарантировать валидность UTF-8 для строк, полученных извне (сеть, файлы).

> **Зачем это Go-разработчику.** Ключевой вывод: `len(s)` возвращает количество **байтов**, а не символов. Строка `"Привет"` содержит 6 символов, но `len("Привет")` вернёт 12, так как каждая буква кириллицы кодируется двумя байтами. Для работы с отдельными символами всегда используйте руны.

## 5. Преобразования между строками, байтовыми и рунными срезами

### string ↔ \[]byte

Строковое значение можно явно преобразовать в байтовый срез и наоборот. Байтовый срез — это срез с базовым типом элемента `[]byte`.

При преобразовании строки в `[]byte` результирующий байтовый срез представляет собой **глубокую копию** базовой последовательности байтов строки. Причина: элементы среза изменяемы, а байты строки неизменяемы — они не могут делить одну память.

При преобразовании `[]byte` в строку базовая последовательность байтов результирующей строки также является полной копией среза. Каждое такое преобразование требует выделения памяти.

Стандартный компилятор Go оптимизирует некоторые особые случаи таких преобразований, чтобы не создавать глубокие копии (например, когда срез не используется после преобразования).

### string ↔ \[]rune

Строковое значение можно явно преобразовать в рунический срез `[]rune` и наоборот.

При преобразовании `[]rune` в строку каждый элемент среза кодируется в UTF-8 (1–4 байта) и сохраняется в результирующей строке. Если значение руны выходит за пределы допустимых кодовых точек Unicode, оно отображается как `0xFFFD` — **заменяющий символ Unicode** (Unicode replacement character). `0xFFFD` кодируется в UTF-8 тремя байтами: `0xEF 0xBF 0xBD`.

При преобразовании строки в `[]rune` байты строки рассматриваются как UTF-8 представления кодовых точек. Невалидные последовательности UTF-8 преобразуются в руну `0xFFFD`.

### Прямое преобразование \[]byte ↔ \[]rune

Прямое преобразование между `[]byte` и `[]rune` в Go не поддерживается. Возможные обходные пути:

* использовать строку как промежуточное значение: `[]rune(string(byteSlice))` — удобно, но требует двух глубоких копирований;
* использовать функцию `Runes` из стандартного пакета `bytes` для преобразования `[]byte` в `[]rune` (обратного преобразования в пакете `bytes` нет);
* использовать функции из пакета `unicode/utf8` для потокового декодирования.

### Определение глубокой копии

**Глубокая копия** (deep copy) — это процесс создания нового объекта, полностью дублирующего исходный, включая все вложенные структуры, указатели, слайсы и другие данные. В отличие от поверхностной копии (shallow copy), глубокая копия гарантирует, что изменения в новом объекте не повлияют на исходный.

> **Зачем это Go-разработчику.** Преобразования `string ↔ []byte` и `string ↔ []rune` — потенциально дорогие операции, выделяющие память. В высоконагруженном коде избегайте частых преобразований. Если вы работаете с байтовыми данными (например, из сети), используйте пакет `bytes` вместо преобразования в `string` и обратно. Начиная с Go 1.11, выражение `len([]rune(s))` оптимизируется компилятором и не создаёт глубокой копии.

## 6. Итерация по строкам: for-range и длина

### Итерация по рунам

Цикл `for-range`, применённый к строке, перебирает **кодовые точки Unicode** (руны), а не байты. Невалидные последовательности UTF-8 интерпретируются как руна `0xFFFD`.

Значение индекса в `for-range` может быть не непрерывным: индекс представляет собой **байтовую позицию** начала руны в строке, и поскольку на одну руну может приходиться несколько байтов, индексы могут перескакивать:

```go
s := "Hello, 世界"
for i, r := range s {
    fmt.Printf("%d: %c\n", i, r)
}
// 0: H
// 1: e
// ...
// 7: 世   (занимает байты 7, 8, 9)
// 10: 界  (занимает байты 10, 11, 12)

```

### Длина строки в байтах

`len(s)` возвращает количество байтов в строке `s`. Временная сложность — **O(1)**, так как длина хранится непосредственно в структуре `string`.

### Количество рун

Для получения количества рун в строке существует три способа:

* использовать цикл `for-range`;
* использовать `utf8.RuneCountInString(s)` из пакета `unicode/utf8`;
* использовать `len([]rune(s))`.

Эффективность первых двух способов практически одинакова. Начиная с Go 1.11, компилятор оптимизирует третий способ, чтобы избежать ненужного глубокого копирования, и он также эффективен. Временная сложность всех трёх способов — **O(n)**, где n — длина строки в байтах.

> **Зачем это Go-разработчику.** Для посимвольного обхода строки всегда используйте `for-range` — это идиоматично и корректно обрабатывает многобайтовые руны. Никогда не итерируйтесь индексами `for i := 0; i < len(s); i++` для обхода символов — это даст отдельные байты, а не символы.

## 7. Сравнение строк: механика и оптимизации

### Как работает сравнение

Сравнение двух строк — это побайтовое сравнение их базовых массивов.

Компилятор Go применяет следующие оптимизации:

* Для операторов `==` и `!=`: если длины строк не равны, строки гарантированно не равны — дальнейшее сравнение байтов не требуется.
* Если **указатели** на базовые байтовые последовательности сравниваемых строк равны, то результат сравнения сводится к сравнению длин. Две строки с одинаковым указателем на данные и одинаковой длиной идентичны (O(1)).
* Если указатели не равны, производится побайтовое сравнение от начала к концу — O(n), где n — длина меньшей строки.

### Сложность сравнения

Для двух одинаковых строк сложность сравнения зависит от того, указывают ли они на одну и ту же последовательность байтов:

* Указатели равны → O(1)
* Указатели различны → O(n)

> **Зачем это Go-разработчику.** При использовании строк в качестве ключей `map` сравнение ключей использует эти же оптимизации. Если вы многократно создаёте одинаковые строки, компилятор может интернировать их и сравнение станет O(1). Знание о том, что подстрока разделяет память с родительской строкой, объясняет, почему сравнение подстроки с родителем или с другой такой же подстрокой может быть быстрее.

## 8. Пакет strings: основные операции

Пакет `strings` стандартной библиотеки предоставляет функции для работы со строками в кодировке UTF-8. Ниже функции сгруппированы по категориям.

### Поиск

| Функция                  | Назначение                                                      |
| ------------------------ | --------------------------------------------------------------- |
| `Contains(s, substr)`    | Проверяет, содержит ли строка подстроку                         |
| `ContainsAny(s, chars)`  | Проверяет, содержит ли строка хотя бы один символ из набора     |
| `HasPrefix(s, prefix)`   | Проверяет, начинается ли строка с указанного префикса           |
| `HasSuffix(s, suffix)`   | Проверяет, заканчивается ли строка указанным суффиксом          |
| `Index(s, substr)`       | Возвращает индекс первого вхождения подстроки (или -1)          |
| `LastIndex(s, substr)`   | Возвращает индекс последнего вхождения подстроки (или -1)       |
| `IndexAny(s, chars)`     | Возвращает индекс первого вхождения любого символа из набора    |
| `LastIndexAny(s, chars)` | Возвращает индекс последнего вхождения любого символа из набора |
| `Count(s, substr)`       | Количество непересекающихся вхождений подстроки                 |

### Трансформация

| Функция                   | Назначение                                                   |
| ------------------------- | ------------------------------------------------------------ |
| `ToUpper(s)`              | Преобразует все буквенные символы в верхний регистр          |
| `ToLower(s)`              | Преобразует все буквенные символы в нижний регистр           |
| `Replace(s, old, new, n)` | Заменяет первые `n` вхождений `old` на `new` (`n < 0` — все) |
| `ReplaceAll(s, old, new)` | Заменяет все вхождения `old` на `new`                        |

### Разделение и сборка

| Функция            | Назначение                                                |
| ------------------ | --------------------------------------------------------- |
| `Split(s, sep)`    | Разбивает строку на срез подстрок по разделителю          |
| `Join(elems, sep)` | Объединяет элементы среза в одну строку через разделитель |

### Обрезка

| Функция                 | Назначение                                                    |
| ----------------------- | ------------------------------------------------------------- |
| `Trim(s, cutset)`       | Удаляет символы из `cutset` с обеих сторон строки             |
| `TrimLeft(s, cutset)`   | Удаляет символы из `cutset` слева                             |
| `TrimRight(s, cutset)`  | Удаляет символы из `cutset` справа                            |
| `TrimSpace(s)`          | Удаляет пробельные символы с обеих сторон                     |
| `TrimPrefix(s, prefix)` | Удаляет префикс, если строка с него начинается                |
| `TrimSuffix(s, suffix)` | Удаляет суффикс, если строка им заканчивается                 |
| `TrimFunc(s, f)`        | Удаляет символы с обеих сторон, для которых `f(rune) == true` |
| `TrimLeftFunc(s, f)`    | То же, только слева                                           |
| `TrimRightFunc(s, f)`   | То же, только справа                                          |

`TrimLeft` и `TrimRight` не стоит путать с `TrimPrefix` и `TrimSuffix`. Первые удаляют **любой набор символов** по одному с края, вторые — **конкретную подстроку** только если она точно совпадает с краем.

> **Зачем это Go-разработчику.** Пакет `strings` — основной инструмент обработки текста в Go. Практически любая задача по очистке пользовательского ввода, парсингу логов, форматированию вывода решается функциями этого пакета. Знание его API избавляет от написания велосипедов и снижает вероятность ошибок, особенно при работе с UTF-8.

## 9. Эффективная конкатенация: strings.Builder

### Проблема конкатенации через +

Каждое использование `+` или `+=` для строк создаёт новую строку и копирует в неё данные обеих операндов. В цикле это приводит к квадратичной сложности O(n²):

```go
// Плохо: O(n²) аллокаций и копирований
var s string
for _, item := range items {
    s += item + ", "
}

```

На каждом шаге цикла создаётся новая строка, в которую копируются все предыдущие данные плюс новое значение.

### strings.Builder

`strings.Builder` решает эту проблему. Он хранит внутренний буфер `[]byte` и при каждой записи дописывает данные в него. При нехватке ёмкости буфер расширяется с запасом, что даёт **амортизированную сложность O(n)**.

```go
// Хорошо: амортизированное O(n)
var b strings.Builder
for _, item := range items {
    b.WriteString(item)
    b.WriteString(", ")
}
result := b.String()

```

### Основные методы

| Метод            | Назначение                                    |
| ---------------- | --------------------------------------------- |
| `WriteString(s)` | Дописывает строку                             |
| `WriteByte(c)`   | Дописывает один байт                          |
| `WriteRune(r)`   | Дописывает одну руну                          |
| `Grow(n)`        | Резервирует место минимум для `n` байтов      |
| `Reset()`        | Сбрасывает буфер для повторного использования |
| `String()`       | Возвращает накопленную строку                 |
| `Len()`          | Возвращает текущую длину                      |
| `Cap()`          | Возвращает текущую ёмкость                    |

Рекомендации:

* используйте `Grow(n)` перед циклом, если вы знаете или можете оценить итоговый размер — это исключает дополнительные расширения буфера;
* используйте `Reset()` для переиспользования Builder'а между итерациями;
* не копируйте `strings.Builder` после первой мутации — это приведёт к панике.

> **Зачем это Go-разработчику.** Формирование тела HTTP-ответа, сборка JSON вручную, генерация HTML-шаблонов, построение SQL-запросов — во всех этих случаях `strings.Builder` критически важен для производительности. Разница между `+=` и `Builder` на тысячах итераций может составлять порядки.

## 10. Пакет bytes: строковые операции над \[]byte

### Зеркальный API

Пакет `bytes` предоставляет тот же набор функций, что и `strings`, но для типа `[]byte`:

* `bytes.Contains`, `bytes.HasPrefix`, `bytes.HasSuffix`
* `bytes.Index`, `bytes.LastIndex`, `bytes.Count`
* `bytes.Split`, `bytes.Join`
* `bytes.Trim`, `bytes.TrimSpace`, `bytes.TrimPrefix`
* `bytes.ToUpper`, `bytes.ToLower`
* `bytes.Replace`, `bytes.ReplaceAll`

### Когда использовать bytes

Если данные уже представлены в виде `[]byte` — например, прочитаны из файла, получены из сети, сгенерированы хеш-функцией — работайте с `bytes` напрямую вместо преобразования в `string` и обратно:

```go
// Плохо: лишнее преобразование и аллокация
data, _ := io.ReadAll(r)
s := string(data)
if strings.Contains(s, "needle") { ... }

// Хорошо: работаем с байтами напрямую
data, _ := io.ReadAll(r)
if bytes.Contains(data, []byte("needle")) { ... }

```

> **Зачем это Go-разработчику.** Каждое преобразование `[]byte` в `string` и обратно создаёт глубокую копию — а значит, аллокацию. Работа с `bytes` напрямую исключает эту цену. В высокопроизводительном сетевом коде и при обработке больших объёмов данных это даёт ощутимый прирост производительности.

## 11. Преобразование строк в числа и обратно: strconv

Пакет `strconv` (string conversion) предназначен для преобразования между строками и числовыми типами.

### Строка в число

| Функция                       | Назначение                                                   |
| ----------------------------- | ------------------------------------------------------------ |
| `Atoi(s)`                     | Строка → `int`. Сокращение для `ParseInt(s, 10, 0)`          |
| `ParseInt(s, base, bitSize)`  | Строка → `int64` с указанием системы счисления и разрядности |
| `ParseFloat(s, bitSize)`      | Строка → `float64`                                           |
| `ParseBool(s)`                | Строка → `bool`                                              |
| `ParseUint(s, base, bitSize)` | Строка → `uint64`                                            |

### Число в строку

| Функция                              | Назначение                                               |
| ------------------------------------ | -------------------------------------------------------- |
| `Itoa(i)`                            | `int` → строка. Сокращение для `FormatInt(int64(i), 10)` |
| `FormatInt(i, base)`                 | `int64` → строка в указанной системе счисления           |
| `FormatFloat(f, fmt, prec, bitSize)` | `float64` → строка с указанием формата и точности        |
| `FormatBool(b)`                      | `bool` → строка (`"true"` / `"false"`)                   |
| `FormatUint(i, base)`                | `uint64` → строка в указанной системе счисления          |

### Обработка ошибок

Все функции парсинга (`Atoi`, `ParseInt`, `ParseFloat` и другие) возвращают ошибку, которую **необходимо проверять**. Пропуск проверки — одна из самых частых ошибок:

```go
// Ошибка: игнорируется невалидный ввод
i, _ := strconv.Atoi(userInput)

// Правильно: ошибка обрабатывается
i, err := strconv.Atoi(userInput)
if err != nil {
    return fmt.Errorf("некорректное число %q: %w", userInput, err)
}

```

> **Зачем это Go-разработчику.** Парсинг query-параметров HTTP-запроса, значений переменных окружения, конфигурационных файлов, пользовательского ввода — повседневная задача. `strconv` — основной инструмент для этой работы. Всегда проверяйте ошибки парсинга: невалидный ввод — не исключительная ситуация, а штатный сценарий.

## 12. Типичные ошибки и лучшие практики

### Индексация строки возвращает байт

`str[i]` возвращает `i`-й **байт**, а не `i`-й символ. Для строки `"Привет"` выражение `s[0]` вернёт `0xD0` (первый байт буквы «П»), а не саму букву.

```go
// Неправильно: индексация по байтам
for i := 0; i < len(s); i++ {
    fmt.Printf("%c", s[i]) // выводит отдельные байты — кракозябры для кириллицы
}

// Правильно: итерация по рунам
for _, r := range s {
    fmt.Printf("%c", r)
}

```

### Конкатенация в цикле

Использование `+=` в цикле создаёт новую строку на каждой итерации. Всегда используйте `strings.Builder`:

```go
// Плохо
var result string
for _, s := range strs {
    result += s
}

// Хорошо
var b strings.Builder
for _, s := range strs {
    b.WriteString(s)
}
result := b.String()

```

### Разрыв руны при срезе

Срез строки по индексам работает на уровне байтов. Если граница среза попадает в середину многобайтовой руны, получается невалидная UTF-8 строка:

```go
s := "Привет"
sub := s[0:3]  // разрывает букву «П» (2 байта) и «р» (2 байта) — невалидный UTF-8

```

### Сравнение без учёта регистра

`strings.ToLower(s1) == strings.ToLower(s2)` создаёт две промежуточные строки. Используйте `strings.EqualFold(s1, s2)` для регистронезависимого сравнения.

### Утечка памяти через подстроки

Подстрока удерживает в памяти весь родительский массив байтов. Если вы сохраняете маленькую подстроку, а большая родительская строка должна быть освобождена — большая строка не освободится:

```go
// Проблема: bigString из 1 МБ не будет собрана GC
substring := bigString[0:5]

// Решение (Go 1.18+)
substring := strings.Clone(bigString[0:5])

```

### Не забывайте про bytes

Если данные пришли как `[]byte`, не преобразуйте их в `string` только для того, чтобы использовать функции `strings`. Используйте `bytes` напрямую.

> **Зачем это Go-разработчику.** Знание этих ошибок экономит часы отладки. Большинство из них не вызывают ошибок компиляции и падений, но приводят к повреждению данных (разрыв рун), утечкам памяти (подстроки) и деградации производительности (конкатенация в цикле).

## 13. Заключение

Строки в Go — это больше, чем просто последовательности символов. Понимание их внутреннего устройства, связи с байтами и рунами, а также владение инструментарием стандартной библиотеки позволяет писать корректный и производительный код.

Ключевые выводы:

* Строка — неизменяемая обёртка над `[]byte` с разделяемой памятью при присваивании и подсрезе.
* `len(s)` — байты, `for-range` — руны. Никогда не путайте их.
* Преобразования `string ↔ []byte` и `string ↔ []rune` создают глубокие копии — избегайте лишних преобразований.
* Сравнение строк оптимизировано компилятором: O(1) при общем указателе, O(n) в общем случае.
* `strings`, `bytes`, `strconv` и `strings.Builder` — ваш основной инструментарий. Знайте, когда применять каждый из них.

Для дальнейшего изучения рекомендуются: пакет `unicode/utf8` для низкоуровневой работы с UTF-8, пакет `fmt` для форматированного вывода и пакет `regexp` для работы с регулярными выражениями.
