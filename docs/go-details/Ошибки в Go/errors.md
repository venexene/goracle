# Ошибки в Go

В отличие от многих языков, где ошибки обрабатываются через исключения
(try/catch), Go придерживается принципа **«ошибки — это значения»**
(errors are values). Ошибка в Go — не аварийная ситуация, ломающая поток
управления, а обычное возвращаемое значение, которое вызывающий код
должен проверить и обработать.

Такой подход делает поток управления явным и предсказуемым: каждый вызов,
который может завершиться неудачей, возвращает ошибку, и программист
вынужден принять осознанное решение о том, что с ней делать. Это
исключает «невидимые» пути исполнения, характерные для исключений, и
делает код более надёжным и читаемым.

> **Зачем это Go-разработчику.** Понимание философии ошибок в Go

> определяет все дальнейшие паттерны: от того, как вы проектируете

> сигнатуры функций, до того, как выстраиваете архитектуру обработки

> сбоев в больших системах.

***

## 1. Базовая обработка ошибок

Центральным элементом системы ошибок в Go является встроенный интерфейс
**error**:

```go
type error interface {
    Error() string
}

```

Любой тип, реализующий этот метод, может быть использован как ошибка.
Стандартный идиоматический паттерн обработки ошибок заключается в
немедленной проверке возвращенного значения error после вызова функции:

```go
if err != nil {
    return err
}

```

Проверка `if err != nil` сразу после вызова функции делает поток
управления ясным и предотвращает использование невалидных результатов.
Возврат ошибки вверх по стеку вызовов позволяет вызывающему коду решить,
как на нее реагировать.

> **Зачем это Go-разработчику.** Это базовый паттерн, который вы будете

> использовать в каждой второй строке кода. Отсутствие проверки `if err

> != nil` — самая частая причина багов у новичков.

## 2. Добавление контекста с `fmt.Errorf`

Простые ошибки часто не несут достаточно информации о том, где в сложной
системе они произошли. Чтобы добавить контекст к ошибке, не теряя при
этом исходную, используется функция **fmt.Errorf** со специальным
глаголом форматирования **%w**.

```go
fmt.Errorf("failed to process: %w", err)

```

Глагол `%w` (от слова wrap — обернуть) в `fmt.Errorf` создает новую
ошибку, которая оборачивает исходную. Текст новой ошибки будет содержать
добавленный вами контекст и текст исходной ошибки. Важно, что исходная
ошибка не теряется — её можно будет позже извлечь или проверить с
помощью функций пакета `errors` (`Is` и `As`). Используйте `%w` ровно
один раз в вызове `fmt.Errorf`. Если хотите просто включить текст ошибки
в сообщение без сохранения возможности развернуть её, используйте **%v**.

> **Зачем это Go-разработчику.** Без оборачивания ошибок вы теряете

> причинно-следственную связь. `%w` — это клей, который строит цепочку

> `ошибка БД → ошибка репозитория → ошибка сервиса → HTTP 500`, и на

> каждом уровне видно, что именно пошло не так.

## 3. Разбор ошибок: `errors.Is` и `errors.As`

Когда ошибка передается вверх по стеку вызовов, часто обернутая
несколько раз, нам могут понадобиться инструменты для её анализа. Пакет
`errors` предоставляет две ключевые функции: **errors.Is** и
**errors.As**. Их часто путают, но они служат разным целям.

Механизм обхода цепочки ошибок основан на функции **errors.Unwrap(err
error) error**. Она вызывает метод `Unwrap()` у переданной ошибки и
возвращает следующую ошибку в цепочке (или `nil`, если обёрток больше
нет). И `errors.Is`, и `errors.As` используют `Unwrap` для
последовательного прохода по всем уровням оборачивания.

### `errors.Is` — проверка на конкретное значение

Функция **errors.Is(err error, target error) bool** проверяет, является
ли ошибка `err` (или любая ошибка в её цепочке обёрток) конкретным
значением ошибки `target`. Эта функция используется для проверки на так
называемые **сигнальные ошибки (sentinel errors)**. Это обычно
экспортируемые переменные типа `error`, предопределённые в пакетах
(например, `io.EOF`, `sql.ErrNoRows`, `os.ErrNotExist`) или ваши
собственные (`var ErrUserNotFound = errors.New("user not found")`).
`errors.Is` проходит по цепочке ошибок (используя метод `Unwrap()`, если
он есть у ошибки) и сравнивает каждую ошибку в цепочке с `target` с
помощью оператора `==`. Если какая-либо ошибка в цепочке реализует метод
`Is(target error) bool`, то будет вызван этот метод.

```go
errors.Is(err, io.EOF)

```

### `errors.As` — проверка типа и извлечение значения

**errors.As(err error, target interface{}) bool** проверяет, является ли
ошибка `err` (или любая ошибка в её цепочке обёрток) ошибкой
определённого типа. Если проверка успешна, она присваивает значение
найденной ошибки переменной `target`. Эта функция нужна, когда вам
недостаточно просто знать, что произошла ошибка определённого рода (как
с `Is`), а нужно получить доступ к полям или методам этой конкретной
ошибки для извлечения дополнительной информации. `target` должен быть
указателем на переменную того типа ошибки, которую вы ищете (например,
`var myErr *MyErrorType`). `errors.As` проходит по цепочке ошибок,
проверяя для каждой, соответствует ли она типу, на который указывает
`target`. Если соответствие найдено, `target` получает значение этой
ошибки, и функция возвращает `true`.

### Ключевое различие

| `errors.Is`                                                                       | `errors.As`                                                                               |
| --------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------- |
| Сравнивает с **конкретным значением** ошибки (например, `os.ErrNotExist`)         | Проверяет **тип** ошибки и извлекает её значение в переменную                             |
| Используйте, когда нужно знать, произошла ли именно эта предопределённая ситуация | Используйте, когда нужен доступ к данным или поведению, специфичным для этого типа ошибки |

> **Зачем это Go-разработчику.** `Is` — ваш инструмент для маршрутизации

> по предопределённым сценариям (EOF → закончить чтение, ErrNotExist →

> создать файл). `As` — инструмент для извлечения деталей (код ошибки,

> время, ID сущности) и принятия решений на их основе.

## 4. Кастомные ошибки

Иногда для передачи полной информации об ошибке недостаточно строки или
стандартных типов. В таких случаях создают **пользовательские
(кастомные) типы ошибок**.

```go
type QueryError struct {
    Query string
    Err   error
}

func (e *QueryError) Error() string {
    return fmt.Sprintf("query %s: %v", e.Query, e.Err)
}

```

**Кастомные ошибки** — это обычно структуры, реализующие интерфейс
`error`. Они позволяют:

* Передавать структурированную информацию (коды ошибок, поля, флаги).
* Осуществлять программный анализ ошибки вызывающим кодом с помощью
  `errors.As` для принятия решений на основе типа или полей ошибки.
* Абстрагировать нижележащие ошибки, предоставляя более высокоуровневое
  представление проблемы.
* Если ваша кастомная ошибка должна оборачивать другую (например, ошибку
  из внешней библиотеки), обязательно реализуйте метод `Unwrap() error`,
  чтобы `errors.Is` и `errors.As` могли «заглянуть» внутрь неё.

### Когда создавать кастомный тип, а когда — sentinel-ошибку

| Ситуация                                                                                        | Инструмент                                           |
| ----------------------------------------------------------------------------------------------- | ---------------------------------------------------- |
| Нужно только обозначить факт ошибки (пользователь не найден, соединение закрыто)                | Sentinel-ошибка: `var ErrNotFound = errors.New(...)` |
| Нужно передать дополнительные данные (код ошибки, какие именно поля невалидны, временная метка) | Кастомный тип-структура                              |
| Вызывающий код должен принимать разные решения в зависимости от свойств ошибки                  | Кастомный тип + `errors.As`                          |

> **Зачем это Go-разработчику.** Кастомные ошибки — мост между

> низкоуровневыми сбоями и бизнес-логикой. Хорошо спроектированный тип

> ошибки позволяет верхним слоям приложения принимать осмысленные

> решения, не зная деталей реализации нижних слоёв.

## 5. Множественные ошибки: `errors.Join`

Начиная с Go 1.20, стандартная библиотека предоставляет функцию
**errors.Join(errs ...error) error** для объединения нескольких ошибок
в одну.

Типичные сценарии использования:

* **Валидация** — собрать все ошибки валидации полей формы и вернуть их
  одной ошибкой, а не останавливаться на первом же невалидном поле.
* **Параллельные операции** — запустить несколько независимых операций и
  собрать все возникшие ошибки.
* **Закрытие ресурсов** — при закрытии нескольких файлов или соединений
  в `defer` собрать ошибки закрытия каждого из них.

```go
var errs []error
if name == "" {
    errs = append(errs, errors.New("name is required"))
}
if age < 0 {
    errs = append(errs, errors.New("age must be non-negative"))
}
if len(errs) > 0 {

    return errors.Join(errs...)
}

```

Результирующая ошибка реализует интерфейс `Unwrap() []error` (для
получения списка исходных ошибок), и с ней работают `errors.Is` и
`errors.As` — они просматривают каждую из объединённых ошибок. Метод
`Error()` объединяет сообщения через перевод строки.

> **Зачем это Go-разработчику.** До Go 1.20 для сбора множественных

> ошибок приходилось использовать сторонние библиотеки (например,

> `hashicorp/go-multierror`) или писать свои обёртки. Теперь это

> встроенная возможность, и её стоит использовать по умолчанию.

## 6. Ошибки в конкурентном коде

Обработка ошибок в горутинах требует особого внимания, потому что ошибка,
возникшая в горутине, по умолчанию **никуда не передаётся** — она просто
теряется, если её явно не вернуть.

### Паттерн с каналом ошибок

Самый простой способ — передавать ошибки из горутины через канал:

```go
errCh := make(chan error, 1)
go func() {
    errCh <- doWork()
}()
if err := <-errCh; err != nil {
    return fmt.Errorf("background work failed: %w", err)
}

```

Буферизованный канал (`make(chan error, 1)`) критически важен: без
буфера горутина-отправитель заблокируется навсегда, если
горутина-получатель уже завершилась (утечка горутины).

### Пакет `golang.org/x/sync/errgroup`

Для более сложных сценариев (несколько параллельных задач, контекст с
отменой) рекомендуется использовать **errgroup**:

```go
g, ctx := errgroup.WithContext(ctx)
g.Go(func() error { return fetchUser(ctx, id) })
g.Go(func() error { return fetchOrders(ctx, id) })
if err := g.Wait(); err != nil {
    return fmt.Errorf("fetch failed: %w", err)
}

```

`errgroup` запускает горутины, собирает первую возникшую ошибку,
отменяет контекст для всех оставшихся задач и ждёт их завершения.

> **Зачем это Go-разработчику.** Потеря ошибок из горутин — молчаливый

> баг, который крайне сложно отлаживать. Паттерн с каналом ошибок или

> `errgroup` — обязательный минимум для любого конкурентного кода.

## 7. Лучшие практики

### Никогда не игнорируйте ошибки

**Всегда проверяйте** `if err != nil` и определяйте соответствующее
действие: логирование, возврат значения по умолчанию или, чаще всего,
передачу ошибки вверх по стеку вызовов. Использование пустого
идентификатора `_` для ошибки допустимо лишь в редчайших, абсолютно
обоснованных случаях.

Не менее важно **избегать использования panic для ожидаемых сбоев**
операций. `panic` предназначен для невосстановимых состояний программы,
сигнализирующих о критической ошибке (например, повреждение внутренних
структур данных), а не о штатном сбое функции.

> **Зачем это Go-разработчику.** Пропущенная проверка ошибки — это

> невалидные данные, которые продолжают свой путь по программе, вызывая

> сбои в самых неожиданных местах.

### Добавляйте контекст при передаче ошибки вверх

При возврате ошибки на уровень выше используйте `fmt.Errorf` с глаголом
`%w`, чтобы обернуть её и добавить контекст. Это создаёт понятную
цепочку ошибок, которая показывает путь, по которому ошибка
«всплывала», и помогает локализовать проблему при отладке.

Сами сообщения об ошибках должны быть информативными, но лаконичными.
По соглашению стандартной библиотеки Go, они обычно начинаются со
строчной буквы и не заканчиваются знаками препинания, если не являются
полными предложениями.

> **Зачем это Go-разработчику.** Сообщение ошибки — это первое, что

> увидит разработчик (или вы сами) в логах продакшена в три часа ночи.

> «ошибка» — бесполезно. `"open users.json: no such file or directory"` —

> уже понятно, куда смотреть.

### Выбирайте правильный инструмент анализа

* Используйте **sentinel-ошибки** (`var ErrNotFound = errors.New(...)`)
  для ситуаций, где важен только факт ошибки.
* Используйте **кастомные типы**, когда нужно передать
  структурированную информацию.
* Используйте **`errors.Is`** для проверки на sentinel-ошибки.
* Используйте **`errors.As`** для извлечения кастомного типа.
* Используйте **`errors.Join`** для сбора нескольких ошибок.
* Всегда реализуйте **`Unwrap() error`** в кастомных типах,
  оборачивающих другие ошибки.

### Обрабатывайте ошибку как можно ближе к источнику

Старайтесь принять решение об ошибке на том уровне стека вызовов, где
имеется достаточно контекста: повторить попытку с другими параметрами,
вернуть пользователю специфическое сообщение, использовать значение по
умолчанию. Это предотвращает «размазывание» логики обработки ошибок по
всему коду и делает её более понятной и локализованной.

> **Зачем это Go-разработчику.** Ошибка `connection refused` на уровне

> HTTP-хендлера — это «503, попробуйте позже». На уровне репозитория —

> это повод для retry с экспоненциальной задержкой. Обрабатывайте там,

> где понятен контекст.
