# Каналы в Go

**Каналы в Go** — это механизм для коммуникации между горутинами и синхронизации их выполнения. Они реализованы как тип данных `chan` и обеспечивают безопасную передачу данных. Канал можно рассматривать как внутреннюю очередь **FIFO (первым вошел — первым вышел) внутри программы**. Некоторые горутины отправляют значения в очередь (канал), а другие горутины получают значения из очереди.

**Горутина** — это легковесный поток выполнения, управляемый рантаймом Go. Горутины запускаются ключевым словом `go` и выполняются конкурентно в общем адресном пространстве. Каналы — основной механизм для безопасного обмена данными между горутинами, реализующий принцип: *«Не общайтесь разделяя память; разделяйте память общаясь»* (Don't communicate by sharing memory; share memory by communicating).

***

## 1. Типы и значения каналов

Подобно массивам, срезам и картам, каждый тип канала имеет тип элемента. Канал может передавать только значения, соответствующие типу элемента канала.

Типы каналов могут быть двунаправленными или однонаправленными. Предположим, `T` — произвольный тип.

* `chan T` обозначает **двунаправленный тип канала**. Компиляторы допускают как прием значений из двунаправленных каналов, так и их отправку.
* `chan<- T` обозначает **канал только для отправки**. Компиляторы не допускают прием значений из каналов только для отправки.
* `<-chan T` обозначает **канал только для приема**. Компиляторы не допускают отправку значений в каналы только для приема.

`T` называется типом элемента этих типов каналов.

Значения двунаправленного канала типа `chan T` могут быть неявно преобразованы как в тип только для отправки `chan<- T`, так и в тип только для приема `<-chan T`, но не наоборот (даже если это явно указано). Значения типа `chan<- T`, предназначенного только для отправки, нельзя преобразовать в значения типа `<-chan T`, предназначенного только для приема, и наоборот. Обратите внимание, что знаки `<-` в литералах типов каналов являются модификаторами.

Каждое значение канала имеет **емкость**. Значение канала с нулевой емкостью называется **небуферизованным каналом**, а значение канала с ненулевой емкостью — **буферизованным каналом**.

Нулевые значения типов каналов представляются предопределенным идентификатором `nil`. Ненулевое значение канала должно быть создано с помощью встроенной функции `make`. Например, `make(chan int, 10)` создаст канал, тип элемента которого — `int`. Второй аргумент вызова функции `make` задает емкость нового создаваемого канала. Второй параметр является необязательным, и его значение по умолчанию равно нулю.

### Сравнение каналов

Любые значения каналов одного типа можно сравнивать друг с другом с помощью операторов равенства. Ненулевой канал, то есть канал, созданный с помощью функции `make`, устроен как многокомпонентное значение, которое внутри содержит ссылку на внутренний объект. Когда вы присваиваете одну переменную канала другой, обе переменные начинают ссылаться на одну и ту же базовую часть этого внутреннего представления. Иными словами, после присваивания оба канала указывают на один и тот же внутренний объект. Это означает, что они полностью разделяют состояние: отправка данных через один канал делает эти данные доступными для получения через другой, и наоборот. Сравнение двух ненулевых каналов с помощью оператора равенства вернёт истину только в том случае, если оба канала ссылаются на один и тот же внутренний объект. Если же каналы были созданы независимо друг от друга, даже при одинаковом типе они будут считаться разными.

### Поведение nil-каналов

`nil`-канал — это канал, которому не было присвоено значение (нулевое значение типа `chan T`). Поведение `nil`-канала отличается от поведения инициализированного канала:

* Отправка в `nil`-канал блокирует горутину навсегда.
* Получение из `nil`-канала блокирует горутину навсегда.
* Закрытие `nil`-канала вызывает `panic`.

> **Зачем это Go-разработчику.** Знание типов каналов (`chan T`, `chan<- T`, `<-chan T`) и их семантики предотвращает ошибки компиляции и документирует намерения. `nil`-каналы блокируют отправку и получение навсегда, но полезны в `select` для динамического отключения веток. Каналы сравнимы через `==`: два канала равны, если созданы одним вызовом `make` или оба `nil`.

## 2. Операции с каналами

Существует пять операций, определяющих канал. Предположим, канал — `ch`. Синтаксис и вызовы этих операций приведены ниже.

### Закрытие канала

```go
close(ch)

```

где `close` — встроенная функция. Аргументом вызова функции `close` должно быть значение канала, и канал `ch` не должен быть каналом только для приема.

### Отправка значения в канал

```go
ch <- v

```

где `v` должно быть значением, присваиваемым типу элемента канала `ch`, и канал `ch` не должен быть каналом только для приема. Обратите внимание, что здесь `<-` — оператор отправки канала.

### Прием значения из канала

```go
<-ch
v := <-ch

```

Операция приема канала всегда возвращает как минимум один результат, который является значением типа элемента канала, и канал `ch` не должен быть каналом только для отправки. Обратите внимание, что здесь `<-` — оператор приема канала. Да, его представление совпадает с оператором отправки канала.

### Прием с проверкой закрытия (comma ok)

В большинстве случаев операция приема значения из канала ведет себя как обычное выражение, которое возвращает ровно одно значение. Однако существует особый случай: когда приём из канала используется как единственное выражение в правой части присваивания, это выражение может породить второе, необязательное значение. Второе значение является нетипизированным логическим флагом, и в таком случае вся конструкция становится многозначным выражением. Этот логический флаг показывает, было ли первое значение успешно получено из канала до того, как канал был закрыт. Если канал уже закрыт и в нём не осталось значений, первое значение будет нулевым для соответствующего типа, а логический флаг примет значение «ложь». Если же канал открыт или даже закрыт, но в нём ещё остались непрочитанные значения, логический флаг будет «истина». Стоит заранее отметить, что из закрытого канала можно получать значения неограниченное количество раз.

Две операции приема канала, используемые в качестве исходных значений в присваиваниях:

```go
v, ok := <-ch
v = <-ch

```

> **Зачем это Go-разработчику.** Все пять операций с каналами потокобезопасны. `close(ch)` — только со стороны отправителя, после завершения всех отправок. `v, ok := <-ch` — стандартный способ безопасного чтения с обнаружением закрытия. `cap(ch)` и `len(ch)` — информационные, не для логики синхронизации.

### Запрос емкости буфера

```go
cap(ch)

```

где `cap` — встроенная функция. Результатом вызова функции `cap` является целочисленное значение.

### Запрос текущего количества элементов в буфере

```go
len(ch)

```

где `len` — встроенная функция. Возвращаемое значение вызова функции `len` — это целочисленное значение. Длина результата — это количество элементов, которые уже были успешно отправлены в запрашиваемый канал, но еще не были получены (извлечены).

### Потокобезопасность операций с каналами

Большинство базовых операций в Go не синхронизированы. Другими словами, они не являются безопасными для параллельного выполнения. К таким операциям относятся присваивание значений, передача аргументов и манипуляции с элементами контейнера и т. д. Однако все только что представленные операции с каналами уже синхронизированы, поэтому для безопасного выполнения этих операций дополнительная синхронизация не требуется.

Как и большинство других операций в Go, присваивание значений каналам не синхронизировано. Аналогично, присваивание полученного значения другому значению также не синхронизировано, хотя любая операция приема канала синхронизирована.

Если запрашиваемый канал является нулевым каналом, обе встроенные функции `cap` и `len` возвращают ноль.

## 3. Буферизованные и небуферизованные каналы

Поведение операций отправки и получения принципиально зависит от того, является ли канал буферизованным.

### Небуферизованные каналы

Небуферизованный канал создается вызовом `make(chan T)` (емкость не указана или равна нулю). Такой канал не имеет внутреннего хранилища — каждая операция отправки требует одновременной операции получения, и наоборот:

* **Отправитель** блокируется до тех пор, пока получатель не заберет значение непосредственно из рук отправителя.
* **Получатель** блокируется до тех пор, пока отправитель не передаст значение.

Небуферизованные каналы гарантируют синхронизацию горутин в точке обмена — это не просто передача данных, но и точка рандеву.

### Буферизованные каналы

Буферизованный канал создается вызовом `make(chan T, N)`, где `N > 0`. Такой канал имеет внутренний кольцевой буфер на `N` элементов:

* **Отправитель** блокируется только тогда, когда буфер полностью заполнен. Пока есть свободное место, отправка происходит без блокировки.
* **Получатель** блокируется только тогда, когда буфер полностью пуст. Пока есть данные в буфере, получение происходит без блокировки.

### Сводная таблица поведения

| Операция      | Небуферизованный канал            | Буферизованный канал                    |
| ------------- | --------------------------------- | --------------------------------------- |
| Отправка      | Блокируется, пока нет получателя  | Блокируется только при полном буфере    |
| Получение     | Блокируется, пока нет отправителя | Блокируется только при пустом буфере    |
| Синхронизация | Гарантирована в точке обмена      | Не гарантирована (асинхронная передача) |

> **Зачем это Go-разработчику.** Небуферизованные каналы — инструмент синхронизации, гарантирующий, что отправитель и получатель встретились в одной точке выполнения. Буферизованные — инструмент асинхронной передачи, позволяющий отправителю продолжить работу, не дожидаясь получателя (до исчерпания буфера).

## 4. Внутреннее устройство каналов

Разобравшись с поведением каналов на уровне пользователя, перейдём к их внутреннему устройству. Понимание структур данных рантайма необходимо для осмысленного чтения раздела о принципах работы операций.

### Структура hchan

В исходном коде Go (файл `runtime/chan.go`) канал представлен структурой **hchan**:

```go
type hchan struct {
    qcount uint    // Текущее количество элементов в буфере.
    dataqsiz uint    // Размер буфера (максимум элементов).
    buf unsafe.Pointer // Указатель на циклический буфер (кольцевой массив) данных.
    elmesize uint16   // Размер одного элемента.
    closed uint32    // Флаг закрытия канала (0 – открыт, 1 – закрыт).
    elemptype *_type   // Тип элемента (для проверок типов и GC).
    sendx uint    // Индекс в буфере для следующей отправки.
    recvx uint    // Индекс в буфере для следующего получения.
    recvq waitq    // Очередь горутин, ожидающих получения.
    sendq waitq    // Очередь горутин, ожидающих отправки.
    lock mutex    // Множество для защиты состояния канала.
}

```

**Буфер (buf):**

Буфер представляет собой **циклический массив**, организованный как кольцевая очередь, для хранения элементов, переданных через канал. Для небуферизованных каналов, то есть созданных с помощью вызова `make(chan T)`, буфер отсутствует, и соответствующее поле `buf` равно `nil`. Для буферизованных каналов, созданных как `make(chan T, N)`, выделяется массив размером `N`, умноженный на размер одного элемента типа `T`.

**Индексы sendx и recvx:**

Эти два индекса указывают на позиции в буфере для следующей операции отправки и следующей операции получения соответственно. Когда индекс при очередной операции достигает конца буфера, он автоматически сбрасывается в ноль благодаря кольцевой структуре буфера.

**Очереди sendq и recvq:**

Обе очереди имеют тип `waitq`, который представляет собой связанный список горутин, заблокированных при выполнении операций с каналом. Очередь `sendq` содержит горутины, ожидающие возможности отправить данные в канал, что происходит когда канал полностью заполнен или, в случае небуферизованного канала, когда нет готового получателя. Очередь `recvq` содержит горутины, ожидающие получения данных из канала, что происходит когда канал пуст.

**Мьютекс (lock):**

Мьютекс защищает все поля структуры канала от одновременного доступа из разных горутин, обеспечивая потокобезопасность операций отправки, получения и закрытия канала.

### Структура waitq

**Waitq (wait queue)** — это очередь ожидающих горутин. Она используется в каналах для хранения отправителей и получателей.

```go
// runtime/runtime2.go
type waitq struct {
    first *sudog
    last  *sudog
}

```

**Поля waitq:**

* `first` — указатель на первый элемент в очереди (голову списка)
* `last` — указатель на последний элемент в очереди (хвост списка)

```go
// Добавление sudog в очередь
func (q *waitq) enqueue(sgp *sudog)

// Удаление sudog из очереди
func (q *waitq) dequeue() *sudog

// Проверка пустоты очереди
func (q *waitq) empty() bool

```

### Структура sudog

**Sudog (synchronization wait object for goroutines)** представляет заблокированную горутину в очереди ожидания, то есть содержимое `waitq`.

```go
// runtime/runtime2.go
type sudog struct {
    // Следующие поля защищены мьютексом канала (hchan.lock)
    g *g    // Горутина

    // isSelect указывает, что операция является частью select
    isSelect bool

    next *sudog    // Следующий элемент в очереди
    prev *sudog    // Предыдущий элемент в очереди
    elem unsafe.Pointer // Указатель на данные (элемент для отправки/получения)

    // Следующие поля никогда не защищаются мьютексом канала
    acquiretime int64
    releasetime int64
    ticket uint32

    parent *sudog // semaRoot дерево
    waitlink *sudog // g.waiting list или semaRoot
    waittail *sudog // semaRoot

    // Канал, для которого ожидает горутина
    c *hchan
}

```

**Ключевые поля sudog:**

* `g *g` — указатель на горутину, которая ожидает в очереди
* `next/prev *sudog` — указатели для организации двусвязного списка в `waitq`
* `elem unsafe.Pointer` — указатель на данные:
  * Для отправителя: указывает на данные, которые нужно отправить
  * Для получателя: указывает на память, куда положить полученные данные
* `isSelect bool` — флаг, указывающий что операция является частью `select`
* `c *hchan` — канал, на котором заблокирована горутина
* Тайминги — `acquiretime`, `releasetime` для отладки и профилирования

### Пул sudog (кэширование)

Для повышения эффективности и снижения нагрузки на сборщик мусора в Go реализован механизм пула для структур `sudog`. Принцип работы этого пула заключается в переиспользовании `sudog` вместо их постоянного создания и уничтожения. Когда горутина блокируется на канале или в `select`, необходимая структура `sudog` берётся из специального пула, если он не пуст, или создается новая, если пул пуст. Когда горутина разблокируется и `sudog` больше не нужен, он не удаляется, а возвращается обратно в пул для будущего использования. Такое кэширование значительно уменьшает количество выделений памяти и, как следствие, снижает нагрузку на сборщик мусора.

## 5. Принцип работы операций

Теперь, зная внутренние структуры данных (`hchan`, `waitq`, `sudog`), рассмотрим пошагово, как выполняются операции отправки, получения и закрытия канала на уровне рантайма.

### Блокировка горутины при отправке

```go
ch := make(chan int, 1)
ch <- 1    // Буфер заполнен
ch <- 2    // Горутина блокируется

```

Когда горутина не может немедленно отправить данные в канал, например потому что канал заполнен или у небуферизованного канала нет получателя, запускается процедура блокировки. Для текущей горутины создается (или берётся из пула) структура `sudog`, которая представляет собой элемент очереди ожидания. В этой структуре поле `elem` устанавливается на указатель того значения, которое горутина пытается отправить. Затем этот `sudog` добавляется в очередь `sendq` канала. После этого горутина переводится в состояние ожидания, уступая управление планировщику, и остается заблокированной до тех пор, пока не появится возможность завершить отправку.

### Разблокировка горутины при получении

```go
val := <-ch // Получаем из канала

```

Когда появляется получатель, который готов забрать данные из канала, планировщик проверяет очередь `sendq` этого канала. Из очереди извлекается первый `sudog`, то есть тот, который ждал дольше всех. Затем данные, на которые указывает поле `sudog.elem`, копируются в переменную получателя, обозначенную как `val`. После успешного копирования горутина, связанная с этим `sudog`, пробуждается. Она переходит из состояния ожидания обратно в состояние готовности к выполнению и вскоре получит управление от планировщика, завершив свою операцию отправки.

### Отправка данных в небуферизованный канал

Когда вы отправляете значение в небуферизованный канал, операция не может завершиться, пока данные не будут переданы непосредственно получателю. Поэтому сначала проверяется, есть ли ожидающие получатели в очереди `recvq`. Если такие получатели есть, данные передаются напрямую горутине-получателю (в ее стек), минуя любое промежуточное хранение. После этого горутина-получатель разблокируется и может продолжить выполнение. Если же ожидающих получателей нет, текущая горутина вынуждена заблокироваться. Для этого формируется структура `sudog`, которая помещается в очередь `sendq`. Затем горутина приостанавливается с помощью функции `gopark`, обращающейся к планировщику. Планировщик меняет состояние горутины на `Waiting`, и выполнение останавливается до тех пор, пока не появится подходящий получатель, который заберёт отправленное значение.

### Отправка данных в буферизованный канал

В случае буферизованного канала отправка сначала пытается использовать свободное место в кольцевом буфере. Если в буфере есть свободные ячейки, данные копируются в буфер по текущему индексу `sendx`. Затем индекс `sendx` увеличивается, а счётчик количества элементов в буфере `qcount` инкрементируется. При этом никакой блокировки не происходит, и горутина продолжает выполнение. Однако если буфер полностью заполнен, отправка начинает вести себя так же, как и в небуферизованном случае. Текущая горутина блокируется, для неё создаётся `sudog`, который добавляется в очередь `sendq`. Далее вызывается `gopark`, планировщик переводит горутину в состояние `Waiting`, и она остается приостановленной до появления свободного места в буфере или до момента, когда какой-либо получатель забирает данные из буфера.

### Получение данных из небуферизованного канала

Когда вы пытаетесь получить значение из небуферизованного канала, операция может завершиться только при наличии готового отправителя. Если в очереди `sendq` есть ожидающие отправители, данные копируются напрямую от горутины-отправителя к горутине-получателю, минуя какой-либо буфер. После этого отправитель разблокируется и может продолжить работу. Если же ожидающих отправителей нет, то получать данные неоткуда, и текущая горутина вынуждена заблокироваться. Для этого формируется структура `sudog`, которая добавляется в очередь `recvq`. Затем вызывается функция `gopark`, планировщик переводит горутину в состояние `Waiting`, и она остается приостановленной до тех пор, пока не появится отправитель, который передаст ей данные напрямую.

### Получение данных из буферизованного канала

В случае буферизованного канала логика получения более сложная и включает несколько этапов. Если в очереди `sendq` есть ожидающие отправители, это означает, что буфер заполнен, поскольку отправители блокируются только при полном буфере. В такой ситуации данные из буфера сначала копируются получателю, а затем данные от первого ожидающего отправителя извлекаются из его `sudog` и помещаются в буфер по индексу `sendx`, после чего `sendx` увеличивается. Если же ожидающих отправителей нет, но в буфере есть данные, данные просто копируются из буфера по индексу `recvx`. Затем `recvx` увеличивается, а счётчик `qcount` уменьшается. После этого получатель проверяет очередь `recvq` на наличие спящих горутин, которые ждут возможности прочитать из пустого буфера. Если такая горутина обнаруживается, получатель забирает ее элемент и помещает в буфер по адресу `sendx`, который затем увеличивается на единицу. Для пробуждения этой спящей горутины получатель обращается к функции `goready`, которая через планировщик меняет состояние горутины с `Waiting` на `Runnable`, и она становится готовой к выполнению. Если же буфер пуст и ожидающих отправителей нет, текущая горутина блокируется. Для неё создается `sudog`, который добавляется в очередь `recvq`, после чего вызывается `gopark`, и горутина переводится в состояние `Waiting` до появления нового отправителя или данных в буфере.

### Закрытие канала

Когда вы закрываете канал с помощью функции `close`, первым делом устанавливается внутренний флаг `closed` в единицу, сигнализируя о том, что канал больше не принимает новые данные. После этого все горутины, которые находятся в очереди получателей `recvq` и ждали данных из этого канала, разблокируются. Каждая из них получает нулевое значение для соответствующего типа данных, поскольку новых значений уже не будет. В очереди отправителей `sendq` все горутины, которые заблокированы в ожидании возможности отправить данные в этот канал, не могут быть разблокированы нормальным образом, потому что отправка в закрытый канал запрещена. В результате каждая такая горутина вызывает `panic`.

> **Зачем это Go-разработчику.** Отправка в небуферизованный канал блокирует отправителя до получения; в буферизованный — только при заполненном буфере. Получение из пустого канала блокирует получателя. Закрытие пробуждает получателей с нулевыми значениями, но вызывает `panic` у отправителей. Правило: **закрывает канал только отправитель**, после завершения всех отправок.

## 6. Select

Оператор `select` — ключевая конструкция для работы с несколькими каналами одновременно. Он позволяет горутине ожидать операции на любом из набора каналов.

### Синтаксис и семантика

```go
select {
case v := <-ch1:
    // сработает, если из ch1 можно прочитать
case ch2 <- x:
    // сработает, если в ch2 можно записать
case v, ok := <-ch3:
    // сработает, если из ch3 можно прочитать (с проверкой закрытия)
default:
    // сработает, если ни один case не готов
}

```

`select` блокируется до тех пор, пока хотя бы один из `case` не станет готовым к выполнению. Если готовы несколько `case` одновременно, `select` **выбирает один из них равновероятно** (псевдослучайным образом). Если присутствует блок `default` и ни один `case` не готов, выполняется `default` без блокировки.

### Внутренняя механика select

```go
select {
    case x := <-ch1:
    // ...
    case ch2 <- y:
    // ...
    default:
    // ...
}

```

Когда в программе используется оператор `select` с несколькими `case`, поведение становится сложнее, чем при работе с одним каналом. Для каждого `case`, который не является `default`, создаётся отдельная структура `sudog`. Все эти `sudog` связываются в специальный список, чтобы планировщик мог отслеживать сразу несколько каналов одновременно. Когда один из `case` срабатывает, то есть операция отправки или получения на соответствующем канале становится возможной, остальные `sudog` удаляются из очередей ожидания соответствующих каналов. Это необходимо, чтобы горутина не осталась заблокированной на нескольких каналах одновременно после того, как она уже была разблокирована одним из них.

### nil-каналы в select

`case` на `nil`-канале **никогда не срабатывает**. Это позволяет динамически включать и отключать отдельные ветки `select`:

```go
var ch chan int          // nil-канал
select {
case <-ch:               // никогда не сработает
    fmt.Println("won't happen")
case <-time.After(time.Second):
    fmt.Println("timeout")
}

```

> **Зачем это Go-разработчику.** Select — основной инструмент мультиплексирования каналов в Go. Выбор готового `case` равновероятен, `default` делает select неблокирующим. `nil`-каналы в `case` никогда не срабатывают — это позволяет динамически включать/отключать ветки. Внутренне select работает через `sudog` и псевдослучайный порядок опроса каналов.

## 7. Практические паттерны

### For-range по каналу

Цикл `for range` по каналу читает значения до тех пор, пока канал не будет закрыт:

```go
ch := make(chan int)
go func() {
    for i := 0; i < 5; i++ {
        ch <- i
    }
    close(ch)
}()

for v := range ch {
    fmt.Println(v) // 0, 1, 2, 3, 4
}

```

После закрытия канала цикл автоматически завершается. Это идиоматичный способ обработки всех значений из канала.

> **Зачем это Go-разработчику.** `for range` — стандартный паттерн для чтения из канала до его закрытия. Без него пришлось бы вручную проверять `v, ok := <-ch` в бесконечном цикле.

### Done-канал и контекст

Для graceful shutdown (корректного завершения) горутин используется отдельный канал — **done-канал**, сигнализирующий о необходимости завершить работу:

```go
func worker(done <-chan struct{}, tasks <-chan Task) {
    for {
        select {
        case task := <-tasks:
            process(task)
        case <-done:
            return // корректное завершение
        }
    }
}

```

Канал `done` типа `chan struct{}` не передаёт полезных данных — важен лишь сам факт закрытия канала (закрытый канал всегда готов к чтению). В современном Go вместо сырого done-канала часто используют `context.Context`:

```go
func worker(ctx context.Context, tasks <-chan Task) {
    for {
        select {
        case task := <-tasks:
            process(task)
        case <-ctx.Done():
            return
        }
    }
}

```

> **Зачем это Go-разработчику.** Done-канал/контекст — основной механизм управления жизненным циклом горутин в Go. Без него горутины могут «повиснуть» навсегда, вызывая утечки.

### Pipeline (конвейер)

Pipeline — это цепочка горутин, соединённых каналами, где выход одной горутины является входом для следующей:

```go
func gen(nums ...int) <-chan int {
    out := make(chan int)
    go func() {
        for _, n := range nums {
            out <- n
        }
        close(out)
    }()
    return out
}

func sq(in <-chan int) <-chan int {
    out := make(chan int)
    go func() {
        for n := range in {
            out <- n * n
        }
        close(out)
    }()
    return out
}

// Композиция: gen → sq → потребитель
for n := range sq(gen(1, 2, 3, 4, 5)) {
    fmt.Println(n) // 1, 4, 9, 16, 25
}

```

Каждый этап закрывает свой выходной канал после завершения, что позволяет следующему этапу корректно завершить `for range`.

> **Зачем это Go-разработчику.** Pipeline — одна из фундаментальных моделей конкурентной обработки в Go. Позволяет разбить сложную обработку на независимые этапы, каждый из которых может масштабироваться отдельно.

### Fan-in / Fan-out

* **Fan-out** — несколько горутин читают из одного канала, распределяя работу.
* **Fan-in** — несколько горутин пишут в один канал, объединяя результаты.

Fan-in через `sync.WaitGroup`:

```go
func fanIn(chs ...<-chan int) <-chan int {
    out := make(chan int)
    var wg sync.WaitGroup
    for _, ch := range chs {
        wg.Add(1)
        go func(c <-chan int) {
            defer wg.Done()
            for v := range c {
                out <- v
            }
        }(ch)
    }
    go func() {
        wg.Wait()
        close(out)
    }()
    return out
}

```

Ключевой момент: выходной канал `out` закрывается только после того, как **все** горутины-отправители завершили работу (`wg.Wait()`). Это гарантирует, что потребитель не останется заблокированным в `for range` по `out`.

> **Зачем это Go-разработчику.** Fan-in/Fan-out — стандартные паттерны для распараллеливания обработки и агрегации результатов в конкурентных программах.

### Направленные каналы в сигнатурах функций

Использование однонаправленных каналов в параметрах функций делает намерения явными и защищает от ошибок на уровне типов:

```go
func produce(out chan<- int) {   // только отправка
    for i := 0; i < 10; i++ {
        out <- i
    }
    close(out)
}

func consume(in <-chan int) {    // только приём
    for v := range in {
        fmt.Println(v)
    }
}

```

Попытка отправить в `in` или получить из `out` вызовет ошибку компиляции.

> **Зачем это Go-разработчику.** Однонаправленные каналы в сигнатурах функций документируют намерения и предотвращают случайные ошибки (например, закрытие канала получателем). Компилятор гарантирует корректность.

## 8. Типичные ошибки и лучшие практики

### Ошибки, вызывающие panic

* **Отправка в закрытый канал** — гарантированная `panic`. Правило: закрывает канал всегда отправитель, и только когда все отправки завершены.
* **Повторное закрытие канала** — `panic`. Закрытие уже закрытого канала недопустимо.
* **Закрытие nil-канала** — `panic`.

### Утечка горутин (goroutine leak)

Горутина, заблокированная навсегда на операции с каналом, никогда не будет собрана сборщиком мусора. Типичный сценарий утечки:

```go
func leak() {
    ch := make(chan int)
    go func() {
        ch <- 42 // отправитель заблокирован: никто не читает
    }()
    // ch не читается, горутина висит вечно
}

```

Решение: всегда обеспечивать парность отправки и получения, использовать done-канал или контекст с таймаутом для гарантированного завершения.

### Дедлок (deadlock)

Взаимная блокировка возникает, когда горутины ждут друг друга. Простейший пример:

```go
ch := make(chan int)
ch <- 1 // deadlock: небуферизованный канал, нет получателя
fmt.Println(<-ch)

```

Рантайм Go обнаруживает deadlock, когда *все* горутины заблокированы, и вызывает фатальную ошибку.

### Правило: отправитель закрывает канал

Закрытие канала со стороны получателя — распространённая ошибка. Получатель не может знать, завершил ли отправитель работу. Правило:

* **Канал закрывает та горутина, которая в него пишет.**
* Если пишут несколько горутин, используйте `sync.WaitGroup` и закрывайте канал после `wg.Wait()` в отдельной горутине.

### Использование sync.WaitGroup для координации

```go
func main() {
    ch := make(chan int)
    var wg sync.WaitGroup

    for i := 0; i < 5; i++ {
        wg.Add(1)
        go func(n int) {
            defer wg.Done()
            ch <- n
        }(i)
    }

    go func() {
        wg.Wait()
        close(ch) // безопасное закрытие после всех отправок
    }()

    for v := range ch {
        fmt.Println(v)
    }
}

```

## 9. Сборка мусора и каналы

Каналы участвуют в механизме сборки мусора. Если на канал больше нет активных ссылок из программы, он считается недостижимым и может быть удален сборщиком мусора. Это справедливо даже в том случае, если буфер канала еще не пуст и содержит данные, — на такие данные тоже больше нет ссылок, поэтому вся структура подлежит уничтожению.

> **Зачем это Go-разработчику.** Канал, на который больше нет ссылок, будет собран сборщиком мусора, даже если в нём остались данные. Однако заблокированные на этом канале горутины не будут разбужены — они останутся висеть (утечка горутин). Поэтому перед удалением ссылки на канал убедитесь, что все горутины, работающие с ним, корректно завершены.

## 10. Заключение

Каналы — фундаментальный механизм конкурентного программирования в Go, реализующий модель CSP (Communicating Sequential Processes). Ключевые выводы:

* Канал — это типизированная FIFO-очередь для безопасной передачи данных между горутинами.
* Небуферизованные каналы синхронизируют горутины в точке обмена; буферизованные — обеспечивают асинхронную передачу с ограниченной ёмкостью.
* Все операции с каналами (`close`, `send`, `receive`, `cap`, `len`) потокобезопасны.
* Внутренне канал — это структура `hchan` с кольцевым буфером, очередями ожидания (`sendq`, `recvq`) и мьютексом.
* `select` позволяет работать с несколькими каналами одновременно; `nil`-каналы в `select` дают возможность динамически управлять набором ожидаемых операций.
* Идиоматические паттерны — `for range`, done-канал/контекст, pipeline, fan-in/fan-out — покрывают большинство практических сценариев.
* Фундаментальное правило безопасности: **отправитель закрывает канал**.

Для углублённого изучения конкурентности в Go рекомендуется ознакомиться с пакетами `context`, `sync` (`WaitGroup`, `Mutex`, `Once`, `ErrGroup`) и статьёй «Go Concurrency Patterns» в блоге Go.
