# Замыкания в Go

## 1. Определение замыкания

### Лексическое окружение

**Лексическое окружение** (**lexical environment**) — это совокупность переменных и параметров, доступных в точке определения функции. Когда функция определена внутри другой функции, она «видит» все переменные внешней функции.

В Go любая функция, которая ссылается на переменные из внешней области видимости, образует замыкание. Для этого не требуется специального синтаксиса — механизм встроен в язык на уровне компилятора.



### Три компонента замыкания

Технически замыкание включает три элемента:

* **Внешняя функция** — задаёт область видимости, в которой определены переменные и параметры, составляющие лексическое окружение.
* **Лексическое окружение** — переменные и параметры внешней функции, к которым обращается внутренняя функция.
* **Внутренняя функция** — функция, использующая переменные внешней функции. В Go это, как правило, **анонимная функция** (**anonymous function**), определённая внутри другой функции.



### Замыкание в Go: не только анонимные функции

В Go замыкания чаще всего создаются через анонимные функции, поскольку именно они естественным образом определяются внутри других функций. Однако **именованная функция**, определённая внутри другой функции (через присваивание переменной), тоже образует замыкание, если обращается к переменным внешней области.

```go
func outer() func() int {
    count := 0
    var inner func() int
    inner = func() int {
        count++
        return count
    }
    return inner
}
```

> **Зачем это Go-разработчику.** Понимание того, что замыкание — это не синтаксическая конструкция, а свойство функции + её окружения, помогает избежать ошибок при передаче функций в горутины и при использовании `defer`.

## 2. Базовый пример: счётчик через замыкание

### Пошаговый разбор

Рассмотрим классический пример — функцию-счётчик:

```go
func outer() func() int {
    count := 0
    inner := func() int {
        count++
        return count
    }
    return inner
}

counter := outer()
fmt.Println(counter()) // 1
fmt.Println(counter()) // 2
```

Разберём выполнение по шагам:

1. Функция `outer` определяет переменную `n` со значением `0` и внутреннюю функцию `inner`.
2. `inner` инкрементирует `n` и выводит её значение. Переменная `n` не является ни параметром, ни локальной переменной `inner` — она принадлежит `outer`.
3. `outer` возвращает `inner` как значение типа `func()`.
4. При вызове `outer()` результат присваивается переменной `fn`. Теперь `fn` — это функция `inner`, **сохранившая доступ к&#x20;****`n`**.
5. Три вызова `fn()` трижды инкрементируют одну и ту же переменную `n`, давая вывод `1`, `2`, `3`.

Ключевой момент: `n` продолжает существовать после выхода из `outer` именно потому, что на неё ссылается замыкание. Компилятор Go размещает такие переменные в куче, продлевая им жизнь.



### Сокращённая форма

Тот же счётчик можно записать компактнее — вернуть анонимную функцию сразу, без промежуточной переменной:

```go
func outer() func() int {
    count := 0
    return func() int {
        count++
        return count
    }
}
```

Смысл не меняется: анонимная функция захватывает `n` из `outer`, и каждый вызов `fn()` увеличивает счётчик.

> **Зачем это Go-разработчику.** Паттерн «счётчик через замыкание» — простейшая демонстрация инкапсуляции состояния без структур и методов. В реальном коде так делают генераторы уникальных ID, счётчики вызовов и обёртки с сохранением состояния.

## 3. Замыкание с параметрами: фабрика функций

### Функция, возвращающая функцию

Более практичный паттерн — **фабрика функций** (**function factory**): внешняя функция принимает параметры и возвращает новую функцию, «настроенную» этими параметрами:

```go
func multiply(n int) func(int) int {
    return func(m int) int {
        return n * m
    }
}
```

Здесь `multiply` принимает `n` и возвращает `func(m int) int` — функцию, умножающую свой аргумент `m` на `n`. Возвращённая функция запоминает `n` из вызова `multiply`.



### Различие параметров внешней и внутренней функций

Важно не путать параметры разных уровней. При создании замыкания:

```go
multiplyBy5 := multiply(5)
```

Число `5` передаётся в параметр `n` функции `multiply`. Оно фиксируется в замыкании.

При вызове замыкания:

```go
fmt.Println(multiplyBy5(6)) // 30
```

Число `6` передаётся в параметр `m` внутренней функции. Результат: `5 * 6 = 30`.

Таким образом, `result1` — это замыкание, которое «зашило» в себя множитель `5`. Аналогично можно создать `result2` с множителем `10` и т.д.



### Альтернативная форма вызова

Замыкание можно вызвать и без промежуточной переменной — напрямую по цепочке:

```go
fmt.Println(multiply(3)(5)) // 15
```

Здесь `multiply(3)` возвращает функцию, которая немедленно вызывается с аргументом `5`, давая `15`.

> **Зачем это Go-разработчику.** Фабрики функций через замыкания — это Go-аналог частичного применения. Полезно для создания настроенных обработчиков HTTP, middleware, функций-компараторов для сортировки и конфигурируемых преобразований данных.

## 4. Ссылочный характер захвата

### Go всегда захватывает переменные по ссылке

В Go замыкание захватывает **ссылку на переменную**, а не её значение на момент создания замыкания. Это фундаментальное свойство, незнание которого — источник многих ошибок.

```go
func main() {
    x := 1
    f := func() {
        fmt.Println(x)
    }
    x = 2
    f() // выведет 2, а не 1
}
```

Замыкание `f` хранит ссылку на `x`. Когда `x` меняется до вызова `f`, замыкание видит актуальное значение, а не то, что было в момент создания.



### Несколько замыканий разделяют одну переменную

Если несколько замыканий захватывают одну и ту же переменную, они разделяют её:

```go
func main() {
    x := 0
    inc := func() { x++ }
    dec := func() { x-- }
    inc()
    inc()
    dec()
    fmt.Println(x) // 1
}
```



### Как добиться захвата по значению

Чтобы зафиксировать значение на момент создания замыкания, нужно скопировать его в локальную переменную внутри той же области видимости:

```go
for i := 0; i < 3; i++ {
    i := i        // копия на каждой итерации
    go func() {
        fmt.Println(i) // каждое замыкание видит свою копию
    }()
}
```

> **Зачем это Go-разработчику.** Ссылочный захват — причина ошибок в циклах с горутинами и при использовании `defer` в цикле. Всегда проверяйте, не изменится ли захваченная переменная между созданием замыкания и его вызовом.

## 5. Захват переменных в циклах

### Классическая ошибка

До Go 1.22 переменная цикла `i` была одной и той же на всех итерациях. Замыкание, созданное в теле цикла, захватывало ссылку на неё, и все замыкания видели последнее значение:

```go
func main() {
    var funcs []func()
    for i := 0; i < 3; i++ {
        funcs = append(funcs, func() {
            fmt.Println(i)
        })
    }
    for _, f := range funcs {
        f() // 3, 3, 3 — не 0, 1, 2!
    }
}
```



### Решение: копирование в локальную переменную

Классическое исправление — скопировать `i` в новую переменную внутри тела цикла. `i := i` создаёт новую переменную с тем же именем, но в своей области видимости:

```go
for i := 0; i < 3; i++ {
    i := i // новая переменная, копия значения
    funcs = append(funcs, func() {
        fmt.Println(i) // своя i для каждой итерации
    })
}
// вывод: 0, 1, 2
```

Альтернативный подход — передать `i` как параметр:

```go
for i := 0; i < 3; i++ {
    funcs = append(funcs, func(j int) func() {
        return func() { fmt.Println(j) }
    }(i))
}
```



### Изменения в Go 1.22+

Начиная с Go 1.22, переменная цикла `for` с `:=` создаётся заново на каждой итерации. Это означает, что замыкание захватывает свою копию, и классическая ошибка больше не воспроизводится.

Однако если вы используете `for` с предобъявленной переменной (`var i int; for i = 0; ...`) или поддерживаете версии до 1.22, старое поведение сохраняется. Явное копирование остаётся хорошей практикой для обратной совместимости.

> **Зачем это Go-разработчику.** Это самая распространённая ошибка с замыканиями в Go. Знание трёх решений (копирование, параметр, Go 1.22) необходимо для код-ревью и отладки.

## 6. Замыкания и горутины

### Проблема: горутина захватывает изменяющуюся переменную

Горутина, запущенная в цикле, исполняется асинхронно. К моменту её реального выполнения переменная цикла уже изменилась:

```go
for i := 0; i < 5; i++ {
    go func() {
        fmt.Println(i) // какая i будет напечатана? Неопределённо
    }()
}
time.Sleep(time.Second)
```

Без копирования возможен любой вывод: повторяющиеся значения, пропуски, все `5` — зависит от планировщика.



### Решение: передача через параметр или копирование

Самый надёжный способ — передать значение как параметр горутины:

```go
for i := 0; i < 5; i++ {
    go func(n int) {
        fmt.Println(n) // n — копия, своя у каждой горутины
    }(i)
}
```

Либо использовать копирование внутри тела цикла (для Go 1.22+ это происходит автоматически при `:=`, т.е. с этой версии эта проблема тоже неактуальна):

```go
for i := 0; i < 5; i++ {
    i := i
    go func() {
        fmt.Println(i)
    }()
}
```



### Пример с sync.WaitGroup

В реальном коде горутины почти всегда координируются через `sync.WaitGroup`:

```go
var wg sync.WaitGroup
for i := 0; i < 5; i++ {
    wg.Add(1)
    go func(n int) {
        defer wg.Done()
        fmt.Println("goroutine", n)
    }(i)
}
wg.Wait()
```

Передача `i` как параметра `n` гарантирует, что каждая горутина получит своё значение независимо от времени запуска.

> **Зачем это Go-разработчику.** Практически любой код до версии 1.22, запускающий горутины в цикле, сталкивается с этой проблемой. Привычка всегда передавать переменную цикла как параметр горутины — признак опытного Go-разработчика.

## 7. Производительность

### Как Go реализует замыкания

На уровне компилятора замыкание в Go — это значение типа `func`, которое внутри представлено **анонимной структурой** из двух полей:

* Указатель на скомпилированный код функции.
* Указатели на захваченные переменные (или сами переменные, если они небольшого размера).

Эта структура аллоцируется в куче, когда переменная «убегает» из области видимости.



### Escape analysis и захваченные переменные

**Escape analysis** — это статический анализ компилятора, определяющий, может ли переменная быть размещена на стеке или должна уйти в кучу. Переменная, захваченная замыканием, **всегда убегает в кучу**, если замыкание переживает функцию, в которой переменная определена.

Рассмотрим два случая:

* **Замыкание не покидает функцию.** Если замыкание только вызывается внутри той же функции, компилятор может оставить переменную на стеке и встроить вызов.
* **Замыкание возвращается или передаётся наружу.** Переменная аллоцируется в куче, и каждое обращение к ней идёт через указатель.

Проверить escape analysis можно командой:

```
go build -gcflags="-m" ./...
```

В выводе строки вида `moved to heap: n` покажут, какие переменные ушли в кучу из-за замыкания.



### Влияние на сборщик мусора

Каждое замыкание, покидающее функцию, создаёт как минимум одну аллокацию в куче: саму структуру замыкания. Если замыкание захватывает несколько переменных, аллокаций может быть больше.

При массовом создании замыканий в цикле это приводит к:

* Росту нагрузки на **сборщик мусора** (**garbage collector**, GC).
* Фрагментации кучи.
* Увеличению `GC CPU` в профилях.



### Рекомендации

| Ситуация                                             | Рекомендация                                                 |
| ---------------------------------------------------- | ------------------------------------------------------------ |
| Замыкание вызывается только внутри создавшей функции | Использовать свободно, накладных расходов почти нет          |
| Замыкание возвращается наружу (фабрика)              | Оправдано: это осознанная плата за гибкость                  |
| Массовое создание замыканий в горячем цикле          | Рассмотреть альтернативы: структуру с методами, пул объектов |
| Замыкание захватывает большой объект                 | Помнить, что объект не будет собран GC, пока живо замыкание  |

> **Зачем это Go-разработчику.** Понимание стоимости замыканий помогает писать производительный код. В горячих путях предпочитайте структуры с методами; в холодных — замыкания оправданы удобством и читаемостью.

## 8. Преимущества и практическое применение

### Инкапсуляция состояния

Замыкание скрывает состояние от внешнего кода. Доступ к переменной возможен только через возвращённую функцию:

```go
func NewCounter() func() int {
    count := 0
    return func() int {
        count++
        return count
    }
}
```

Внешний код не может изменить `count` напрямую — только через вызов счётчика. Это легковесная альтернатива структуре с неэкспортируемым полем и методом.

Применения: счётчики вызовов, генераторы ID, ограничители частоты (rate limiter), кеши с ленивой инициализацией.



### Фабрики функций и частичное применение

Замыкание позволяет зафиксировать часть параметров и получить функцию с меньшим числом аргументов:

```go
func MultiplyBy(factor int) func(int) int {
    return func(n int) int {
        return n * factor
    }
}

double := MultiplyBy(2)
fmt.Println(double(5)) // 10
```

Применения: конфигурируемые middleware, настраиваемые обработчики HTTP, пайплайны преобразования данных.



### Callback-функции и обработчики

Замыкания — естественный способ передать контекст в callback без глобальных переменных. Стандартный пример — `http.HandleFunc`:

```go
func makeHandler(message string) http.HandlerFunc {
    return func(w http.ResponseWriter, r *http.Request) {
        fmt.Fprintln(w, message)
    }
}
http.HandleFunc("/hello", makeHandler("Hello, World!"))
```

Другой пример — сортировка с пользовательским компаратором, зависящим от внешних данных:

```go
threshold := 10
sort.Slice(data, func(i, j int) bool {
    return data[i] > threshold && data[j] <= threshold
})
```



### Сохранение контекста при отложенном выполнении

Замыкание сохраняет контекст для `defer` и горутин. В частности, можно захватить значение до его изменения:

```go
func processFile(name string) {
    f, _ := os.Open(name)
    defer func() {
        f.Close() // f захвачено — Close вызовется с правильным файлом
    }()
    // работа с f
}
```

> **Зачем это Go-разработчику.** Замыкания — не академическая концепция, а повседневный инструмент. Они встречаются в каждом втором обработчике HTTP, в каждой сортировке с контекстом, в каждом координаторе горутин. Владение замыканиями сокращает объём boilerplate-кода и делает архитектуру чище.

## 9. Итог

### Краткое резюме

**Замыкание** — это функция, сохраняющая доступ к переменным своей внешней области видимости. В Go оно создаётся неявно: любая анонимная (или именованная) функция, ссылающаяся на переменные извне, образует замыкание.

Ключевые свойства:

* **Ссылочный захват.** Замыкание хранит ссылку на переменную, а не её значение. Изменение переменной после создания замыкания влияет на его поведение.
* **Продление жизни.** Захваченная переменная живёт, пока живо замыкание. Компилятор размещает её в куче.
* **Инкапсуляция.** Переменная, захваченная замыканием, недоступна извне напрямую — только через вызов замыкания.

### Основные ошибки и как их избежать

| Ошибка                                             | Причина                                                 | Решение                                                          |
| -------------------------------------------------- | ------------------------------------------------------- | ---------------------------------------------------------------- |
| Все замыкания в цикле видят последнее значение `i` | Захват по ссылке; `i` одна на все итерации (до Go 1.22) | `i := i` в теле цикла или параметр                               |
| Горутина печатает не то значение                   | Горутина стартует позже, чем меняется переменная        | Передавать значение как параметр горутины                        |
| Утечка памяти                                      | Замыкание захватило большой объект и живёт долго        | Обнулять ссылки, когда замыкание больше не нужно                 |
| Неожиданное поведение нескольких замыканий         | Они разделяют одну переменную                           | Копировать значение в локальную переменную для каждого замыкания |



### Когда применять замыкания

* Для инкапсуляции состояния без создания типа-структуры.
* Для создания настроенных функций (фабрик, middleware, обработчиков).
* Для передачи контекста в callback-функции.
* Когда код с замыканием читается проще, чем эквивалент со структурой.



### Когда предпочесть альтернативы

* В горячих циклах — структура с методами эффективнее по памяти.
* Когда состояние разрастается и требует нескольких методов — структура понятнее.
* Когда нужно сериализовать состояние — замыкание не сериализуется.
